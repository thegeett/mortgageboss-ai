# LP-18 — Verification Run Model

- **Ticket:** LP-18 — Verification run model; wire findings.verification_id FK (SET NULL)
- **Epic:** Epic 2 — Database & Models (tenth ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Created the **`Verification`** model — the record of one execution of the
verification engine against a loan file (the engine is Phase 3). A run **groups**
the findings it produced and holds run-level metadata (status, trigger, timing,
denormalized red/yellow/green counts, AI cost). This ticket also **wired the FK**
on `findings.verification_id` that LP-17 deferred (ADR-063 → ADR-066).

The defining design point: findings *reference* the run but **belong to the loan
file**, so deleting a run **preserves** its findings — the FK is `SET NULL`, not
CASCADE (ADR-064). This completes the verification data foundation.

Run straight through (sequential, no mid-ticket gates).

## What Was Built

### Model (`app/models/verification.py`)

Two enums + the `Verification` model (Base + UUID/Timestamp/SoftDelete mixins).

| Column | Type | Notes |
| --- | --- | --- |
| `loan_file_id` | UUID FK → loan_files | required, indexed, `ondelete=CASCADE` (owned child) |
| `status` | `VerificationStatus` enum | default `RUNNING`, indexed, VARCHAR + CHECK |
| `trigger` | `VerificationTrigger` enum | required, VARCHAR + CHECK |
| `started_at` / `completed_at` | timestamptz | nullable, timezone-aware |
| `red_count` / `yellow_count` / `green_count` | int | default 0, denormalized (ADR-065) |
| `total_tokens_used` | int | nullable, AI cost |
| `total_cost_estimate` | float | nullable, AI cost |
| `error_detail` | text | nullable, set on `FAILED` |

Relationships: `loan_file` (back_populates `verifications`), `findings`
(back_populates `verification`, **no cascade**, `passive_deletes=True`). `LoanFile`
gained `verifications` (cascade). `Finding` gained the `verification_id`
`ForeignKey(... SET NULL)` and a `verification` relationship.

### Run ↔ Finding relationship (the SET NULL subtlety)

- A run **groups** its findings via `findings.verification_id`.
- Findings **belong to the loan file** (ADR-061), so the FK is **`SET NULL`**:
  **deleting a run preserves its findings** and nulls their `verification_id`.
- `Verification.findings` has **no destructive cascade** and uses
  `passive_deletes=True`, so the database's `ON DELETE SET NULL` does the work
  (required for the async ORM not to lazy-load-and-null the children on delete).
- By contrast, deleting the loan file **does** remove its runs and findings (they
  are its owned children). Asymmetry by design.

### Migration (`69fb46c9cc5b`) — two tables

1. **Creates `verifications`**: all columns, CHECK constraints on `status` /
   `trigger`, CASCADE FK to loan_files, indexes on `loan_file_id` / `status`.
2. **Alters `findings`**: adds `fk_findings_verification_id_verifications` →
   verifications.id with `ondelete=SET NULL`.

Downgrade drops the findings FK **first**, then the table (reverse order).
Hand-cleaned to the repo style.

### Create helper (`app/services/verifications.py`)

`create_verification_run(db, *, loan_file_id, trigger)` creates a `RUNNING` run
with `started_at = utcnow()` and zero counts, then flushes. The engine that
evaluates rules, produces findings, and sets counts/completion is Phase 3.

## Decisions Made

- **ADR-064** — Verification run groups findings; findings reference but aren't
  owned by it (FK SET NULL; relationship no-cascade + `passive_deletes`).
- **ADR-065** — Denormalized red/yellow/green summary counts on the run.
- **ADR-066** — `findings.verification_id` FK added in LP-18 (completes ADR-063).

> **ADR numbering:** the ticket referenced ADR-066…068. The actual next free
> numbers were **064–066** (LP-17 used 060–063). Used 064–066; references to
> scoping/cascade and findings-belong-to-file point at the real ADR-052 / ADR-061.

## Notes

- This completes the verification data foundation and the **deferred FK from
  LP-17** — `findings.verification_id` now has referential integrity.
- `passive_deletes=True` on `Verification.findings` is required: without it the
  async ORM tries to load-and-null the findings on run deletion and raises
  `MissingGreenlet`; with it, the DB-level SET NULL handles it.
- Tests that hard-delete then re-query capture object ids **before**
  `expire_all()` — accessing an attribute on an expired (or deleted) object would
  trigger a sync lazy-load in the async context.
