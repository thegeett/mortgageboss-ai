# LP-60 — Tier 1 Income/Employment Documents

- **Ticket:** LP-60 — Tier 1 income/employment extractors
- **Epic:** Phase 2 — Document Handling (Tier 1 extractors)
- **Status:** Completed
- **Date:** 2026-06-18

## Summary

The first Tier-1 **extractor** batch: full structured extraction for the
income/employment cluster — **1099**, **VOE** (Verification of Employment),
**Profit & Loss**, and the income/employment **Letter of Explanation**. The
classifier already recognizes these (LP-59) and the catalog marks them Tier 1
(LP-58); this ticket builds their extractors following the established LP-39a
pattern and registers them so the Tier-1 routing reaches them. Backend only.

## What Was Built

Four extractors under `app/ai/extraction/`, each mirroring the existing
pay_stub/w2/bank_statement modules, plus their prompts and registry entries:

- `form_1099.py` + `prompts/extraction/form_1099.txt` → `extract_1099`
- `voe.py` + `prompts/extraction/voe.txt` → `extract_voe`
- `profit_and_loss.py` + `prompts/extraction/profit_and_loss.txt` →
  `extract_profit_and_loss`
- `letter_of_explanation.py` + `prompts/extraction/letter_of_explanation.txt` →
  `extract_letter_of_explanation`
- Registered all four in `EXTRACTORS` (`app/ai/extraction/__init__.py`).

## The Established Pattern (followed, not reinvented)

Each extractor follows **LP-39a** exactly, using the existing extractors as
templates:

- A `<Type>Extraction` model = a **typed core** of `TypedField`s (each a coerced
  `value` + its `SourceLocation` of page + verbatim snippet) + an
  `additional_sections` **grouped catch-all**.
- A `<Type>ExtractionResult` with `data` / `status` / `confidence` / `reasoning`
  / token usage and a graceful `.failed()` classmethod — the same interface the
  pipeline + `create_extraction_version` consume uniformly.
- The shared tolerant parser (`parse_typed_core` / `parse_catch_all` /
  `derive_status`), Decimals for money, dates as `date`, **honest nulls** (never
  fabricated), and **metadata-only logging** (status/confidence/counts — never
  values/bytes).

## The Typed Cores (sensible starters — refine with Priya)

Shaped for Phase 3 income verification (the income side of DTI). The catch-all is
the safety net — anything not in the typed core is still captured and can be
promoted later.

- **1099** — `form_subtype`, `payer_name`, `payer_tin`, `recipient_name`,
  `recipient_tin` (sensitive), `tax_year`, `income_amount`.
- **VOE** — `employer_name`, `employee_name`, `position_title`,
  `employment_status`, `start_date`, `end_date`, `current_income_amount`,
  `income_frequency`, `ytd_income`, `hours`,
  `probability_of_continued_employment`.
- **P&L** — `business_name`, `period_start`, `period_end`, `total_revenue`,
  `total_expenses`, `net_profit` (the key self-employment figure; individual
  expense lines → catch-all).
- **LOE** — `subject`, `explanation_summary`, `referenced_employer`,
  `referenced_date`, `referenced_amount` (additional references → catch-all).

## 1099 Subtypes (the variation handling)

The 1099 is a **series** (NEC / INT / DIV / MISC / R) with different relevant
boxes. Rather than five types/extractors, **one** extractor carries a
`form_subtype` slug + a single `income_amount` — the primary figure *for that
subtype* (NEC box 1, INT box 1 interest, DIV box 1a, R box 1 gross distribution,
…), selected by the prompt; every specific box lands in the catch-all. The
subtype preserves the income-vs-asset distinction for Phase 3 (NEC ≈
self-employment income; INT/DIV ≈ asset income). Tested: NEC vs INT (etc.) extract
the right subtype + amount.

## LOE Prose Approach

A Letter of Explanation has no fixed form, so its typed core is deliberately
**light** — what is being explained (`subject`), a faithful `explanation_summary`,
and a single primary referenced employer/date/amount if the letter centers on one.
Any further references go to the catch-all (a "References" section). The same
document type also appears in the LP-63 borrower-info context (it is filed under
`borrower_info` in the catalog); the extractor is shared — the distinction is what
the letter explains.

## Shaped for Phase 3 Income Verification

These four are the major non-pay-stub / non-W-2 income sources: other income
(1099), employer-verified employment (VOE), self-employment income (P&L), and
explained income/employment events (LOE). Their typed cores feed the deterministic
income side of DTI in Phase 3; the catch-all + the AI cross-source layer have the
rest of each document.

## PII

The 1099 `recipient_tin` (an SSN for an individual recipient) follows the **W-2
SSN discipline (ADR-147)**: extracted into the typed core for the Phase 3 identity
cross-check, but **never logged** (only counts + the non-PII subtype) and masked
in display. The raw value lives only in the tenant-scoped extraction JSON.
Metadata-only logging across all four.

## Decisions Made

- **ADR-169** — four Tier-1 income/employment extractors following the established
  pattern; the 1099 series folded into one extractor (subtype slug + primary
  amount); the LOE prose-light core; sensitive TINs masked + never logged; typed
  cores are V1 starters refined with Priya; accuracy honestly scoped (mechanism
  tested, real-sample accuracy validated over time).

## Assumptions

- Field sets are sensible starters **refined with Priya** (her exact needs are
  domain knowledge).
- **Accuracy** against real forms is validated as real documents flow through —
  **no sample 1099/VOE/P&L/LOE documents were available** when this was built.
- **Phase 3** consumes the extracted income data.
- **LP-61..64** add the other Tier-1 clusters (assets, property, borrower-info,
  tax returns) the same way.

## Verification Performed

- **No sample documents** were present (`/mnt/user-data/uploads/` absent), so the
  tests verify the **mechanism/shape**, not per-form accuracy (stated honestly).
- **Each extractor** — returns the typed-core + catch-all shape; per-field source
  locations present; types coerced (Decimals/dates/ints); honest nulls; a
  coercion loss → `PARTIAL`; all-null core → `FAILED`; unparseable → `None`;
  `extract_*` handles success / AI failure / malformed / empty / unsupported
  gracefully (never raises).
- **1099 subtypes** — NEC/INT/DIV/R extract the right subtype + primary amount.
- **Registry routing** — each of the four now routes to its extractor (Tier 1),
  no longer the classified-only no-extractor fallback; an extraction is persisted.
- **PII** — the 1099 recipient TIN is never logged but remains available to the
  caller (tenant-scoped JSON, masked in display).
- **No regression** — the pay_stub/w2/bank_statement extractors are unchanged and
  their suites pass.
- **Suite** — `uv run pytest` green (**779 passed**, +62 over LP-59); `ruff` +
  `ruff format` clean; `mypy` clean (119 source files).
