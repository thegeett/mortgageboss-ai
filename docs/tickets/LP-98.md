# LP-98 — Undo for resolved findings + the Resolved section (completes the epic)

- **Ticket:** LP-98 (seventh and FINAL ticket of the finding-presentation epic, LP-92..98)
- **Epic:** Phase 3.5 — Finding presentation
- **Status:** Completed
- **Date:** 2026-07-01

## Summary

Resolved findings sit in a **Resolved section** below the open findings (LP-94 retains them),
each compact with an **Undo**. Undo of an APPLIED finding **reverses the data change** (restoring
the recorded pre-apply state via `applied_record` — exact, not approximated) and **recomputes**
the DTI/LTV back; Undo of Accept-risk / Override just reopens. This completes the finding-
resolution flow — **View fix previews before Apply (LP-97), Undo reverses after (LP-98)** — and the
epic. See **ADR-223**.

## The type-specific reversal (the hard part)

`app/services/finding_resolution.py` `undo_finding` + `_reverse_applied_change`:

- **Undo-APPLIED** — restore the recorded pre-apply state (the one source of truth, `applied_record`,
  LP-97 verified):
  - `add_liability` → soft-delete the **exact** liability that was added (by its `liability_id`).
  - `correct_income` → restore the income item to its recorded `from` value.
  Then `mark_recompute_needed` → the DTI/LTV read live and recompute back to their **exact** pre-apply
  values. The finding returns to OPEN and its `applied_record` is cleared. (We restore the recorded
  row/value, never subtract an amount — the reversal is exact.)
- **Undo-OVERRIDDEN / Undo-ACCEPTED_RISK** → flip to OPEN (no data change was made).
- Non-resolved finding → `CannotUndoError` (a 400 at the endpoint).

**Audited:** a new `ActivityType.FINDING_UNDONE` (+ a CHECK-constraint migration) records who / when /
the reversal. Tenant-scoped (the reversal only touches the finding's own file).

## The endpoint + UI

- `POST /loan-files/{id}/findings/{finding_id}/undo` → reverses + returns the re-filtered status.
- `FindingPublic` now exposes `applied_record` (the Resolved-card effect + the Undo basis).
- The Resolved section (`findings-list.tsx` `ResolvedGroup`, already below the open findings from
  LP-95) now renders an **Undo** button per finding; the card shows the applied **effect** derived
  from `applied_record` ("Applied · added $500/mo obligation"). Undo fires the resolve mutation with
  `kind: "undo"`; the finding moves back to open and the counts update.

## Composition with LP-94

Resolved findings are retained (LP-94) → they populate the Resolved section → undoable. After
Undo-Applied the data is reversed and the finding is OPEN, so the un-applied issue **re-detects** on
the next run (correct — the issue is back). After Undo-Accept/Override it's a normal open finding
again. No conflict.
