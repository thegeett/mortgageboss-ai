# LP-42 — Document Processing Pipeline Tasks

- **Ticket:** LP-42 — Document processing pipeline tasks (read→classify→extract→persist→needs), enqueue on upload
- **Epic:** Epic 5 — Document Upload & Processing
- **Status:** Completed
- **Date:** 2026-06-12

## Summary

The integration payoff of Epic 5: the Celery task that, on upload, chains **read bytes →
classify (Haiku) → route extraction by type → extract if pay stub (Sonnet) → persist a
versioned extraction → update the needs list on a match → log activity → set a terminal
status** — and wires the upload endpoint to **enqueue** it. This assembles LP-35..LP-41 into
the working document pipeline. Backend only (the UI is LP-43).

- **`app/tasks/document_processing.py`** — `process_document(document_id)` (Celery task,
  `documents.process_document`) → `run_async(_run(...))` → `_process_document(db, ...)` (the
  async core, taking a session so it's directly testable). Plus `_extract_pay_stub_branch`,

- **Enqueue wiring** — `app/api/documents.py` upload, *after commit*, calls

- **Task registration** — added to the LP-41 Celery `include` list.
- **Minimal extension** — `PayStubExtractionResult` (LP-39) gained `input_tokens` /

## The Pipeline (chain)

`PENDING` → read bytes (storage) → **CLASSIFYING** → `classify_document` → set
`document_type` / `category` (provisional map) / `classification_confidence` → **CLASSIFIED**
→ gate → **route by type** → (pay stub: **EXTRACTING** → `extract_pay_stub` → versioned
`Extraction`) → needs-match → activity → **terminal**. **Classification routes extraction**:
Phase 1 has only the pay-stub branch; other types are **classified-only** (COMPLETED, no
extraction — expected V1).

## Status Lifecycle in Motion

The task transitions and **commits** `Document.status` at each stage, so the LP-43 tab sees
progress live:

`PENDING → CLASSIFYING → CLASSIFIED → (EXTRACTING) → COMPLETED | NEEDS_REVIEW | FAILED`

`Document.status` is the source of truth (the UI polls it); Celery's result backend is
secondary.

## Resilience

- **Graceful** classify/extract outcomes (`unknown` / `failed`) or **low confidence** (< 0.5)
  → **`NEEDS_REVIEW`** — expected, the human-review signal (NOT failed).
- **Unexpected** exceptions (storage/DB/etc.) → **`FAILED`** with a *safe* `processing_error`
  (`"processing error"` — never raw PII). One document's failure **never crashes the worker
  or the batch**.
- **Every handled path reaches a terminal status** — never stuck in CLASSIFYING/EXTRACTING.
- `_mark_failed` sets FAILED on the loaded doc + commits (no rollback needed for the common
  storage/AI error); only a genuinely broken transaction triggers a rollback-reload-retry,
  and even that can't raise.

## Retry-Safety

Re-running `process_document` is safe: extraction uses `create_extraction_version` (a **new
version**, one current — not a duplicate), and needs-matching only acts on an **`OUTSTANDING`**
need (a `RECEIVED` one is left alone). Verified by a test that runs the pipeline twice → two
extraction versions (one current) and the need satisfied exactly once.

## Cost Tracking

On a pay-stub extraction the pipeline records `tokens_used` (input + output) and
`cost_estimate` (`estimate_cost`, LP-37) on the `Extraction`. Token usage comes from the
minimally-extended `PayStubExtractionResult` (the extraction module now surfaces the call's
usage). If usage is absent (no call), tokens/cost are `None`.

## Enqueue on Upload

The upload endpoint, **after** the documents are stored + committed (`PENDING`), enqueues
`process_document.delay(str(doc.id))` per document — fire-and-forget, the upload returns
immediately. An enqueue failure (broker down) is caught and logged; it **does not fail the
upload** (the bytes + record are safe and reprocessable).

## Verification Performed

`uv run pytest` → **480 passed** (12 new for the pipeline + 1 enqueue test); ruff check +
format, `mypy app/` (strict), pre-commit all clean. Pipeline tests (AI/storage mocked):

- **Happy path (pay stub)** → COMPLETED; `document_type`/`category`/`confidence` set; a current
  `Extraction` with `SUCCEEDED`, `tokens_used = 390`, `cost_estimate > 0`, `model_used` set.
- **Classified-only (w2)** → COMPLETED, `extract_pay_stub` not called, no extraction.
- **Low-confidence / unknown** → NEEDS_REVIEW, no extraction (gated before it).
- **Extraction failure** → NEEDS_REVIEW + `processing_error`; a FAILED extraction version recorded.
- **Unexpected error** (storage.read raises) → FAILED with the safe message; **no raise**, the
  raw error text is absent from `processing_error`.
- **Missing document** → quiet no-op.
- **Needs satisfaction** → an OUTSTANDING income need → RECEIVED + `satisfied_by_document_id` +
  `satisfied_at`; no match → no-op.
- **Retry-safe** → two runs → two extraction versions (one current), need satisfied once.
- **Privacy** → extracted values (`SECRETCORP`, `9999.99`) absent from captured logs.
- **Upload enqueues** → `process_document.delay` called once per uploaded document, with each
  document's id (broker mocked).

**Live end-to-end (worker running, real DB + Redis + storage):** the configured
`ANTHROPIC_API_KEY` is a placeholder, so a real classify/extract → COMPLETED couldn't be shown
— **but** the full wiring was exercised live: started the Celery worker (`celery -A
app.tasks.celery_app worker --pool=solo`; `documents.process_document` registered), uploaded a
PDF via the API, and **watched the document transition `pending → needs_review` within ~2s**
with no manual action. The worker log showed: task received → `ai_call_failed`
(AuthenticationError, fail-fast) → `classification_ai_failed` (graceful) →
`document_needs_review` → **task succeeded** (returned, didn't raise) → **worker still alive**.
So enqueue → worker → sync/async bridge → status transitions → the graceful-failure path →
resilience are all confirmed live; only the COMPLETED-via-real-extraction path needs a valid
key (it is covered by the happy-path unit test). Test data cleaned up; worker stopped.
