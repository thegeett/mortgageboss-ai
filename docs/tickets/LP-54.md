# LP-54 — MISMO Upload Endpoint + Processing

- **Ticket:** LP-54 — MISMO upload endpoint
- **Epic:** Phase 1.5 — MISMO Import
- **Status:** Completed
- **Date:** 2026-06-12

## Summary

The API entry point for MISMO import (the primary file-creation path): a thin POST endpoint that
accepts a MISMO file upload (XML or HTML-wrapped), runs parse (LP-51) → create (LP-53) **inline**,
commits, and returns the created loan file with parse warnings. Backend only — the endpoint + tests;
no frontend (LP-55 builds the UI that calls this).

## What Was Built

- **`POST /api/v1/loan-files/import-mismo`** (in the loan-files router) — depends on `CurrentUser` +
  `DbSession`; multipart `file` upload. Reads the bytes (chunked, capped), validates at the
  boundary, runs `parse_mismo` → `create_loan_file_from_mismo` inline, commits, reloads scoped, and
  returns `MismoImportResponse`. Status `201`.
- **`MismoImportResponse`** (`app/schemas/mismo.py`) — `{ loan_file: LoanFileDetail, warnings:
  list[str] }`. The file is the existing `LoanFileDetail` (SSN masked), so MISMO and manual creation
  return the same shape.

## Inline, Not Celery (the conscious decision)

MISMO parsing is **fast, deterministic** work — lxml on a ~60 KB file plus a few inserts, **no AI**.
Document processing is slow/AI-bound, so it uses Celery (LP-41/42); MISMO import does **not** need
it. Running parse → create **in-request** is appropriate, simpler (no enqueue / poll /
status-lifecycle), and the response *being* the created file matches **import-directly** — the
frontend navigates straight to the populated file. (If a real file ever proved slow, it could move
to background later.)

## Thin Orchestration

The endpoint is the boundary only: read bytes → size cap → `parse_mismo` → `create_loan_file_from_mismo`
→ commit → reload → respond. All the work is in LP-51 (parse) and LP-53 (map + create). It reloads
and returns via `LoanFileDetail.from_model` exactly like the manual create endpoint — MISMO and
manual **converge** on the same response.

## Graceful Errors (LP-46 envelope)

| Case | Result |
|---|---|
| Unparseable / not-MISMO (`MismoParseError`) | `400` — "This file couldn't be parsed as a MISMO file." |
| No borrower **and** no loan (`MismoImportError`, the floor) | `422` — "…missing the borrower and loan data needed…" |
| **Partial parse** (missing optional fields) | `201` — the created file **+ `warnings`** (success-with-warnings) |
| Empty upload | `422` — "No file provided." |
| Oversized (> ~10 MB) | `413` |
| Missing `file` field | `422` (FastAPI) |
| Unexpected | safe `500` (LP-46 global handler) |

All messages are safe (no stack trace / internals / PII); the underlying exception detail is never
surfaced. Content-type is **not** over-restricted — `parse_mismo` validates the content (XML and
HTML-wrapped both accepted).

## Tenant Scoping + PII

- `company_id` is **always** the authenticated user's; the created file is scoped to their company
  (a second company gets `404` on it — verified).
- The SSN is **masked** in the response (existing `LoanFileDetail` → `masked_ssn`) and the raw SSN /
  `inbox_token` never appear (verified the response body).
- Logging is **metadata-only**: `loan_file_id`, `source_format`, warning count — never the file
  content, SSN, names, or amounts (verified via captured logs).

## Verification Performed

13 integration tests (real stack — httpx + DB + auth + storage temp dir):
- Real fixture → `201`, correct file (loan `1104000.00`, Conventional/Purchase, borrower Mahesh
  Chhotala with **masked** SSN), reachable via the normal scoped GET (**inline** — exists
  immediately).
- HTML-wrapped → `201`; partial parse (stripped property value) → `201` + `warnings`.
- Malformed → safe `400` (no `Traceback`/`lxml` leaked); valid-XML-not-MISMO → `400`; floor (empty
  DEAL) → `422`.
- Empty file → `422`; oversized (>10 MB) → `413`; missing field → `422`; unauthenticated → `401`.
- Tenant scoping (another company → `404`); response doesn't leak the raw SSN / `inbox_token`; no
  SSN/names/amounts in logs (metadata event present).
- `pytest` full suite **628 passed**; ruff + mypy clean.

## Decisions Made

- **ADR-163** — MISMO upload endpoint: inline (not Celery), thin orchestration, graceful error
  mapping, partial-parse warnings in the success response.

## Assumptions

- LP-55 (the upload UI) calls this endpoint.
- Fast/inline is correct for MISMO (no AI); re-import/versioning is deferred.
- Response shape: `{ loan_file, warnings }`; the floor matches LP-53 (`MismoImportError` → `422`).

## Notes

- Could move to a background job if real MISMO files ever get slow (they don't — ~60 KB,
  milliseconds).
- The endpoint lives in the loan-files router (a clean `POST /loan-files/import-mismo`), so MISMO
  import sits alongside manual creation.
