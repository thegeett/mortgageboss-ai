# LP-59 — Comprehensive Classification (all ~80 types → type + category + confidence)

- **Ticket:** LP-59 — Comprehensive Classification
- **Epic:** Phase 2 — Document Handling (Infrastructure & Tiering)
- **Status:** Completed
- **Date:** 2026-06-18

## Summary

Makes the three-tier model (LP-58) real across **breadth**. LP-58 built tier-aware
routing but seeded the catalog with a starter set, and the classifier knew ~3
types — so most documents classified as `unknown` and defaulted to Tier 3. This
ticket expands the catalog to the **full ~80-type taxonomy** (88 types), rewrites
the classification prompt so Haiku recognizes all of them (per-type distinguishing
indicators), and makes routing **confidence-gated**. The catalog and the prompt
are kept in sync by construction (one taxonomy). Backend only.

## What Was Built

- **The ~80-type catalog** (`app/documents/catalog.py`) — expanded from LP-58's
  starter to **88 types** (18 Tier 1, 70 Tier 2) across all seven categories,
  organized by category. An industry-standard starter; refine with Priya.
- **The comprehensive classification prompt** — `app/ai/classification_prompt.py`
  holds `DOCUMENT_TYPE_INDICATORS` (a distinguishing indicator per type) and
  `render_classification_prompt()`, which builds the system prompt by iterating
  the catalog (grouped by category) and injecting type + indicator into the
  template (`prompts/classification/document_classifier.txt`, rewritten to hold
  the framing/output rules + a `{document_type_catalog}` placeholder).
- **Catalog/prompt sync** — the prompt's type list is *derived from the catalog*;
  a test asserts the indicator set exactly equals the catalog set. One source of
  truth for "what types exist".
- **Type + category + confidence** — `ClassificationResult` gained an advisory
  `category` (parsed from the model); the authoritative category stored on the
  document remains the **catalog's** `get_category` (one source of truth).
- **Confidence-gated routing** — the pipeline now branches on *confidence*, not
  on the `unknown` slug: low confidence → `NEEDS_REVIEW`; high-confidence
  `unknown` → Tier 3 (the generic analyzer).
- **Tests** — catalog coverage/spread, catalog↔prompt sync, the confidence gates,
  no regression on the 3 existing types.

## Catalog/Prompt Sync (one type list)

The catalog is the single source of truth for the type list. The prompt is built
from it: `render_classification_prompt()` iterates `types_for_category(...)` and
injects each type + its `DOCUMENT_TYPE_INDICATORS` entry. The framing/output rules
live in the `.txt` template (editable prose); only the type listing is generated.
A test (`test_indicators_exactly_cover_the_catalog`) asserts the two sets are
equal, so they cannot drift — add a catalog type without an indicator and CI fails.

## Confidence Handling

Routing branches on **confidence** (threshold `0.5`), not on the `unknown` slug:

| Outcome | Meaning | Route |
| ------- | ------- | ----- |
| High confidence + known type | clear match | that type's tier (LP-58) |
| **Low** confidence | unsure *which* known type (could be one) | `NEEDS_REVIEW` (LP-44 override) |
| **High**-confidence `unknown` | confidently *none* of the known types | **Tier 3** (generic analyzer) |
| `unknown` @ 0.0 (error fallback) | AI failed / unreadable | `NEEDS_REVIEW` |

This distinguishes "unsure which type" (a human should confirm) from "confidently
novel" (the analyzer's job). Every document reaches a terminal status.

## Notes

Accuracy across all ~80 types is **real-document-dependent**: tests verify the
*mechanism* (sync, routing, confidence gates) + a *representative spread*, not
exhaustive per-type accuracy. The taxonomy and the per-type indicators are an
industry-standard starting point to refine with Priya and tune against real
labeled documents over time. Do not read the test pass as per-type accuracy.
