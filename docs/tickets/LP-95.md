# LP-95 — Finding card restructure: four-part layout + progressive disclosure

- **Ticket:** LP-95 (fourth ticket of the finding-presentation epic, LP-92..98)
- **Epic:** Phase 3.5 — Finding presentation
- **Status:** Completed
- **Date:** 2026-07-01

## Summary

The finding card is now the **four-part layout** — **What we found** / **Why it matters** /
**Suggested fix** / **Source** — with **progressive disclosure** (collapsed by default, expandable).
The AI why/fix slots are **scaffolded but empty** (LP-96 fills them) and **degrade gracefully**:
no empty boxes today, and the card looks complete with just the deterministic content. Frontend-
only, reusing existing stored data. See **ADR-220**.

## The design (plan §3.8)

Four things, two authors:

- **What we found** + **Source** — DETERMINISTIC (the trustworthy core; already stored:
  `finding.message`, `details.reasoning`, `source_page`/`source_snippet`, LP-92 labels).
- **Why it matters** + **Suggested fix** — AI help (added + populated in **LP-96**; scaffolded here).

## What changed (`frontend/components/file/verification/finding-card.tsx`)

**Collapsed (default):**
- Headline (`finding.message`) + a one-line "what we found" (the AI specifics / a summary; omitted
  for deterministic findings whose headline already carries the specifics — no duplication).
- The readable meta (LP-92 `findingTypeLabel` · confidence · origin badge · overlay · docs-requested).
- The action buttons (Apply / Accept risk / Request docs / Override / Add note — unchanged).
- Understandable + actionable without expanding → the list stays scannable.

**Expanded (a single "Details" affordance — no longer gated on having a source):**
- A new `FindingSection` renders four clearly-headed sections:
  - **What we found** — `details.reasoning` (always).
  - **Why it matters** — slot: renders only if `details.why_it_matters` (LP-96).
  - **Suggested fix** — slot: renders only if `details.suggested_fix` (LP-96).
  - **Source** — the document page + verbatim snippet (the view-source click-through), plus the
    authority (the LP-92 label + origin); degrades to "No single document line — {authority}." when
    there's no page/snippet.

**Graceful degradation (critical):** the Why/Suggested-fix slots render **only when populated**.
LP-96 hasn't added them, so today they're absent — no empty boxes, placeholders, or gaps. The AI
why/fix live **on expand**, not inline.

**Resolved findings render compact:** headline + disposition + a what-was-done line (reason /
"Applied — incorporated into the file."); no expander, no four-part. (The Resolved-section
placement + Undo are LP-98.)

## Reuse (no backend/model change)

`message` (headline), `details.reasoning` (what we found), `source_page`/`source_snippet` (source
+ view-source), `findingTypeLabel` (LP-92 authority label), `origin`/`confidence`/`category`
(meta). The why/fix reads (`details.why_it_matters` / `details.suggested_fix`) are defensive — LP-96
populates them.

## Tests

`finding-card.test.tsx`: collapsed-by-default → expands to the four-part + source; graceful
degradation (no empty why/fix boxes when unpopulated; What-we-found + Source still render);
why/fix slots render when populated (the LP-96 shape); a no-source finding still expands + labels
the authority (LP-92 intact); resolved renders compact (no Details expander, no four-part); an
applied finding shows a compact what-was-done line. The existing actions/origin/overlay/docs-
requested tests are unchanged.

## Verification

Frontend-only. `biome` / `tsc` / `vitest` (249) / `build` — all green. No regression on LP-92
labels, LP-93/94 identity/re-run, the view-source click-through, or the existing actions.

## Scope boundary

Does **not** add the AI why/fix content (**LP-96**), View-fix (**LP-97**), or Undo / the Resolved-
section placement (**LP-98**) — it scaffolds the empty slots + graceful degradation and renders
resolved compact. The four sections are delineated so LP-96's content drops into place with no
rework.
