# LP-66 — Tier 3: Generic Analyzer + Findings

- **Ticket:** LP-66 — Tier 3 generic analyzer + findings infrastructure
- **Epic:** Phase 2 — Document Handling (Tier 2/3 handlers)
- **Status:** Completed
- **Date:** 2026-06-19

## Summary

Builds **two related things**: (1) the **Tier 3 generic analyzer** — the
"understand anything" fallback for unrecognized documents (fills the LP-58 Tier 3
stub); and (2) the **findings infrastructure** that earlier tickets have been
waiting for (LP-63's divorce-decree obligations, LP-67's implications engine, and
Phase 3's cross-source verification all depend on it). This **completes the
three-tier handling**: Tier 1 extracts, Tier 2 summarizes, Tier 3 analyzes-anything.
Primarily backend; the heavy display is LP-72.

## The Two Pieces

- **A. The generic analyzer** (`app/ai/generic_analyzer.py`, fills the Tier 3 stub):
  one flexible analysis for any document → a structured-but-flexible output. No
  per-type logic.
- **B. The findings infrastructure** (the architecturally significant part): a
  structured, **uniform-across-tiers** `DocumentFinding` model recorded via one
  shared `create_document_finding` that BOTH the analyzer's `key_findings` AND the
  Tier 1 divorce decree's obligations use.

## The Findings Model

`DocumentFinding` (`app/models/document_finding.py`, table `document_findings`) — a
single-document **observation** that may affect the loan:

- `finding_type` (obligation / property_interest / income_related /
  discrepancy_candidate / other — a small refine-with-Priya enum), `description`,
  common typed fields `amount` + `frequency`, a flexible `details` JSON catch-all
  (findings vary — an obligation has amount+frequency, a property finding has an
  address), and `status` (open / reviewed / dismissed; default open).
- Source-linked to its `document` (FK `ondelete=CASCADE`) and **tenant-scoped
  transitively** via `document → loan_file → company` (no own `company_id`,
  ADR-052). A tenant-isolation test confirms a finding is reachable only via its
  company's loan file.
- **`create_document_finding`** (`app/services/document_findings.py`) is the
  **shared** recording mechanism every source uses — so findings are uniform.

### Distinct from the Phase 3 verification `Finding`

A pre-existing `Finding` (table `findings`) already models the Phase 3
**verification result** (a rule's red/yellow/green flag with a resolution trail). A
`DocumentFinding` is a different concept — an **input observation** from one
document; Phase 3 reads these and may *produce* a verification `Finding`. Two
genuinely different things → **two models / two tables**, not an overload. (The
ticket said "Finding model"; the pre-existing `Finding` made `DocumentFinding` the
honest name — flagged in review.)

## The Generic Analyzer

One flexible mechanism for ALL Tier 3 docs (no per-type logic). Reads any document
→ generic slots: `document_type_guess`, `key_parties` (name + role), `key_dates`
(date + description), `key_amounts` (value + context), `key_findings` (things that
may affect the loan), `summary`, `full_text`. **Sonnet** (it is *understanding*,
not a cheap one-liner — but it surfaces for a human, so moderate-stakes), generous
budget. The analysis + the full text are stored on the document; each `key_finding`
is recorded as a `DocumentFinding`. Graceful: `analyze_document` returns `None` on
failure → the document still finalizes (analysis null, no findings) — never
stuck/crash.

## Wiring the Deferred LP-63 Findings (closing the loop)

The divorce decree's captured support obligations (LP-63's typed core) are wired
into findings: in `_extract_branch`, on a successful extraction,
`record_findings_from_extraction` translates a `DivorceDecreeExtraction`'s
obligations into `DocumentFinding`s via the **same** `create_document_finding` the
Tier 3 analyzer uses. A divorce decree's `$1,200/mo` obligation becomes the same
kind of finding a Tier 3 court order's judgment does — **uniform findings across
tiers**. This is the "later" the LP-63 deferral pointed to.

## Full-Text Indexing

Tier 3 docs' `full_text` is stored on the document and given a **GIN full-text
index** (`to_tsvector('english', full_text)`) — unrecognized docs can't be found by
type, so full-text search matters most for them. This ticket stores + indexes the
data; the **search UI is future**.

## Findings = the LP-67 + Phase 3 Feedstock

Findings are **visible + recorded**: persisted, and surfaced via a tenant-scoped
read endpoint `GET /loan-files/{id}/findings` (`ScopedLoanFile` → 404
cross-company). LP-67 reads them to suggest needs; Phase 3 cross-checks them against
the stated data (and may produce verification `Finding`s). This ticket **records**
single-document findings; Phase 3 does the **cross-source** comparison.

## Decisions Made

- **ADR-175** — the Tier 3 generic analyzer + the document-findings infrastructure
  (structured, uniform across tiers via one shared `create_document_finding`,
  source-linked + tenant-scoped, visible + recorded); distinct from the Phase 3
  verification `Finding`; the divorce-decree wiring closing the LP-63 loop;
  full-text indexed; moderate accuracy (human-in-the-loop).

## Assumptions

- **Moderate accuracy** — findings/analysis are surfaced for a human to assess
  (human-in-the-loop), not silently fed to calculations. Refine-able with
  real/varied documents; the finding-type set refines with Priya.
- The **full tier-aware display** (Tier 3 analysis + findings) is **LP-72**; the
  **full-text search UI** is future.
- **LP-67** reads findings to suggest needs; **Phase 3** cross-checks them.

## Verification Performed

- **No sample documents** present (`/mnt/user-data/uploads/` absent), so the
  analyzer tests verify the mechanism/shape, not accuracy against real varied docs
  (stated honestly — moderate-stakes, human-in-the-loop).
- **The Finding model** — created, source-linked; the flexible details work (an
  obligation with amount+frequency; a property finding with an address in details);
  **tenant-isolation** at both the service level (a loan file sees only its own
  findings) and the endpoint level (cross-company → 404).
- **The generic analyzer** — the structured-but-flexible parse (generic slots; bad/
  empty entries dropped; amounts coerced; honest nulls); Sonnet + generous budget;
  graceful failure (`None`); never logs the bytes / full text / values.
- **The Tier 3 path** — `key_findings` recorded as `DocumentFinding`s; the analysis
  + full text stored (full text indexed); terminal status; graceful on failure;
  **one shared path** (multiple different unknown types through the same mechanism).
- **The divorce-decree wiring** — a divorce decree (Tier 1) extraction records its
  obligation as a `DocumentFinding` via the pipeline — uniform with the analyzer's.
- **Findings visible + recorded** — queryable for a loan file (service + endpoint).
- **Migration** — applies (table + 2 Document columns + GIN full-text index) and
  reverses; verified against dev Postgres.
- **No regression** — Tier 1 (LP-60..64) + Tier 2 (LP-65) + the original 3 unchanged.
- **Suite** — `uv run pytest` green (**963 passed**, +25 over LP-65); `ruff` +
  `ruff format` clean; `mypy` clean (136 source files).
