# LP-78.1 — Cross-source result caching by input fingerprint

- **Ticket:** LP-78.1 — don't re-run the AI when inputs are unchanged (hotfix on LP-78)
- **Epic:** Phase 3 — Verification Engine & Calculators (Arc B)
- **Status:** Completed
- **Date:** 2026-06-29

## Summary

"Run verification" (LP-78) re-ran the AI cross-source pass on every click — even on an unchanged
file. Because the pass is an open-ended AI task (non-deterministic even at temperature 0), the
processor saw the *same* discrepancies described/counted slightly differently each run, eroding
trust and wasting AI cost/latency. This adds **input-fingerprint caching**: if the inputs haven't
changed since the last run, the trigger returns the **cached** findings without calling the AI; only
changed inputs (or an explicit force) re-run it. The back half of the staleness model. See **ADR-193**
and [`phase-3-verification-design.md`](../phase-3-verification-design.md) §12.5.

## What Was Built

- **Input fingerprint** (`app/services/cross_source.py::compute_input_fingerprint`) — a stable SHA-256
  over the assembled stated-vs-verified context (canonical: dict keys sorted, **lists sorted by their
  canonical form** so row order is irrelevant; the compared substance only, no timestamps/ids).
- **Stored on the run** — `verifications.input_fingerprint` (migration `a1c4e7b9f320`), set when a
  cross-source pass completes (`run_cross_source`).
- **Cache on trigger** (`app/api/verification.py`) — the POST computes the current fingerprint (a cheap
  DB read, no AI) and compares to the last completed run; **match → return the cached run, no AI**;
  **differ → fresh pass**. Reconciled with staleness (a cached match clears a stale flag).
- **Force-rerun escape hatch** — `POST …/verification/run?force=true` (+ a "Re-run anyway" affordance in
  the panel) bypasses the cache.
- **Tests** — fingerprint stability + endpoint cached/changed/force/staleness; a frontend panel test.

## The Mechanism

1. **Fingerprint the inputs.** `compute_input_fingerprint(context)` canonicalizes the assembled context
   (sorted keys + order-free lists) and hashes it. Same inputs → same fingerprint; reordering rows →
   same; any value change → different.
2. **Store it** on the completed run alongside the findings.
3. **On trigger** — compute the current fingerprint and compare to the last completed run's:
   - **Match** → return that run's findings; the cross-source **AI is not invoked** (instant, free,
     byte-identical — the same stored rows).
   - **Differ** (a document added/changed, or stated data edited) → a fresh RUNNING run + the worker AI
     pass, which stores the new findings + the new fingerprint.
4. **Staleness reconciliation** — the fingerprint is the precise mechanism behind "changed": a document
   change (which marks STALE) changes the fingerprint, so *stale ⇔ fingerprint differs*. A cached return
   with a matching fingerprint clears the stale flag, so the two never disagree.
5. **Force-rerun** — `force=true` re-runs the AI even on a match (available, not the default).

## Why Caching, Not Determinism

The cross-source pass is open-ended perception — non-deterministic even at temperature 0. The fix is
**not** to make the AI agree with itself (impossible) but to **not re-ask it when there is nothing new
to compare**. Fingerprinting the exact compared substance makes "unchanged" precise; returning the
stored findings is identical by construction. This kills "click repeatedly, get different results" at
the source and removes wasted AI cost/latency.

## Decisions Made

- **ADR-193** — cross-source result caching by input fingerprint: hash the inputs, re-run only when they
  change, reconciled with staleness, with a force-rerun escape hatch.

## Assumptions

- The fingerprint hashes the assembled context (the compared substance) — not volatile metadata.
- The worker is needed only for an *actual* (changed-input/forced) re-run; the cached path is pure DB.
- LP-79's dial still re-filters already-computed findings without re-running (independent of this cache).
- The cross-source capability + the findings model are unchanged — this is a caching layer in front.

## Verification Performed

- **Fingerprint stability** — same inputs → same hash; reordering rows → same; changing a value → diff.
- **Stored** — a completed pass stores the fingerprint of the inputs it compared.
- **Unchanged re-run** — a matching fingerprint returns the cached completed run and **does not enqueue
  the AI** (asserted).
- **Changed re-run** — a differing fingerprint enqueues a fresh RUNNING pass.
- **Force** — `force=true` enqueues the AI despite a match.
- **Staleness** — a cached return with matching inputs clears the stale flag.
- **No regression** — backend **1187 passed**; frontend **174 passed**; `ruff`/`mypy --strict` clean;
  `pnpm lint`/`typecheck`/`build` pass; the migration applies + reverses.

## Notes

- The cached path is fast (no AI latency) and returns the same stored findings — so repeated clicks on an
  unchanged file are instant and identical, which is the whole point.
