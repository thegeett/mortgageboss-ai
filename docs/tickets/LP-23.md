# LP-23 — Authentication Endpoints (login, refresh, logout)

- **Ticket:** LP-23 — Auth endpoints with hybrid token transport
- **Epic:** Epic 3 — Authentication (second ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Built the three auth endpoints under `/api/v1/auth` — **login**, **refresh**,
**logout** — as thin orchestration over the LP-22 token/password utilities. The
substance is the **hybrid token transport**: the short-lived access token is
returned in the JSON body (held in client memory), and the long-lived refresh token
is set as an **httpOnly, path-scoped** cookie that JavaScript can't read. Plus an
`authenticate_user` service with an **anti-enumeration** failure path, request/
response schemas, and a full endpoint+service test suite.

This ticket is backend-only. The current-user dependency and `GET /auth/me` are
LP-24; frontend token handling is LP-25.

## What Was Built

### Schemas (`app/schemas/auth.py`)

- `LoginRequest` — `email: EmailStr`, `password: str`.
- `UserPublic` — `id`, `email`, `first_name`, `last_name`, `role`, `company_id`
  (`from_attributes`); **no** `hashed_password`/`is_active`/timestamps.
- `TokenResponse` — `access_token`, `token_type="bearer"`, `user: UserPublic`. The
  refresh token is deliberately **not** modelled (cookie-only).

### Auth service (`app/services/auth.py`)

- `authenticate_user(db, *, email, password) -> User` — looks up by email; raises a
  **single generic** `AuthenticationError` for unknown-email *or* wrong-password
  (same type and message), and runs a bcrypt compare against a throwaway hash on the
  unknown-email path so timing doesn't leak existence. Raises `InactiveUserError`
  when credentials are valid but `is_active` is false.
- `get_user_by_id(db, user_id) -> User | None` — used by refresh to re-load and
  re-check the user.

### Endpoints (`app/api/auth.py`)

- `POST /login` — `authenticate_user`; on success sets the refresh cookie and returns
  `TokenResponse`; `AuthenticationError → 401` (generic), `InactiveUserError → 403`.
- `POST /refresh` — reads the `refresh_token` cookie (missing → `401`);
  `verify_token(..., REFRESH)` with all three LP-22 errors → `401`; `get_user_by_id`,
  user missing/inactive → `401`; otherwise issues a new access token and a **rotated**
  refresh cookie.
- `POST /logout` — clears the refresh cookie (same path/flags) → `204`.
- Cookie helpers `_set_refresh_cookie` / `_clear_refresh_cookie`; constants
  `REFRESH_TOKEN_COOKIE = "refresh_token"`, `REFRESH_COOKIE_PATH =
  "/api/v1/auth/refresh"`.

### Wiring (`app/main.py`)

`app.include_router(auth_router, prefix="/api/v1")` (constant `API_V1_PREFIX`). A
startup check (and a test) confirm the mounted refresh route equals
`REFRESH_COOKIE_PATH`.

## The Hybrid Transport (the cookie flags and why)

| Token | Transport | Rationale |
| --- | --- | --- |
| access | JSON body → client memory | short-lived; readable by JS to attach as Bearer; dies with the tab |
| refresh | httpOnly cookie, never in body | long-lived/powerful; unreadable by JS (XSS resistance) |

Refresh cookie flags: `httponly=True` (XSS); `secure=settings.is_production` —
**env-conditional** (`False` over dev HTTP, `True`/HTTPS in prod; never hardcoded);
`samesite="lax"` (CSRF); `path="/api/v1/auth/refresh"` (scope — sent only on refresh);
`max_age` = `jwt_refresh_token_expire_days`. Logout clears with the **same path/flags**.

## Security Properties

- **Anti-enumeration** — unknown email and wrong password return the identical generic
  `401`; a service test asserts the two raise the same error type *and* message, and
  an endpoint test asserts the two responses are byte-identical.
- **No PII in the access token** — the login response token decodes to exactly
  `{sub, type, exp, iat}` (reuses the LP-22 property); no role/email/company.
- **Refresh token is cookie-only** — a test asserts the refresh value never appears in
  the login body, and that `hashed_password` is absent.
- **Rotation-lite** — each refresh issues a new refresh cookie (sliding window).

## Decisions Made

- **ADR-076** — Hybrid token transport (access in body, refresh in httpOnly cookie).
- **ADR-077** — Anti-enumeration auth failures (identical 401; bcrypt on the miss path).
- **ADR-078** — Refresh-token rotation-lite; no server-side reuse detection in V1.
- **ADR-079** — No public registration; no login rate limiting in V1.

> **ADR numbering:** LP-22 used 073–075; the next free numbers were **076–079**.

## Assumptions

- **Current-user dependency / `GET /auth/me` / route protection are LP-24** — these
  endpoints are unauthenticated by nature.
- **Frontend token handling is LP-25** — backend-only here.
- **No public signup** (ADR-079); users are admin/seed-provisioned.
- **No login rate limiting in V1** (ADR-079; Phase 7).
- **Email is globally unique** (ADR-042) — lookup needs no company scope.
- **`EmailStr` rejects reserved TLDs** (`.test`), so endpoint tests use `@acme.com`;
  the service test uses the model directly (plain string) so `.test` is fine there.

## Verification Performed

- **Service** (`tests/services/test_auth.py`, 4): valid credentials return the user;
  unknown-email and wrong-password raise the identical error (type + message);
  inactive → `InactiveUserError`; `get_user_by_id` returns user / `None`.
- **Endpoints** (`tests/api/test_auth_endpoints.py`, 12): login success (200, token +
  user, httpOnly/lax/scoped cookie, `secure` absent in dev); body has no
  `hashed_password` / refresh value; access token has only `sub/type/exp/iat`; wrong
  password → 401 + no cookie; unknown email → byte-identical 401; inactive → 403;
  refresh with valid cookie → 200 + rotated cookie; refresh with no cookie / access
  token / tampered token / inactive user → 401; logout → 204 + `Max-Age=0` clear.
- **Route/cookie consistency** — verified the mounted refresh path equals
  `REFRESH_COOKIE_PATH`.
- **Full suite:** `uv run pytest` → **213 passed** (197 → +16). `ruff check`,
  `ruff format --check`, `mypy app/`, and `pre-commit run --all-files` all clean.

## Notes

- **V1 gaps (documented):** no login rate limiting (Phase 7); CSRF posture is
  SameSite-only (no CSRF token); stateless — no server-side revocation of access or
  refresh tokens; rotation-lite without reuse-detection.
- Endpoint tests override `get_db` with the rollback `db_session` so the created user
  is visible to the endpoint and nothing commits. The refresh cookie's path scope is
  honoured by httpx's jar, so the rotation test exercises the real browser behaviour.
