# LP-64 ‚Äî Tier 1 Tax Returns (carved-out, complex)

- **Ticket:** LP-64 ‚Äî Tier 1 tax return extractor (the nested one)
- **Epic:** Phase 2 ‚Äî Document Handling (Tier 1 extractors)
- **Status:** Completed
- **Date:** 2026-06-18

## Summary

The final and most complex Tier-1 extractor: **tax returns**. A tax return is not
one form ‚Äî it is a **nested bundle** (Form 1040 + a variable set of schedules). The
extractor handles a 1040 typed core + typed income-critical schedule
sub-structures (Schedule C / E / K-1, present-or-null, repeatable) + the catch-all.
**This completes Tier 1** ‚Äî every Tier-1 catalog type now has an extractor. Backend
only.

## What Was Built

- `app/ai/extraction/tax_return.py` ‚Üí `extract_tax_return` (the nested extractor)
  with `ScheduleC`, `ScheduleEProperty`, `ScheduleE`, `K1` sub-models.
- `prompts/extraction/tax_return.txt` (generous-budget bundle prompt).
- Registered `tax_return` in `EXTRACTORS` ‚Äî **Tier 1 is complete**.

## Why Tax Returns Are the Hard One

- A "tax return" = Form 1040 + a **variable** set of schedules (Schedule C
  self-employment, Schedule E rental, Schedule B/D, K-1, Schedule 1/2/3, plus
  included W-2s/1099s). Which are present depends on the borrower.
- Multi-page, dense IRS forms; the qualifying-income calculation is non-trivial
  (Phase 3) ‚Äî but the **figures** must be captured here.
- **The self-employed case is the point.** For a W-2 employee the return is largely
  redundant; for a **self-employed** borrower it is THE primary income document ‚Äî
  Schedule C `net_profit` is the qualifying-income figure (the real sample borrower
  had self-employment income from multiple LLCs).

## The Nested Structure

Extends ‚Äî not replaces ‚Äî the LP-39a shape:

- **1040 core** (typed): `tax_year`, `filing_status`, `taxpayer_names`,
  `taxpayer_ssn_masked` (sensitive), `total_income`, `adjusted_gross_income`,
  `wages`, `taxable_income`.
- **`schedule_c`** ‚Äî a **list** (multiple businesses): `business_name`,
  `gross_receipts`, `total_expenses`, `net_profit`.
- **`schedule_e`** ‚Äî present-or-null: a `properties` list (`address`,
  `rents_received`, `total_expenses`, `net_income`) + `total_net_rental_income` +
  `depreciation`.
- **`k1s`** ‚Äî a **list**: `entity_name`, `ownership_pct`, `ordinary_income`.
- **`additional_sections`** catch-all ‚Äî every other schedule (B/D/1/2/3),
  attachments, anything else.

Each schedule field is a `TypedField` with `SourceLocation`, parsed through the
**same** shared typed-core parser ‚Äî so the nesting reuses existing machinery, not a
new shape.

## Schedule C ‚Äî the Self-Employment Heart

`net_profit` on Schedule C is the self-employed borrower's qualifying-income
figure. It is typed, carries its source, and supports **multiple** Schedule Cs (a
list). The key test asserts the net_profit is captured (with source) across two
Schedule Cs.

## Variable Composition Handling

"A 1040 plus whatever schedules this borrower has":

- A schedule **absent** ‚Üí an empty list (`schedule_c` / `k1s`) or `null`
  (`schedule_e`) ‚Äî never assumed.
- A fully-empty schedule entry is **dropped** (no hallucinated schedules); an
  all-empty Schedule E is treated as absent.
- Repeatable schedules (Schedule C, E properties, K-1) are **lists**.
- A W-2 employee's return (no schedules) extracts cleanly with empty
  schedules ‚Äî tested.

## The Generous Budget

A multi-page, multi-schedule bundle is the most content of any extractor, so
`max_tokens` is **16384** (vs 4096 for single-form extractors). A truncated /
malformed multi-schedule response still fails gracefully (`.failed()`) ‚Äî tested.

## Shaped for Phase 3 Qualifying-Income

This extractor **captures the figures**; **Phase 3** derives qualifying income from
them (combining Schedule C net profit + add-backs like depreciation, etc.) and does
the **two-year comparison**. This ticket extracts one return's figures accurately;
it does not compute income.

## PII

The taxpayer SSN is captured **masked** (`taxpayer_ssn_masked`, last 4; the model
masks it) and **never logged** (ADR-147). Tax returns are among the most sensitive
documents ‚Äî metadata-only logging (counts + which-schedules-present), no return
values or SSN in logs. Tested.

## Decisions Made

- **ADR-173** ‚Äî nested tax-return extraction: a 1040 core + typed income-critical
  schedules (C list / E present-or-null with properties / K-1 list) + catch-all;
  variable composition with no hallucination; the generous budget; the same result
  interface despite nested data; captures figures for Phase 3; SSN masked + never
  logged; **Tier 1 complete**; accuracy emphatically scoped.

## Assumptions

- Which schedules/figures to type (vs. catch-all) is a starter **refined with
  Priya**.
- **Phase 3** does the qualifying-income math + the two-year comparison.
- **Accuracy needs real returns most acutely of any extractor** ‚Äî see below.

## Verification Performed

- **No sample returns** present (`/mnt/user-data/uploads/` absent), so the tests
  verify the nested **mechanism/shape**, NOT accuracy against real returns (stated
  emphatically).
- **Nested shape** ‚Äî the 1040 core + the typed schedules + the catch-all extract
  with source locations; types coerced (Decimals/year-int); honest nulls.
- **Schedule C net_profit** (the heart) ‚Äî captured with source across multiple
  Schedule Cs (the key assertion).
- **Variable composition** ‚Äî with/without each schedule; multiple Schedule Cs;
  Schedule E properties list + depreciation; empty/junk rows dropped; a W-2
  employee return has no schedules; a schedule-only return still succeeds.
- **Generous budget** ‚Äî `max_tokens >= 16000` asserted; a truncated response ‚Üí
  `FAILED`.
- **PII** ‚Äî the SSN (even masked) and all return values never appear in logs; only
  counts / presence flags; the masked SSN remains available to the caller.
- **Registry routing** ‚Äî `tax_return` routes to its extractor (Tier 1, category
  Income/Employment); **a test asserts every Tier-1 catalog type now has an
  extractor** (Tier 1 complete).
- **No regression** ‚Äî the existing extractors (LP-60..63 + the original 3)
  unchanged and their suites pass.
- **Suite** ‚Äî `uv run pytest` green (**923 passed**, +22 over LP-63); `ruff` +
  `ruff format` clean; `mypy` clean (130 source files).

## Notes

**Supply real (synthetic / redacted) self-employed returns to validate accuracy ‚Äî
this is the most important extractor to validate against real documents, and the
hardest.** A multi-schedule extractor tested only against constructed inputs is
especially unproven; the nested mechanism is solid, but per-figure accuracy across
the variety of real returns is unverified until real returns flow through. Refine
which schedules/figures to type with Priya.
