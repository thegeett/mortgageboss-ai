## FHA is Genuinely Different from Conventional (the reason it's a separate program)

| Dimension | Conventional (LP-82/83) | FHA (LP-84) |
|---|---|---|
| Source | Fannie Selling Guide (B-sections) | **HUD Handbook 4000.1** (II.A, Appendix 1.0) |
| Credit min | Layered DU-no-min / manual-620 / delivery-floor | **Tiered MDCS** 580‚Üí3.5% / 500-579‚Üí10% / <500 ineligible |
| DTI | DU-50% / manual-45% **hard** ceiling | **Compensating-factors mitigable** 31/43 baseline ‚Üí 40/50 uplift |
| MIP | none | **UFMIP 1.75% + annual table + LTV-90% duration** |

## The Headline ‚Äî MDCS is TIERED to the down payment, NOT a flat min

- `fha.credit.mdcs_eligibility_floor` ‚Äî ‚â•500, RED (below 500 ‚Üí ineligible for FHA-insured financing). II.A.4.
- `fha.credit.mdcs_minimum_down_3_5_tier` ‚Äî ‚â•580, YELLOW (580+ ‚Üí 3.5% down; 500-579 flags the 10% tier). II.A.4.
- `fha.credit.mdcs_low_tier_down_payment` ‚Äî down payment ‚â•10%, RED, **gated** to `credit.mdcs < 580`. II.A.4.

A 540 score with 5% down fires the low-tier 10%-down requirement; a 620 score closes that gate (standard 3.5%).

## The Compensating-Factors "Conservative-Flag / Accept-Risk" DTI Model (mitigable, not hard)

FHA DTI findings are **MITIGABLE**, not silent hard cutoffs ‚Äî the plan's specific FHA requirement:

- `fha.dti.front_end_baseline` (‚â§31%, **YELLOW**) / `fha.dti.back_end_baseline` (‚â§43%, **YELLOW**) ‚Äî over the
  baseline is a flag **resolvable by documenting a compensating factor** (LP-75 resolution ‚Äî OVERRIDDEN-with-
  reason / APPLIED).
- `fha.dti.front_end_max_with_factors` (‚â§40%, **RED**) / `fha.dti.back_end_max_with_factors` (‚â§50%, **RED**) ‚Äî
  the uplifted ceiling; hard only past here (compensating factors cannot rescue it).
- `fha.dti.compensating_factors_required` ‚Äî ‚â•1 documented factor, **gated** to back-end DTI > 43%.
- `fha.dti.payment_shock_reviewed` ‚Äî payment shock as a compensating-factor input (to verify).

A 47% back-end DTI fires the YELLOW baseline (mitigable) but passes the 50% RED ceiling; a 55% breaches it. The
DTI rules **consume LP-76's computed `dti.front_end_pct` / `dti.back_end_pct`** (read, never recompute). These
supersede the LP-74 sample `fha.dti.back_end_max` (a 57% placeholder).

## MIP ‚Äî No Conventional Analog (FHA-defining)

- **UFMIP 1.75% (175 bps)** of the base loan amount + a present-check (missing ‚Üí RED).
- **Annual MIP** as rate-as-data (~15-75 bps, most 55; the ‚â§75 starter is the upper bound, table to verify).
- **The LTV-90% duration rule:** LTV > 90% ‚Üí annual MIP for **life**; LTV ‚â§ 90% ‚Üí **11 years** (132 months) ‚Äî
  two LTV-gated rules **reading LP-77's `ltv.ltv_pct`**.
- MIP rules use the **`DOCUMENTATION`** category ‚Äî there is no dedicated `MORTGAGE_INSURANCE` category and
  adding one would require a migration (the category column is a CHECK-constrained VARCHAR); a dedicated MI
  category is a deferred promotion.

## Program-Gating (both ways)

`registry.resolve(program=FHA)` returns the FHA grounded rules; `resolve(program=CONVENTIONAL)` returns none of
them (and no `conv.*` rule resolves for an FHA file). The Conventional test file (Mahesh) is Conventional ‚Äî the
FHA rules never fire on it. Tests use a **constructed FHA fixture** (FileFacts) + a DB-backed FHA loan file.

## Cross-Links (confirmed, not duplicated)

- The DTI rules **consume LP-76's computed `dti.front_end_pct` / `dti.back_end_pct`** (`dti.back_end_pct` is
  promoted today ‚Üí evaluable; `dti.front_end_pct` promotion-pending).
- The MIP duration rules **read LP-77's `ltv.ltv_pct`** (the canonical LTV fact path the sample `fha.ltv.*`

## Typed-Core Reuse + Promotion

The FHA income/asset rules read the **same promoted paths as LP-82** (`assets.gift.total_amount`,
`assets.retirement.total_amount`, `income.self_employment.monthly_amount`, `assets.largest_deposit_amount`,
`dti.back_end_pct`) ‚Äî evaluable today on an FHA file, no new promotion. The FHA-specific facts (`credit.mdcs`,
`down_payment.pct`, `dti.front_end_pct`, `ltv.ltv_pct`, the MIP fields) are promotion-pending ‚Äî recorded
not-evaluated (graceful) until the typed extraction grows.

## Overlay-Overrideability (FHA overlays are common)

A constructed Sun-West overlay tightens `fha.credit.mdcs_eligibility_floor` (500‚Üí620) and `fha.mip.ufmip_rate`
by `rule_id` (LP-80). Most lenders set 580-640 floors over FHA's 500/580 ‚Äî a strong enforcement case.

## Verification Performed

- **Structure** ‚Äî 31 rules (8 credit + 6 DTI + 6 income + 5 asset + 6 MIP), each investor/FHA, typed reads,
  threshold/rate-as-data, severity, structured HUD citation (`section` + `retrieved=2026-06`), `starter=True`.
- **Tiered MDCS** ‚Äî eligibility floor (‚â•500 RED), 3.5%-tier (‚â•580), gated low-tier 10%-down; 540/5% fires the
  low tier, 620 closes the gate, <500 is ineligible. NOT a flat min.
- **Mitigable DTI** ‚Äî baseline YELLOW / uplifted-ceiling RED; 47% back-end is a mitigable YELLOW within the 50%
  ceiling; 55% breaches it; the compensating-factor requirement gates on back-end > 43%. The rules consume the
  computed front-/back-end.
- **MIP** ‚Äî UFMIP 1.75% (175 bps) + RED missing-MIP present-check; the duration rules read `ltv.ltv_pct` and
  gate on 90% (96.5% ‚Üí life applies, 11-year doesn't; 85% ‚Üí 11-year applies, life doesn't).
- **Program-gating** ‚Äî FHA rules resolve for FHA files only, not Conventional (and vice versa).
- **Overlay-overrideability** ‚Äî an FHA MDCS floor (500‚Üí620) + the MIP rate overridden by `rule_id`.
- **FHA fixture** ‚Äî a constructed manual FHA file (540/5%, 47% DTI, 96.5% LTV, no MIP) fires the tiered MDCS,
  the manual-UW flag, the mitigable DTI baselines, the compensating-factor requirement, the missing UFMIP, and
  the life-of-loan MIP duration ‚Äî but NOT the hard 50% ceiling.
- **Test-file-shaped FHA record** ‚Äî a DB-backed FHA loan file reuses the LP-82 promotions (gift + DTI) and is
  program-gated.
- **No regression** ‚Äî backend **1283 passed** (+21 LP-84); `ruff`/`mypy --strict` clean. No migration.
