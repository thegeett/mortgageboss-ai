# LP-38 — Document Classification Module

- **Ticket:** LP-38 — Document classification module (classify_document, typed result, prompt-as-file, graceful failure)
- **Epic:** Epic 5 — Document Upload & Processing
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Built the document **classification** module: given a document's already-extracted text,
`classify_document(text)` returns what KIND of document it is (`pay_stub`, `bank_statement`,
`w2`, …) with a confidence and reasoning. First real use of the LP-37 AI wrapper and the
first act of the system "understanding" a document; classification routes extraction (LP-39
is type-specific). Backend only; tests mock the wrapper (no real calls, no key).

## What Was Built

- **`ClassificationResult`** (Pydantic, `app/ai/classification.py`) — `document_type: str`,
  `confidence: float` (`ge=0, le=1`), `reasoning: str`, plus the
  `ClassificationResult.unknown(reason)` graceful fallback.
- **Prompt file** — `app/ai/prompts/classification/document_classifier.txt`, a clearly
  marked **STARTER** that instructs the model to return JSON
  (`document_type` / `confidence` / `reasoning`) and use `unknown` when unsure.
- **Prompt loader** — `app/ai/prompt_loader.py::load_prompt(relative_path)`: reads a prompt
  from the prompts dir (CWD-independent, path-escape-checked, `@cache`d). The pattern
  extraction (LP-39) reuses.
- **`classify_document(text)`** — empty-text short-circuit, prompt load, LP-37 `complete`
  call with the classification model, defensive JSON parse, graceful `unknown` on any
  failure, metadata-only logging.
- **`_parse_classification_json` / `_extract_json_object` / `_coerce_confidence`** — the
  defensive parser.

## The Contract

`classify_document(text: str) -> ClassificationResult` — always returns a typed result,
never raises. `document_type` is a **flexible string** (LP-15), not an enum. This module
**returns** a result; persisting it onto the `Document` is the pipeline's job (LP-42), which
treats `unknown` / low-confidence as `NEEDS_REVIEW`.

The text is sent as the **user** message; the file-based prompt is the **system** prompt
(keeps the instructions and the untrusted document content separate).

## Prompt-as-File (a POC asset)

The prompt is loaded from the file, never hardcoded in Python — so swapping it is a content
edit. **The shipped prompt is a STARTER** (clearly marked at the top of the file): the real,
tuned classification prompt is a POC asset and must be pasted into
`document_classifier.txt`, replacing the starter. No code change is needed to swap it.

## Graceful Failure

`classify_document` **never raises**. Failure paths all return
`ClassificationResult.unknown(...)`:

| Situation | Result |
| --- | --- |
| empty / whitespace / `< 10` chars | `unknown("empty or insufficient text")` — **no API call** |
| `AIClientError` from the wrapper | `unknown("AI call failed")` |
| unparseable / non-JSON response | `unknown("could not parse classification")` |
| valid but low confidence | returned **as-is** (the pipeline flags review, not this module) |

The defensive parser extracts the first balanced `{...}` object (tolerating ```` ```json ````
fences and surrounding prose), clamps `confidence` to `[0, 1]` (coercing numeric strings,
rejecting bools/junk → `0.0`), and treats a missing/empty `document_type` as `unknown`.

## Privacy

The document text and the model's raw response carry borrower PII and are **never** logged.
Only metadata is logged here — `classification_succeeded` carries `document_type` +
`confidence`; the failure logs carry no content. The wrapper (LP-37) logs call metadata
(tokens/latency), also content-free. A test seeds PII into the text and the response and
asserts neither appears in captured logs.

## Uses the LP-37 Wrapper

Calls `app.ai.client.complete(...)` (not the SDK) with
`settings.anthropic_model_classification` (a cheaper Haiku-class model intended for
classification — a `TODO`-marked placeholder to verify) and a small `max_tokens` (512, the
output is a tiny JSON object). Token usage flows back via the wrapper for cost tracking.

## Decisions Made

- **ADR-120** — classification returns a typed result; type is a flexible string.
- **ADR-121** — prompts stored as files, loaded at runtime (starting with classification).
- **ADR-122** — graceful failure: classification never crashes the pipeline.

> **ADR numbering:** LP-37 used 117–119; the next free numbers were **120–122**.

## Assumptions

- **PDF text extraction** is LP-40 (this takes text, not bytes/PDFs); **field extraction**
  is LP-39; the **pipeline that calls this and persists the result** is LP-42; the **full
  ~100-type taxonomy** is Phase 2.
- The shipped prompt is a **starter to replace with the POC prompt**.
- Reused the existing `anthropic_model_classification` setting (the spec's
  `ai_classification_model`) — both name the classification model.
- Tests **mock** the wrapper — no real calls, no key (CI safe).

# LP-38 (Modification) — Classify From the Full Document (not text)

- **Date:** 2026-06-11
- **Status:** Completed
- **ADR:** ADR-127 (revises ADR-120/121/122; follows the LP-37 revision / ADR-126)

## What Changed

Following the full-document AI decision (ADR-126), classification now reads the **full
document** (PDF/image bytes) natively with the Haiku-class model — no OCR, no pre-extracted
text — using the LP-37 document/image content block.

- **Signature:** `classify_document(text: str)` → **`classify_document(content: bytes,
  media_type: str)`** (async).
- **Call construction:** the loaded prompt stays the `system` instruction; the document is
  sent as the `user` message via `build_document_message(content=..., media_type=...)`
  (LP-37 revision). The model is unchanged (`settings.anthropic_model_classification`,
  Haiku-class), `max_tokens=512`.
- **Short-circuit:** an **empty** (`len 0`) document or an **unsupported** `media_type`
  (outside `application/pdf` / `image/jpeg` / `image/png` / `image/jpg`) returns
  `ClassificationResult.unknown("empty or unsupported document")` **without an API call**.
  (A defensive `try/except ValueError` around `build_document_message` is also kept, though
  the pre-check already filters unsupported types.)

## What Stayed (no regression)

`ClassificationResult` + `.unknown(...)`, the **defensive JSON parser** (`_parse_classification_json`,
unchanged — fences/preamble/malformation → graceful), the **graceful-failure contract**
(`AIClientError` → `unknown("AI call failed")`, unparseable → `unknown("could not parse
classification")`, **never raises**), the file-based **starter prompt**, the **Haiku model**,
and **metadata-only logging**.

## Privacy (reinforced)

The document **bytes**, their **base64**, the message content, and the raw response are
**never** logged — only `document_type` + `confidence` on success. The privacy test now seeds
PII into the **bytes** and asserts neither the raw byte content nor the base64 payload appears
in captured logs.

## Supported Media Types

`application/pdf`, `image/jpeg`, `image/png` (`image/jpg` normalized to `image/jpeg`).
Matches the LP-36 upload allowlist and the LP-37 document-block support.

## Verification

`uv run pytest` → **440 passed**, `tests/ai/` passes **without a real key** (wrapper mocked).
ruff check + format, `mypy app/` (strict), pre-commit all clean. The classification tests were
adapted to bytes + media type:

- **Success (PDF)** — returns the parsed result; asserts `complete` was called with a
  **document block** whose base64 round-trips to the original bytes.
- **Image input (PNG)** — uses an **image block** (`media_type` `image/png`).
- **Fenced JSON / low-confidence** — parse unchanged; low confidence returned as-is.
- **Malformed → unknown**; **AIClientError → unknown** (not propagated; never raises).
- **Empty document → unknown without calling**; **unsupported media type
  (`text/plain` / `application/zip` / `image/gif` / `""`) → unknown without calling**.
- **Confidence clamp** — `1.5 → 1.0`.
- **Privacy** — raw byte content + base64 + response content absent from logs; metadata
  success log present.
