## Summary

The full Wireframe-5 verification tab — the production surface Priya uses daily — built by **EXTENDING LP-81's
minimal tab** (not rebuilding it) with the deferred richness + surfacing everything that landed since (the full
rule set LP-82..85, the deterministic cross-source LP-86, all six calculators LP-76/77/87, the overlays LP-80).
The design challenge was complexity management: powerful without clutter. See **ADR-210** (the tab) + **ADR-211**
(accept-risk + request-docs). Full-stack, heavily frontend.

## Extends LP-81 (does NOT rebuild)

LP-81's composition, findings list, aggression dial, cross-source trigger + staleness, needs indicator, and
re-run stability are untouched and still work. LP-88 adds on top:

| Addition | What | Where |

| **Source-origin + overlay + program** | deterministic vs AI·novel badge; lender overlay; Conv/FHA header | `finding-card.tsx`, panel header, `program` on status |

## The filter pills compose with the dial (two orthogonal filters)

The aggression dial (LP-79) sets the **confidence** floor — which findings are in-scope. The pills slice that
in-scope set by **severity** + **category**. `FindingsList` applies the dial cutoff first, then `matchesFilters`.
The stats row + the pills reflect the in-scope set; the list reflects the further slice.

## The full per-finding action set

- **Apply / Override / Note** — LP-81 (unchanged).
- **Accept-risk** (LP-88) — acknowledge a REAL finding the processor proceeds with (distinct from override's
  dismissal); for the FHA compensating-factors (LP-84) + subject-to-repair (LP-85) conditional model. Reuses the
  existing `ACCEPTED_RISK` resolution state (no migration). Optional rationale (the compensating factor).
- **Request-docs** (LP-88) — create a `FINDING`-origin needs item (priority from severity) the borrower must
  satisfy; the finding is **marked** (`details.docs_requested`) but stays OPEN. Reuses `create_needs_item` (no
  migration). The needs list + Phase-4 communication act on the item.

## Six calculators — progressive disclosure (the complexity-management core)

Six calculators can't all be expanded without overwhelming the tab. `CalculatorsSection` shows a scannable strip
of six summary tiles (title + headline + a status dot — the at-a-glance picture) and expands exactly ONE into its
full transparent/overrideable calculator (DTI/LTV via LP-76/77's components; MI/self-employed/reserves/max-loan
via LP-87's `CalculatorCard`). The summary hooks share the query cache with the full components — expanding
doesn't refetch.

## Surfacing what landed since LP-81

- **Deterministic vs AI cross-source (LP-86):** each finding shows a `deterministic` (stable/certain) vs
  `AI · novel` (the frontier) badge from `origin`.
- **Lender-specific results (LP-80):** the finding shows the lender **overlay** that adjusted its threshold
  (from `details.overlay_applied`); the tab header shows the **program** (Conv/FHA — a new `program` field on
  the status).
- **The full rule set (LP-82..85):** the findings list already renders all origins uniformly; these are now
  program-aware + overlay-adjusted, distinguished from the AI novel findings.

## Backend (minimal — no migration)

- `POST …/findings/{id}/accept-risk` → `accept_risk_finding` (existing `ACCEPTED_RISK` state).
- `POST …/findings/{id}/request-docs` → `request_docs_for_finding` (existing `create_needs_item`).
- `GET …/verification/runs` → the run history (runs were already versioned).
- `program` added to `VerificationStatusPublic`.
- All tenant-scoped (cross-company → 404); activity-logged.

## Complexity management (production-quality)

Hierarchy (stats → calculators strip → dial → pills → findings), progressive disclosure (one calculator
expanded; findings dial-then-pill filtered; stats summarizing), and reuse of the established card/badge/pill
idiom keep it scannable despite six calculators + ~120 rules' findings. Loading/error/empty preserved; PII
masked. The page shrank from six always-expanded calculators to the strip + panel.
