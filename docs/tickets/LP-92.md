# LP-92 — Readable finding labels: teach the finding-display layer about the `xsrc.` namespace

- **Ticket:** LP-92 (first ticket of the finding-presentation epic, LP-92..98)
- **Epic:** Phase 3.5 — Finding presentation
- **Status:** Completed
- **Date:** 2026-06-30

## Summary

A small, frontend-only fix. Deterministic cross-source findings displayed an ugly raw-rule-id
meta-label — e.g. **"Xsrc Income Employer Count Matches Items"** — because the finding-display
layer only recognized the AI cross-source `cross_source.` prefix and fell back to prettifying
the raw rule_id for the LP-86 deterministic `xsrc.*` rules. This teaches the layer about the
`xsrc.` namespace and makes the fallback a readable **category** label — never a raw rule_id.
See **ADR-217**.

## The bug (confirmed from the trace)

`frontend/lib/verification/finding-display.ts`:
- `CROSS_SOURCE_PREFIX = "cross_source."` (`:12`); `TEMPLATES` keyed on the post-`cross_source.`
  type (`:15-25`).
- `findingType` (`:28`) stripped **only** the `cross_source.` prefix → returned `null` for
  `xsrc.*`.
- `findingTypeLabel` (`:56`) fell back to `type ?? finding.rule_id` and prettified it →
  for `xsrc.income.employer_count_matches_items` produced "Xsrc Income Employer Count Matches
  Items".
- The AI layer (LP-78) uses `cross_source.*` → matched → readable label. The LP-86
  deterministic rules use `xsrc.*` → never matched → the raw-rule-id meta-label.

The **headline** was already fine: `finding-card.tsx:141` uses `findingHeadline`, which for
`xsrc.*` falls through to `finding.message` — and the backend renders a readable message
(e.g. "Stated employer count (2) does not match the income-item count (3)."). So only the
secondary gray meta-label (`finding-card.tsx:157` = `findingTypeLabel`) was broken.

## The fix (frontend-only)

`frontend/lib/verification/finding-display.ts`:
- **`findingType`** now recognizes + strips **both** `cross_source.` and `xsrc.` (an `xsrc.*`
  finding resolves to a type instead of null). Headline/detail behavior is unchanged — the
  stripped `xsrc.` remainder isn't a `TEMPLATES` key, so the headline stays `finding.message`.
- Added `CATEGORY_LABELS` + a `categoryLabel()` helper (income → "Income", credit → "Credit",
  …, with a "Verification check" generic fallback).
- **`findingTypeLabel`** is now readable and never returns a raw rule_id:
  - AI cross-source (`cross_source.*`) → the canonical type, e.g. "Income Variance" (unchanged).
  - Deterministic cross-source (`xsrc.*`) → category + descriptor, e.g. "Income · Cross-source
    check".
  - Everything else (single-source `conv.*` / `fha.*`, document findings) → the readable
    category label — never "Conv Dti …".

`finding-card.tsx:157` already renders `findingTypeLabel(finding)`; no card change needed. The
headline (`:141`) is untouched.

## Tests

`frontend/lib/verification/finding-display.test.ts` (new cases):
- `xsrc.*` → "Income · Cross-source check" (and no "Xsrc"); `findingType` non-null; headline
  unchanged (= `finding.message`).
- An `xsrc.identity.*` finding labels by its own category ("Credit · Cross-source check").
- AI `cross_source.*` still labels "Income Variance" (no regression).
- An unmatched rule_id (`conv.dti.back_end_max`) → "Income" (the category), not "Conv Dti …";
  an unknown category → "Verification check".

(the message was already readable).
