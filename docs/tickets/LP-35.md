# LP-35 — Storage Abstraction Layer

- **Ticket:** LP-35 — Storage abstraction layer (interface + local backend, tenant-prefixed UUID paths)
- **Epic:** Epic 5 — Document Upload & Processing (first ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Built the **storage abstraction layer** — the component that saves/reads/deletes document
**bytes** behind a clean interface that hides *where* the bytes live (local filesystem in
dev; S3 in production later). This realizes the LP-15 decision (ADR-057) that a `Document`
row stores a `storage_path`, not the bytes. It is the foundation for the upload endpoint
(LP-36), PDF text extraction (LP-40), and the processing tasks (LP-42).

## What Was Built

- **`StorageBackend` interface** (`app/storage/base.py`) — an ABC with async
  `save` / `read` / `delete` / `get_url`, plus `StorageError`, the `build_storage_path`
  helper, and `_sanitize_extension` (with `SAFE_DEFAULT_EXT` + `ALLOWED_EXTENSIONS`).
- **`LocalStorageBackend`** (`app/storage/local.py`) — filesystem backend rooted at a
  configured directory, with path-traversal-safe resolution and async I/O via
  `asyncio.to_thread`.
- **Factory** (`app/storage/__init__.py::get_storage_backend`) — settings-driven
  (`storage_backend`), an `lru_cache` singleton; returns `LocalStorageBackend` for
  `"local"`, raises a clear `ValueError` for anything unknown, structured so an `"s3"`
  branch is a trivial future addition.
- **Settings** — `storage_backend` (`Literal["local", "s3"]`, default `local`) and
  `storage_local_path` (default `./storage`) already existed in `app/core/config.py`;
  documented them in `.env.example`.

## The Path Pattern

`{company_id}/{file_id}/{document_id}.{ext}` — tenant-prefixed, UUID-named.

- Every path component is a **server-controlled UUID** (company / file / document). No
  user-controlled string ever reaches the path.
- Only the **extension** derives from the filename, and it is sanitized: lowercased,
  stripped to alphanumeric, enforced against an allowlist (`pdf, jpg, jpeg, png, tif,
  tiff, heic, bin`), falling back to `bin` for anything missing/junk/unrecognized.
- `save` returns this **relative** `storage_path` (to persist on `Document.storage_path`)
  — never the absolute filesystem path.

## Security

This ticket has a real security dimension (file/path handling):

- **Server-controlled paths.** Built from UUIDs, never user input; only the (sanitized)
  extension comes from the filename — it can never carry a separator or `..`.
- **Path-traversal defense.** `LocalStorageBackend._resolve_within_root` joins the path
  onto the root, fully resolves it (`Path.resolve()`, which collapses `..` and follows
  symlinks), and confirms the result *is* the root or sits beneath it — otherwise it
  raises `StorageError` **before touching the filesystem**. `../`, `../../etc/passwd`,
  absolute paths, and escaping symlinks are all rejected (tested, including a sentinel
  file outside the root that stays untouched).
- **Root outside any web-served dir.** The storage root (`./storage`, gitignored) is not
  served statically — files are reachable only through the auth'd download endpoint
  (LP-36), so there is no direct-URL authorization bypass. `get_url` returns `None`.
- **Data, never executed.** Stored files are bytes; the extension is metadata, never used
  to decide execution.

## Decisions Made

- **ADR-112** — Storage abstraction with a local backend; S3 deferred to production
  (Phase 7), selected via settings, no calling-code changes.
- **ADR-113** — Tenant-prefixed UUID storage path; path-traversal safety; root outside
  any web-served directory.

> **ADR numbering:** LP-34 used 110–111; the next free numbers were **112–113**.

## Verification Performed

`uv run pytest tests/storage/ -v` → **35 passed**; full suite **325 passed**; ruff check,
ruff format --check, and `mypy app/` (strict) all clean.

- **Round-trip** — `save` then `read` returns the exact bytes (including null/high bytes).
- **Path format** — `save` returns `{company_id}/{file_id}/{document_id}.{ext}`, relative
  (not absolute), and writes under the root.
- **Extension sanitization** — `Paystub.PDF` → `.pdf`; `scan.JPEG` → `.jpeg`; no/empty/
  unrecognized extension → `.bin`; junk with path chars stripped/safe; ext never contains
  a separator.
- **Missing/delete contracts** — `read` of a missing path raises `StorageError`; `delete`
  removes the file; `delete` of a missing file is a no-op (idempotent).
- **`get_url`** — returns `None` for local.
- **CRITICAL — path traversal** — `read`/`delete` with `../`, `../../etc/passwd`, an
  embedded `../..`, absolute paths (`/etc/passwd`, `/tmp/evil.pdf`), and deep escapes all
  raise `StorageError`; a sentinel file outside the root is **never** written, read, or
  deleted; the legitimate deep nested path is allowed.
- **Factory** — returns `LocalStorageBackend` for `"local"`, raises a clear `ValueError`
  for an unknown backend, and is a cached singleton.

## Notes

- Blocking file I/O runs in `asyncio.to_thread` so the async interface never blocks the
  event loop (the S3 backend will use a native async client).
- The storage root (`/storage/`, `/backend/storage/`) was already gitignored from the
  LP-6/LP-8 scaffolding; the `.gitignore` anchors prevent matching the `app/storage/`
  Python package. Removed the now-redundant `app/storage/.gitkeep` (the package has real
  modules now).
- The factory's `ValueError` guard also covers `"s3"` until that backend lands.
