# LP-79 — Aggression dial (confidence-threshold gating)

- **Ticket:** LP-79 — the aggression dial: confidence-cutoff gating of display + blocking
- **Epic:** Phase 3 — Verification Engine & Calculators
- **Status:** Completed
- **Date:** 2026-06-28

## Summary

The **aggression dial** lets a processor control *how thorough* verification is by filtering the
already-computed, confidence-scored findings (LP-75/78) at three levels — **Conservative / Balanced /
Thorough** — each a **confidence cutoff**. The cutoff gates BOTH **display** (below-cutoff findings hidden)
AND **blocking** (below-cutoff findings don't block submission). It is a **read-time view filter** over
LP-78's stored findings: changing it re-filters **instantly** and **never re-runs the AI**. The dial is
**per-file** with a **user-level default** + **per-file override**, and the active level is **recorded at
submission** for auditability. The hard work (confidence scoring, the findings model, the cutoff-taking
blocking computation) shipped with LP-74/75/78; LP-79 wires the cutoff into the read path and builds the
control. See **ADR-194** and [`phase-3-verification-design.md`](../phase-3-verification-design.md) §13.

## What Was Built

- **The cutoff levels** — confirmed in `app/verification/confidence.py`: `AggressionLevel`
  (Conservative 0.8 / Balanced 0.5 / Thorough 0.0), `CONFIDENCE_CUTOFFS`, `cutoff_for_level()` (added).
- **The resolver** (`app/services/aggression.py`) — `resolve_aggression_level(loan_file, user)` (the
  per-file override if set, else the user default) + `active_cutoff(...)`.
- **The read-time filter** — the active cutoff is supplied to: the GET verification status (display set +
  `blocked` + `in_scope_open_count` + the `cutoffs` map), the submit gate (`is_file_blocked`), and the
  DTI/LTV calculators' unresolved-findings alert. No path re-runs the AI.
- **Per-file + user default + recorded-at-submission** — `users.default_aggression_level`,
  `loan_files.aggression_level_override`, `loan_files.submitted_aggression_level` (migration
  `b7f2c1d4e890`); recorded on the `READY_TO_SUBMIT` transition.
- **Endpoints** — `PUT …/verification/aggression` (set/clear the per-file override → re-filtered status,
  no AI), `GET/PUT /users/me/preferences` (the user default).
- **The dial control + the legible consequence** — a three-level segmented control on the verification
  panel (per-file, defaulting to the user default) that re-filters instantly and communicates the in-scope
  / clear↔blocked change.

## Three Levels As Confidence Cutoffs

Conservative (**0.8** — only high-confidence findings), Balanced (**0.5**, default), Thorough (**0.0** —
almost everything). A finding's confidence at/above the active cutoff → **in-scope**; below → out of scope.
Deterministic findings (confidence 1.0) are in-scope at every level. The values are config, tunable over use.

## Gates Display AND Blocking

The active cutoff is the single knob behind both:

- **Display** — below-cutoff findings are hidden; at/above are shown.
- **Blocking** — below-cutoff open findings don't block; at/above must be resolved to submit (the cutoff is
  supplied to LP-75's `is_file_blocked` / `open_in_scope_findings`).

"Resolve all" = "resolve all **at the chosen thoroughness**". LP-79 supplies the cutoff; it does not rebuild
blocking.

## Never Recolors

The dial filters by **confidence**, not **severity**. Red/yellow is intrinsic (set by the rule/generator)
and unchanged by the dial — the dial only changes which findings are in scope. Confidence (how-sure) and
severity (how-bad) are orthogonal and kept separate.

## Instant Re-Filter, No AI Re-Run

The dial is a **read-time view filter** over LP-78's stored findings. `PUT …/verification/aggression`
persists the override and returns the re-filtered status — it never enqueues the cross-source AI and incurs
no cost. One expensive pass; free thoroughness adjustment. (Verified: the dial endpoint does not call
`run_cross_source_pass.delay`.)

## Per-File + User Default + Recorded At Submission

A **user-level default** (`users.default_aggression_level`, via `PUT /users/me/preferences`) + a **per-file
override** (`loan_files.aggression_level_override`; null = use the default). Active level = override if set,
else default. On the gated `READY_TO_SUBMIT` transition the active level is recorded
(`loan_files.submitted_aggression_level`) — "cleared at <level> thoroughness".

## The Legible Consequence

Moving the dial can flip a file clear↔blocked (Thorough surfaces new findings; Conservative drops borderline
ones). The panel communicates it — "Thorough surfaced N more finding(s) to resolve" / "now showing N (M
lower-confidence hidden)" + the blocked/clear submit status — so the change reads as "I asked for more/less
scrutiny and got it", not a surprise. (The UX risk made legible.)

## Decisions Made

- **ADR-194** — aggression dial: confidence-threshold gating (display + blocking), instant re-filter (no
  AI re-run), per-file + user default, recorded at submission, never recolors, the legible consequence.

## Assumptions

- LP-81 surfaces the dial + the in-scope findings list + the calculators' alert (which use the active cutoff).
- The cutoff values (0.8 / 0.5 / 0.0) tune over use; they're config, not hard-coded at the call sites.
- Deterministic findings (confidence 1.0) are effectively always in-scope.
- The cross-source capability + the findings model are unchanged — this filters, never generates/re-runs.

## Verification Performed

- **The cutoffs** — Conservative (high) / Balanced (medium, default) / Thorough (low); a finding's
  confidence vs. the cutoff → in-scope or not.
- **Display + blocking gating** — below-cutoff hidden + non-blocking; at/above shown + blocks (via LP-75's
  computation at the supplied cutoff).
- **Instant re-filter, no AI re-run** — moving the dial re-filters the stored findings; the dial endpoint
  does not enqueue the AI (asserted `calls == []`).
- **Never recolors** — a red finding stays red across dial moves (in-scope changes, not severity).
- **Clear ↔ blocked** — a file clear at Balanced becomes blocked at Thorough (a low-confidence finding
  becomes in-scope) and clear again at Conservative.
- **Per-file + user default + recorded** — the override beats the default; the user default applies absent
  an override; the active level is recorded on the submit transition.
- **Tenant-scoped** — the dial endpoint is 404 cross-company.
- **No regression** — backend **1203 passed**; frontend **182 passed**; `ruff`/`mypy --strict` clean;
  `pnpm lint`/`typecheck`/`build` pass; the migration applies + reverses.

## Notes

- One AI pass produces every confidence-scored finding (LP-78); the dial is the free thoroughness control
  over that single expensive pass — never a trigger to re-run it.
- Confidence ≠ severity: the dial adds/removes findings from view (by confidence); it never recolors them.
- The `str_enum` helper gained an optional `name` so the two `AggressionLevel` columns on `loan_files` get
  distinct CHECK-constraint names (they'd otherwise collide).
