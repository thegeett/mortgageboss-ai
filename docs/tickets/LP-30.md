## Summary

Turned loan-file CRUD into a **workflow**: creation now generates a provisional
program-based initial needs list and logs a `FILE_CREATED` activity; updates and
deletes log activities too. This is the first real use of `NeedsItem`-from-template and
the first adoption of `log_activity` (ADR-073). Existing CRUD/listing is reused, not
rebuilt.

- **Extended** `services/loan_files.py` — did **not** create a parallel `files.py`.
- **Reused** `create_loan_file` (LP-13) and `list_loan_files`/`get_loan_file`/
  `update_loan_file`/`soft_delete_loan_file` (LP-28); id-generation and listing are
  **unchanged**. The new functions wrap/compose them.

## What Was Built

- **`services/needs_templates.py`** — `NeedTemplate` + `UNIVERSAL_NEEDS` (gov ID, pay
  stubs, bank statements, W-2s) + `FHA_NEEDS` (placeholder) + `needs_for_program(...)`.
  Marked **PROVISIONAL** with `TODO(domain)`.
- **`services/loan_files.py`** (extended):
  - `generate_initial_needs_list(db, *, loan_file_id, loan_program)` → `NeedsItem[]`
    (origin `TEMPLATE`; `None` → universal baseline).
  - `create_loan_file_with_setup(db, *, company_id, actor_user_id, ...)` → composes
    `create_loan_file` + needs list + one `FILE_CREATED` activity (needs count in detail).
  - `update_loan_file_with_activity(..., actor_user_id)` → captures old status, applies
    via `update_loan_file`, logs `STATUS_CHANGED {from,to}` or `FILE_UPDATED {changed_fields}`.
  - `soft_delete_loan_file_with_activity(..., actor_user_id)` → logs `FILE_DELETED`, then
    `soft_delete_loan_file`.
- **`app/api/loan_files.py`** — POST/PATCH/DELETE now call the `*_with_setup` /
  `*_with_activity` functions with `actor_user_id=current_user.id`; response contract
  unchanged.
- **`ActivityType`** — added `FILE_UPDATED`, `FILE_DELETED` (+ a CHECK-swap migration).

## The Provisional Needs Template

Deliberately **modest and provisional** (ADR-100): a universal baseline plus an FHA
placeholder, origin `TEMPLATE`. The authoritative program/lender needs lists are a
**Phase 0 closeout item with the domain expert (Priya)** — marked with a `TODO(domain)`
in `needs_templates.py`. It is a simple data structure so the real list is a one-place
expansion. **Not authoritative.**

## Activity Logging Adoption (first use of `log_activity`)

First operation instrumented per ADR-073/ADR-101: `FILE_CREATED` on create,
`STATUS_CHANGED {from,to}` / `FILE_UPDATED {changed_fields}` on update, `FILE_DELETED` on
delete — all with the acting user. Pure mutators stay logging-free; thin
`*_with_activity` wrappers add logging (mirroring the `create_loan_file` /
`create_loan_file_with_setup` split). Two enum values were added so the trail uses
semantically-correct types rather than overloading `NOTE_ADDED`.

> This logs status changes; it is **not** a transition state machine — any-to-any status
> moves remain allowed (enforcement would be a separate ticket).

## Decisions Made

- **ADR-099** — Loan file creation orchestration (file + needs list + activity).
- **ADR-100** — Initial needs list is a provisional program-based template (pending domain capture).
- **ADR-101** — Activity logging adopted for loan files; `FILE_UPDATED`/`FILE_DELETED` added.

> **ADR numbering:** LP-29 used 096–098; the next free numbers were **099–101**.

## Assumptions

- **No status transition state machine** here (logs changes only).
- **Authoritative needs lists pending** (Priya; Phase 0 closeout).
- **Activity-timeline / needs-list UI is later** (frontend).
- **Logging adoption elsewhere is incremental** (ADR-073) — only loan-file create/update/
  delete here.
- **External API contract unchanged** — orchestration is internal side-effects.

## Verification Performed

  PATCH status + DELETE log `STATUS_CHANGED` and `FILE_DELETED` with the actor.
- **Existing tests:** all LP-28/29 CRUD/endpoint tests still pass.
- **Migration:** `alembic upgrade head` applies; `downgrade -1` reverses the CHECK swap;
  re-upgrade works. (A naming-convention pitfall on `drop_constraint` was fixed by using
  explicit SQL with the literal constraint name.)
- **Full suite:** `uv run pytest` → **278 passed**. `ruff check`, `ruff format --check`,
  `mypy app/` (strict, 71 files) all pass.

## Notes

- **The needs template is PROVISIONAL** — pending domain capture; do not treat as
  authoritative.
- During development the migration was briefly mis-applied (Alembic's naming convention
  double-prefixed the constraint name, and a follow-up `downgrade -1` then dropped the
  wrong migration's tables). Recovered by fixing the migration to use explicit SQL,
  re-running `upgrade head` (which recreated the tables), and re-seeding dev data. The
  committed migration is correct and reversible.
