# LP-61 — Tier 1 Asset Documents

- **Ticket:** LP-61 — Tier 1 asset extractors
- **Epic:** Phase 2 — Document Handling (Tier 1 extractors)
- **Status:** Completed
- **Date:** 2026-06-18

## Summary

The second Tier-1 **extractor** batch: the asset/reserves cluster — **investment
account statement**, **retirement account statement**, and **gift letter**. Bank
statements (the most common asset doc) are already Tier 1 (LP-39c); this adds the
other major asset documents. Same established pattern as LP-60, modeled on the
bank statement extractor (the closest template). Backend only.

## What Was Built

Three extractors under `app/ai/extraction/`, their prompts, and registry entries:

- `investment_account.py` + `prompts/extraction/investment_account.txt` →
  `extract_investment_account`
- `retirement_account.py` + `prompts/extraction/retirement_account.txt` →
  `extract_retirement_account`
- `gift_letter.py` + `prompts/extraction/gift_letter.txt` → `extract_gift_letter`
- Registered all three in `EXTRACTORS` (`app/ai/extraction/__init__.py`).

## The Established Pattern (followed, not reinvented)

Each follows **LP-39a**, using the **bank statement** extractor as the closest
template (an asset doc with a masked account, a statement period, and balances): a
typed core of `TypedField`s (coerced `value` + `SourceLocation`) + an
`additional_sections` grouped catch-all; a `<Type>ExtractionResult` with
`data`/`status`/`confidence`/`reasoning`/tokens + `.failed()`; the shared tolerant
parser (`parse_typed_core`/`parse_catch_all`/`derive_status`); Decimals for money,
dates as `date`; honest nulls; metadata-only logging. Investment/retirement are
**flat** (typed core + catch-all) — the decision figure is a balance/value, not a
transaction list, so holdings go to the catch-all.

## The Typed Cores (sensible starters — refine with Priya)

Shaped for Phase 3 asset/reserves verification. The catch-all is the safety net.

- **Investment** — `institution_name`, `account_holder`, `account_number_masked`
  (sensitive), `account_type`, `statement_period_start/end`, `total_value` (the
  reserves figure). Holdings → catch-all.
- **Retirement** — `institution_name`, `account_holder`, `account_number_masked`
  (sensitive), `account_type` (401k/IRA/…), `statement_period_start/end`,
  `vested_balance`, `total_balance`.
- **Gift letter** — `donor_name`, `donor_relationship`, `recipient_name`,
  `gift_amount`, `property_address`, `no_repayment_attestation`.

## Vested-vs-Total (retirement)

A retirement statement carries two balances and the extractor keeps them
**separate**: `total_balance` (the full balance) and `vested_balance` (the portion
the borrower actually owns/can access). Vested is the reserves-relevant figure
(unvested employer funds aren't accessible; even vested funds carry penalties). The
prompt is explicitly told **not** to assume `vested == total`: if only one balance
is shown and vesting isn't mentioned, it fills `total_balance` and leaves
`vested_balance` null. Tested both ways.

## Gift-Letter Attestation (gift vs. debt)

The single most important thing about a gift letter is the **no-repayment
attestation** — the statement that the funds are a genuine gift with no expectation
of repayment. That is what makes the funds an asset rather than an undisclosed
debt. It is captured cleanly in the typed core (`no_repayment_attestation`, as
text), left **null** when the letter doesn't state it (never fabricated). The
extractor is attestation-oriented / prose-aware, like the LOE.

## Shaped for Phase 3 Asset/Reserves Verification

Assets prove the funds for the down payment, closing costs, and reserves. These
typed cores (total value / vested balance / gift amount + attestation) are the
values Phase 3 cross-checks against the **stated** assets imported from the MISMO
(Phase 1.5 — e.g. a file's stated RetirementFund / Stock / GiftOfCash). The
catch-all + the AI cross-source layer have the rest of each document.

## PII

The investment/retirement `account_number_masked` follows the **bank statement
masking discipline (ADR-149)**: the prompt masks it (last 4), it is **never
logged** (counts only), and it is masked in display. The gift letter has no account
number. Metadata-only logging across all three.

## Decisions Made

- **ADR-170** — three Tier-1 asset extractors following the established pattern
  (bank statement as the template); investment/retirement flat with masked
  accounts; retirement tracks vested AND total separately (vested not assumed
  equal to total); the gift letter is attestation-oriented (the no-repayment
  attestation distinguishes a gift from debt); typed cores are V1 starters refined
  with Priya; accuracy honestly scoped.

## Assumptions

- Field sets are sensible starters **refined with Priya**.
- **Accuracy** against real documents is validated as real documents flow through
  — **no sample investment/retirement/gift-letter documents were available** when
  this was built.
- **Phase 3** cross-checks the extracted assets against the stated MISMO assets.
- **LP-62..64** add the other Tier-1 clusters (property, borrower-info, tax
  returns).

## Verification Performed

- **No sample documents** were present (`/mnt/user-data/uploads/` absent), so the
  tests verify the **mechanism/shape**, not per-document accuracy (stated
  honestly).
- **Each extractor** — returns the typed-core + catch-all shape; per-field source
  locations present; types coerced (Decimals/dates); honest nulls; all-null core
  → `FAILED`; unparseable → `None`; `extract_*` handles success / AI failure /
  unsupported / empty gracefully.
- **Vested-vs-total** — both captured and distinct; vested stays null when only a
  total is shown.
- **Gift attestation** — donor/relationship/amount/no-repayment captured; the
  attestation stays null when absent.
- **Registry routing** — each of the three now routes to its extractor (Tier 1,
  category Assets), no longer the classified-only fallback; an extraction persisted.
- **PII** — the (masked) account number is never logged but remains available to
  the caller.
- **No regression** — the existing extractors (pay_stub/w2/bank_statement + LP-60)
  are unchanged and their suites pass.
- **Suite** — `uv run pytest` green (**817 passed**, +38 over LP-60); `ruff` +
  `ruff format` clean; `mypy` clean (122 source files).
