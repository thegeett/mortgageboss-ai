## Summary

Built the **cryptographic foundation** for authentication: pure, tested utilities
for password hashing and for issuing/verifying JWTs. No endpoints, no database
access, no FastAPI dependencies — later tickets call into these.

- **Password hashing** (`app/core/security.py`) — bcrypt (used directly, not
  passlib) plus a minimal length-only password policy.
- **JWT** (`app/core/jwt.py`) — access/refresh token creation and verification with
  PyJWT/HS256, minimal identity-only claims, and three distinct typed error classes.

Run sequentially (password half, then JWT half) with auto-approval; reviewed
thoroughly at the end before commit.

## What Was Built

### Password utilities (`app/core/security.py`)

- `hash_password(plain_password) -> str` — bcrypt with a fresh per-call salt;
  returns the modular-crypt string.
- `verify_password(plain_password, hashed_password) -> bool` — `bcrypt.checkpw`
  (constant-time digest compare); returns `False` (never raises) on a wrong password
  *or* a malformed/non-bcrypt hash, so callers treat every failure uniformly.
- `validate_password_strength(password) -> None` — raises `PasswordPolicyError` if
  under 8 characters or over 72 UTF-8 bytes. Minimal by design (length only).
- Constants `MIN_PASSWORD_LENGTH = 8`, `MAX_PASSWORD_BYTES = 72`.

### JWT utilities (`app/core/jwt.py`)

- `TokenType(StrEnum)` — `ACCESS` / `REFRESH` (the `type` claim value).
- `TokenPayload(BaseModel)` — the verified result: `subject: UUID`,
  `token_type: TokenType`.
- `create_access_token` / `create_refresh_token(subject: str | UUID,
  expires_delta=None) -> str` — sign minimal claims (`sub`, `type`, `iat`, `exp`)
  with HS256; defaults from settings, overridable via `expires_delta`.
- `verify_token(token, expected_type) -> TokenPayload` — verifies signature, expiry,
  and type; returns the typed payload or raises one of the three errors below.
- Error hierarchy: `TokenError` (base) → `TokenExpiredError`, `InvalidTokenError`
  (bad signature/key/malformed/missing claims/non-UUID subject), `WrongTokenTypeError`.

### Dependencies

`uv add bcrypt pyjwt` — `pyjwt==2.13.0` added; `bcrypt` promoted to a direct
dependency (it was already present transitively via `passlib[bcrypt]`). Both ship
their own type information, so no extra stubs were needed.

## Security Decisions

- **bcrypt directly, not passlib** (ADR-073) — avoids the known passlib/modern-bcrypt
  runtime incompatibility; bcrypt is slow-by-design, auto-salted, constant-time on
  compare. Hashing is isolated in `security.py` and swappable (e.g. to Argon2).
- **Minimal, identity-only JWT claims** (ADR-074) — the token carries `sub`/`type`/
  `exp`/`iat` only. No role/email/company/PII, because a signed token is readable and
  long-lived; authorization is looked up live from the DB (LP-24) so the system
  always acts on current truth.
- **Stateless JWT, no revocation in V1** (ADR-075) — rely on bounded access-token
  lifetime; revocation/refresh-rotation deferred to V2. A stolen unexpired access
  token is valid until expiry (documented limitation); deactivating a user blocks new
  actions immediately because authorization is live.

## Decisions Made

- **ADR-073** — bcrypt (via the `bcrypt` library) for password hashing.
- **ADR-074** — JWT auth with minimal, identity-only claims.
- **ADR-075** — Stateless JWT; no revocation/blocklist in V1.

> **ADR numbering:** LP-20 used 070–072; the next free numbers were **073–075**.

## Assumptions

- **Config comes from LP-6** — `jwt_secret_key` (min 32 chars), `jwt_algorithm`
  (`HS256`), `jwt_access_token_expire_minutes` (1440), `jwt_refresh_token_expire_days`
  (30) already exist in `Settings`.
- **These utilities are pure** — they do not read or write the `User` model; login/
  refresh endpoints are **LP-23** and the current-user lookup + HTTP error mapping is
  **LP-24**.
- **Password reset / email verification / token revocation** are out of scope (later).
- **Legacy deps** — `passlib[bcrypt]` and `python-jose` (and their type stubs) in
  `pyproject.toml` are now superseded by `bcrypt` + `pyjwt`; removing them is a small
  follow-up cleanup, left out of this ticket to keep the diff focused (noted in
  ADR-073).

## Verification Performed

- **Password** (`tests/core/test_security.py`, 9 tests): hash ≠ plaintext; same
  password → different hashes, both verify; correct password verifies True, wrong
  False; malformed/empty hash returns False (fails closed); policy rejects too-short
  and too-long and accepts a valid password; max enforced in UTF-8 bytes (multibyte
  case).
- **JWT** (`tests/core/test_jwt.py`, 12 tests): access and refresh round-trip and
  recover the subject; subject round-trips whether passed as UUID or str; wrong-type
  raises `WrongTokenTypeError` (both directions); past-expiry raises
  `TokenExpiredError`; garbage/tampered/wrong-secret raise `InvalidTokenError`;
  missing-claims and non-UUID-subject raise `InvalidTokenError`; the decoded payload
  has **exactly** `{sub, type, exp, iat}` and contains no `role`/`email`/`company_id`/
  `company`/`is_active`.
- **Full suite:** `uv run pytest` → **197 passed** (176 → +21). `ruff check`,
  `ruff format --check`, and `mypy app/` all clean.

## Notes

- **Stateless-JWT limitation** — no server-side revocation in V1 (ADR-075); a stolen
  unexpired access token is valid until `exp`. Mitigated by bounded lifetime and (via
  ADR-074) live authorization lookup; full revocation is a V2 item.
- `verify_password` and `verify_token` deliberately surface failures the same way
  each will be invoked (boolean vs typed exceptions) — `verify_password` never raises;
  `verify_token` raises one of three distinct types for LP-24 to map.
