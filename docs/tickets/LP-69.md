# LP-69 — AI Needs Reasoning: Holistic Propose-with-Reasoning + Confirm + Improve

- **Ticket:** LP-69 — AI needs reasoning (the differentiator)
- **Epic:** Phase 2 — Document Handling → the needs list
- **Status:** Completed
- **Date:** 2026-06-19

## Summary

The **brain** of the needs list and the highest-value capability in the product:
the system reasons **like a loan processor** over the **whole file** (the stated
MISMO data + the documents present + the findings + LP-67's suggestions) and
**proposes** the documents this *specific* file needs — each with **file-specific
reasoning** — for the processor to confirm. Required documents are case-by-case and
unenumerable (self-employed → tax returns + P&L; a stated gift → gift letter +
sourcing; a support obligation → payment history), so a static rule table can't
cover them — reasoning over the file is the right mechanism. Backend only; the
confirm UI is **LP-70**. See **ADR-178**.

## What Was Built

- **`app/services/needs_ai.py`** — the reasoning service:
  - `assemble_file_context(db, loan_file)` → `FileContext` — the whole picture:
    loan purpose/program, stated income + employers (via the Borrower join), assets,
    liabilities, documents present, findings (via the Document join), LP-67's
    suggestions, and **`already_covered`** (existing-need types incl. dismissed +
    present document types) — all scoped to the file's own data.
  - `propose_needs(db, loan_file)` → `list[ProposedNeed]` — assemble → the Sonnet
    call (`complete`, `settings.anthropic_model_extraction`) → parse → reconcile.
    Graceful `[]` on `AIClientError` (never raises into the caller).
  - `apply_ai_needs(db, loan_file)` / `apply_ai_needs_for_file_id(...)` — ingest the
    proposals as `NeedsItem`s via the LP-68 engine (`origin=AI_REASONING`,
    `disposition=PROPOSED`, the reasoning carried through); idempotent (skips
    existing need types/descriptions).
- **`app/ai/prompts/needs/needs_reasoning.txt`** — "reason like a mortgage loan
  processor"; the JSON contract `{"needs":[{need_description, need_type, reasoning}]}`;
  the RECONCILE rule (skip `already_covered`) + the EXPLAINABILITY rule. **A sensible
  starter — marked REFINE WITH PRIYA.**
- **`record_need_correction(db, *, need, action, note)`** in `needs_engine.py` —
  captures confirm/adjust (→ `CONFIRMED`) / dismiss (→ `DISMISSED` + waived).
- **`app/tasks/needs.py`** — the `needs.propose_ai_needs` task (initial reasoning at
  MISMO creation) + the document task now re-reasons (LP-69) after the deterministic
  match (LP-68), **both under the per-file lock**.
- **`app/api/loan_files.py`** — the MISMO import enqueues the initial reasoning
  (fire-and-forget, after commit; an enqueue hiccup must not fail the import).

## The Two Guardrails (non-negotiable)

1. **Explainability** — every proposed need carries reasoning grounded in *this*
   file's data, not boilerplate. The parser **drops** any proposal with empty
   reasoning (`_parse_proposals`).
2. **Confirmation** — proposals ingest as `disposition=PROPOSED` (NOT authoritative),
   never self-confirmed. **The AI proposes (smart) but never disposes (the human
   controls)** — the processor confirms/adjusts/dismisses in LP-70.

## Reconciliation — No Duplication

LP-69 is the **culminating** reasoner over LP-68's deterministic floor + LP-67's
findings-implications + the documents present. The assembled context lists what's
`already_covered`; the prompt is told to skip it **and** a deterministic filter
(`reconcile`) drops covered need types + dedupes. It does not re-propose the floor.

## Triggers (both per-file serialized)

- **At MISMO creation** — reason over the stated data → the initial proposed needs.
  This **absorbs the deferred smart-needs-from-MISMO** (Phase 1.5).
- **Re-proposed** as documents/findings arrive — the picture changed.

Both run as needs-updates under the LP-68 per-file Redis lock — no race.

## Improves From Corrections (V1: capture + simple use)

A processor's confirm/adjust/dismiss is captured on the need's `disposition`. The
**simple V1 use**: `already_covered` folds in existing needs incl. **dismissed**
ones, so a dismissed proposal is **not re-proposed**. A richer corrections store +
a full learning loop is a documented **future** evolution.

## Accuracy — Honestly Scoped

V1 proposes **reasoned, explainable, improvable** needs the processor confirms —
**NOT perfect out of the gate**. Quality improves via the correction signal +
refinement with Priya. The reasoning quality is **the highest-value Priya input**
("walk me through a real file — what do you chase, and why?").

## Cost / Latency / Eval

A real Sonnet call with substantial context, on every MISMO import and every
document arrival (re-reasoning is a cost to watch; debouncing is a future
optimization). Eval is real-file-driven and Priya-dependent — the mock tests verify
the **mechanism + the guardrails**, not proposal quality.

## PII

The assembled context carries PII (names, income, assets). It is sent to the AI but
**never logged** — logging is metadata/counts only.

## Tests

`tests/services/test_needs_ai.py` — 10 tests, the AI **mocked**: context assembly;
GUARDRAIL 1 (reasoning present; a no-reasoning proposal dropped); graceful AI
failure; GUARDRAIL 2 (ingest as PROPOSED, never self-confirmed); reconciliation (no
floor duplication); idempotent re-reasoning; correction-capture (dismiss → not
re-proposed; confirm → CONFIRMED); tenant scoping.

## No Regression

LP-68's engine + floor + LP-67 + the extractors are unchanged; LP-69 plugs into the
same source-agnostic ingestion path. Full suite green (996), mypy clean.

## Deferred / Future

- The full learning loop (a corrections store, prompt/weight adaptation) — V1 is
  capture + the simple "don't re-propose dismissed" use.
- Debouncing the re-reasoning on rapid document arrivals.
- Real-file eval + the Priya refinement of the prompt and the need taxonomy.
