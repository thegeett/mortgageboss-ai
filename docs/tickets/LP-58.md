# LP-58 — Three-Tier Document Model: Foundation & Routing

- **Ticket:** LP-58 — Three-Tier Document Model: Foundation & Routing
- **Epic:** Phase 2 — Document Handling (Infrastructure & Tiering)
- **Status:** Completed
- **Date:** 2026-06-18

## Summary

Opens Phase 2. Scales the document pipeline from 3 types toward ~80-100 by
introducing a **three-tier model** — the seam every later Phase-2 ticket plugs
into. This ticket builds the **mechanism**: the tier concept, a maintainable
document-type **catalog** (type → tier + category, the single source of truth),
and **tier-aware routing** in `process_document`. Tier 1 works fully via the
existing `EXTRACTORS` registry; Tier 2/3 are clean stubs for LP-65/66. The 3
existing types (`pay_stub` / `w2` / `bank_statement`) route as Tier 1, unchanged.
Backend only.

## What Was Built

- **The `Tier` enum** (`app/models/document.py`): `TIER_1` / `TIER_2` / `TIER_3`
  — a small, stable, DB-enforced set (VARCHAR + CHECK, ADR-037).
- **The `tier` model field** (`documents.tier`, nullable until classified) + an
  Alembic migration (`f1ed01fc2713`). Records how a document was *handled*; the
  type→tier *mapping* stays in the catalog, not the DB.
- **The document-type catalog** (`app/documents/catalog.py`) — the heart: a
  `dict[str, (Tier, DocumentCategory)]` with `get_tier` / `get_category` /
  `get_tier_and_category` / `is_cataloged` and a `(Tier 3, Misc)` default.
- **Catalog-driven category** — replaced the Phase-1 provisional
  `_TYPE_TO_CATEGORY` map (in `document_processing.py`) and the override path's
  `category_for_type` (in `api/documents.py`) with the catalog, so tier and
  category come from one place and never drift.
- **Tier-aware routing** (`_route_by_tier` + `_complete_classified_only` +
  `_tier2_summarize_stub` + `_tier3_analyze_stub`) in `process_document`.
- **`tier` exposed** on `DocumentResponse` (read-only; complements `category`).
- **Tests** — a new catalog suite + tier-routing tests; the existing
  pay_stub/w2/bank_statement tests still pass.

## The Three-Tier Model

The level-of-investment model — how much extraction effort a type earns:

- **Tier 1 — first-class (~18):** full structured extraction via the existing
  `EXTRACTORS` registry. High-value docs whose exact data drives Phase 3
  verification.
- **Tier 2 — recognized (~60-80):** classified + categorized + (LP-65) a short
  summary; no deep extraction.
- **Tier 3 — long-tail:** uncataloged types → a generic analyzer (LP-66).

See [`docs/document-model.md`](../document-model.md) for the full write-up.

## The Catalog (source of truth)

`app/documents/catalog.py` maps `document_type → (tier, category)`. Seeded with:

- **The 3 existing Tier-1 types** — `pay_stub`, `w2` (Income/Employment),
  `bank_statement` (Assets).
- **The planned Tier-1 types** (extractors arrive in LP-60..64; cataloged now so
  routing knows they're Tier 1): `tax_return`, `1099`, `voe`, `profit_and_loss`,
  `investment_account`, `retirement_account`, `gift_letter`,
  `purchase_agreement`, `homeowners_insurance`, `mortgage_statement`,
  `property_tax_bill`, `hoa_statement`, `drivers_license`, `divorce_decree`,
  `letter_of_explanation`.
- **A starter Tier-2 set** (a handful per category — enough to prove Tier-2
  routing): `verification_of_deposit`, `award_letter`,
  `social_security_statement`, `flood_certification`, `appraisal_report`,
  `title_commitment`, `credit_report`, `credit_explanation_letter`,
  `closing_disclosure`, `loan_estimate`, `passport`, `social_security_card`.
- **The default** — any uncataloged type → `(Tier 3, Misc)`.

A one-line edit adds/changes a type — no migration, no code branches. The catalog
**grows** in LP-59 (all ~80 types) and **refines with Priya**.

## Tier-Aware Routing

`process_document` consults the catalog **after** classification and branches by
tier. Every document takes exactly one path and reaches a terminal status:

- **Tier 1** → the existing `EXTRACTORS` registry. Extractor built → extract
  (the 3 existing types, **unchanged**). Extractor not built yet (LP-60..64) →
  **classified-only `COMPLETED`** (graceful, not a crash) — its extractor
  registers later and the same path runs extraction.
- **Tier 2** → `_tier2_summarize_stub` → `COMPLETED` (LP-65 fills the summary
  in place).
- **Tier 3** → `_tier3_analyze_stub` → `COMPLETED` (LP-66 fills the analyzer in
  place).
- **Unknown / low-confidence** → `NEEDS_REVIEW` (the pre-existing gate, before
  tier routing).

The stubs are clean insertion points: LP-65/66 fill them without restructuring.

## Extends, Not Rebuilds

- The `EXTRACTORS` registry **is** the Tier-1 mechanism — the 3 existing types
  *are* Tier 1, routing through it byte-for-byte as before.
- The classification pipeline (Haiku → route) became **tier-aware** by inserting
  the catalog lookup; it was not replaced.
- The Phase-1 category map was **reconciled into** the catalog (one source of
  truth), not duplicated.

## Decisions Made

- **ADR-167** — the three-tier model, the catalog as single source of truth, and
  tier-aware routing extending the registry; the clean Tier-2/3 stubs. Records
  the two judgment calls: a Tier-1 type with no extractor yet is **classified-only**
  (matches Phase-1 behavior, no crash), and the stubs reach `COMPLETED`.

## Assumptions

- **LP-59** fills the full ~80-type catalog + the comprehensive classification.
- **LP-60..64** add the Tier-1 extractors (each just registers in `EXTRACTORS`;
  the catalog already lists the types).
- **LP-65 / LP-66** fill the Tier-2 summary / Tier-3 analyzer stubs.
- Tiers, categories, and the catalog **refine with Priya** as the taxonomy
  settles.

## Verification Performed

- **Catalog lookups** — `get_tier` / `get_category` for the 3 existing types,
  planned Tier-1 types, the Tier-2 starter set; unknown/`None`/empty →
  `(Tier 3, Misc)`. Internal consistency (no malformed entry; Tier 3 is the
  default, never cataloged).
- **Routing by tier** — pay_stub → Tier 1 → the existing extractor (unchanged);
  a Tier-1-no-extractor type (`1099`) → classified-only `COMPLETED`; a Tier-2
  type (`credit_report`) → the summarize stub (terminal); an uncataloged type
  (`boat_registration`) → the Tier-3 analyzer stub (terminal).
- **No regression** — all 3 existing types route as Tier 1 and extract as
  before; the full pay_stub/w2/bank_statement processing suite passes.
- **Tier + category recorded** — asserted after processing on every path.
- **Every path terminal** — no document left in `CLASSIFYING` / `EXTRACTING`.
- **Migration up + down** — applies (column + `ck_documents_tier` CHECK created)
  and reverses cleanly (column + constraint dropped); verified against the dev
  Postgres.
- **Suite** — `uv run pytest` green (**686 passed**, +25 over Phase 1.5's 661);
  `ruff check` + `ruff format` clean; `mypy` clean (114 source files).

## Notes

The catalog is the maintainable knowledge base for document handling — it is
where the type→tier+category taxonomy lives and evolves. The tier-aware routing
structure is **complete**: later tickets add extractors and fill the Tier-2/3
stubs without touching the routing shape.
