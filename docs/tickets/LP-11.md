# LP-11 — Multi-Tenancy Models (Company and User)

- **Ticket:** LP-11 — Multi-tenancy models (Company and User)
- **Epic:** Epic 2 — Database & Models (third ticket)
- **Status:** Completed
- **Date:** 2026-06-10

## Summary

Created the first business tables — `Company` and `User` — establishing
**multi-tenancy**, the architecture that isolates each processing company's
data. A `Company` is the tenant; every `User` belongs to exactly one company.
Added the `scope_to_company()` query helper that all future company-owned
queries use for tenant isolation, and proved isolation with a dedicated test.

Implemented in three reviewed stages: (1) `Company` + migration, (2) `User` +
`UserRole` enum + the reciprocal relationship + migration, (3) the
`scope_to_company` helper + tests (company, user, tenant isolation) + docs.

> **Foundation fix surfaced here (ADR-037).** `User.role` was the first enum
> column to reach a real migration, which revealed that the LP-10 `str_enum()`
> helper omitted `create_constraint=True` — so non-native enums were generated
> as a plain `VARCHAR` with **no** CHECK constraint, contradicting ADR-037. Fixed
> the helper; `role` now has `ck_users_userrole` and rejects out-of-range values
> at the database level. Recorded as an amendment under ADR-037.

## What Was Built

### Models Created

| Model | Table | Key fields |
| --- | --- | --- |
| `Company` | `companies` | `name`, `slug` (globally unique, indexed), `settings` (JSON, `{}`), `is_active` |
| `User` | `users` | `company_id` (FK→companies, RESTRICT, indexed), `email` (globally unique, indexed), `hashed_password`, `first_name`, `last_name`, `role` (enum), `is_active` |

Both inherit `Base`, `UUIDMixin`, `TimestampMixin`, `SoftDeleteMixin`.
`User` has a `full_name` property. `UserRole` (`PROCESSOR`, `ADMIN`) lives next
to the model and is mapped via the `str_enum()` helper.

### Migrations

- `..._create_companies.py` — `companies` table; `pk_companies`; unique index
  `ix_companies_slug`.
- `..._create_users.py` — `users` table; FK `fk_users_company_id_companies`
  (`ondelete=RESTRICT`); non-unique `ix_users_company_id`; unique
  `ix_users_email`; CHECK `ck_users_userrole`.

Both verified: `upgrade head` → `downgrade -1` → `upgrade head` all clean;
constraint names follow the LP-9 naming convention; tables inspected in psql.

### Company-scoping helper

`scope_to_company(stmt, model, company_id)` in `app/models/helpers.py` adds
`WHERE company_id = :id`. Typed against a `CompanyScoped` protocol so only
models with a `company_id` are accepted (the tenant root `Company` is correctly
rejected at type-check time). Composes with `only_active()`.

## Multi-tenancy Pattern

- The **tenant is a `Company`**; every business entity links back to one.
- **Every query touching company-owned data MUST be scoped** with
  `scope_to_company()` — forgetting is a tenant data leak. Scoping is explicit,
  not magic (ADR-043).
- **Email is globally unique** (login identity, ADR-042) — the exception to the
  general rule that tenant-owned "unique" fields are unique *per company*.
- **Soft delete + `RESTRICT`** — companies/users are never hard-deleted in
  normal flow (ADR-044).

## Decisions Made

- **ADR-041** — Multi-tenancy via `company_id` scoping from day one.
- **ADR-042** — Email globally unique (not per-company).
- **ADR-043** — Explicit company-scoping helper (no automatic filtering).
- **ADR-044** — Companies/users soft-deleted, FK `ondelete RESTRICT`.
- **ADR-037 (amended)** — `str_enum()` corrected to emit the CHECK constraint
  (`create_constraint=True`) as the ADR originally intended.

## Assumptions

- Sister's processing company is the first tenant; real seed data lands in LP-48.
- `hashed_password` column exists but **no auth logic** here — login/JWT is LP-22.
- PostgreSQL is running locally (dev DB `mortgageboss_dev`); migrations applied
  against it and verified.
- Tests use `create_all`, not migrations (ADR-039), so the new tables are picked
  up automatically once registered in `app/models`.

## Verification Performed

- `uv run pytest` → **28 passed** (17 prior + 11 new).
- **Tenant isolation** (`test_tenant_isolation.py`): a query scoped to Company A
  returns only A's users, never B's, and vice versa; result sets are disjoint;
  every returned row's `company_id` matches. Also composes with `only_active`.
- **Email global uniqueness**: the same email under a *different* company still
  raises `IntegrityError` (expected-failure INSERTs wrapped in `begin_nested()`
  SAVEPOINTs so they don't poison the rollback-isolation transaction).
- **Role enum**: round-trips to `UserRole.ADMIN`; raw stored value is `"admin"`;
  the new `ck_users_userrole` CHECK rejects an out-of-range value (`'banana'`)
  at the DB level.
- **Relationships**: `user.company` and `company.users` load via `selectinload`
  (lazy loading is unavailable under async sessions).
- `uv run ruff check`, `ruff format --check`, `uv run mypy app/` — all pass.

## Notes

- The `Company.users` relationship was added in Stage 2 (with `User`), not
  Stage 1: a `back_populates` relationship to an unregistered class breaks
  `configure_mappers()`, which Alembic autogenerate triggers.
- Slug uniqueness is enforced by a **unique index** (`ix_companies_slug`) from
  `unique=True, index=True` — one object that both enforces uniqueness and
  serves lookups, rather than a separate `UniqueConstraint` + index.
- Autogenerated migrations were cleaned to the LP-9 house style (modern union
  syntax, `collections.abc.Sequence`, descriptive docstrings, `# pragma:
  allowlist secret` on revision ids).
