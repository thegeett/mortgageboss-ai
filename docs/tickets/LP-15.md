# LP-15 — Document Model

- **Ticket:** LP-15 — Document model with categories and processing lifecycle
- **Epic:** Epic 2 — Database & Models (seventh ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Created the **`Document`** model — one row per uploaded file (pay stub, bank
statement, W-2, …) attached to a loan file. The row holds **metadata and a
storage path**; the binary bytes live in the storage backend, never in the
database (**ADR-055**). It introduces a document **category** enum (the
processor's eight library buckets), a flexible **document_type** string (the
~100-type set is a Phase 2 concern), a processing-lifecycle **status** enum, and
upload **provenance** (source + nullable uploader).

This ticket builds the **record**, not the processing. Upload (LP-36),
classification (LP-38), and extraction (LP-39) come later and write to this
model.

Implemented in **two reviewed stages**: (1) the `Document` model + three enums +
the loan-file relationship + migration, (2) tests + docs.

## What Was Built

### Model (`app/models/document.py`)

Three enums (`DocumentCategory`, `DocumentStatus`, `UploadSource`) and the
`Document` model (Base + UUID/Timestamp/SoftDelete mixins).

| Column | Type | Notes |
| --- | --- | --- |
| `loan_file_id` | UUID FK → loan_files | required, indexed, `ondelete=CASCADE` (owned child) |
| `original_filename` | `VARCHAR(256)` | required |
| `mime_type` | `VARCHAR(256)` | required |
| `file_size_bytes` | int | required |
| `storage_path` | `VARCHAR(1024)` | required; path in storage backend, not the bytes |
| `document_type` | `VARCHAR(64)` | nullable, **indexed**, flexible string (no CHECK) |
| `category` | `DocumentCategory` enum | nullable, VARCHAR + CHECK |
| `classification_confidence` | float | nullable, `[0.0, 1.0]` |
| `status` | `DocumentStatus` enum | default `PENDING`, non-null, indexed, VARCHAR + CHECK |
| `processing_error` | text | nullable, set when `FAILED` |
| `upload_source` | `UploadSource` enum | required, VARCHAR + CHECK |
| `uploaded_by_user_id` | UUID FK → users | nullable, indexed, `ondelete=RESTRICT` |

Relationships: `loan_file` (back_populates `documents`), `uploaded_by`
(one-directional to User). `LoanFile` gained `documents`
(`cascade="all, delete-orphan"`).

### Categories (8) and lifecycle

- **Categories:** `assets`, `borrower_info`, `credit`, `disclosures`,
  `income_employment`, `property`, `misc`, `custom`.
- **Lifecycle:** `PENDING → CLASSIFYING → CLASSIFIED → EXTRACTING → COMPLETED`,
  plus `FAILED` (reason in `processing_error`) and `NEEDS_REVIEW` (low-confidence
  classification awaiting processor correction). Set directly by async tasks;
  no state machine in V1.

### Migration (`a513e98e399f_create_documents.py`)

Creates `documents` with both FKs (CASCADE to loan_files, nullable RESTRICT to
users), CHECK constraints on `category`/`status`/`upload_source`, and indexes on
`loan_file_id`, `document_type`, `status`, `uploaded_by_user_id`. Hand-cleaned to
match the repo's migration style.

## Decisions Made

- **ADR-053** — `document_type` is a flexible indexed string (governed at the app
  layer); `category` is a DB-enforced enum. Categories are small and stable;
  types are large and evolving (finalized Phase 2), so an enum would force a
  migration per new type.
- **ADR-054** — `DocumentStatus` processing lifecycle; tasks set status directly,
  no state machine in V1; `NEEDS_REVIEW`/`FAILED` drive the review/error UI.
- **ADR-055** — Storage path in the DB, bytes in the storage backend; storing
  binaries in Postgres bloats the DB and backups.
- **ADR-056** — `UploadSource` enum + nullable `uploaded_by_user_id` (set only for
  `USER_UPLOAD`); RESTRICT keeps an uploader from being hard-deleted out from
  under their documents.

> **ADR numbering:** the ticket text referenced ADR-053/054 for LP-14 and asked
> for ADR-055…058. In reality LP-14 used ADR-051 (encryption) and ADR-052
> (transitive scoping *and* owned-child cascade). The next free numbers were
> **053–056**, used here; references to scoping/cascade point at the real
> **ADR-052**.

## Assumptions

- **The full ~100 document-type list is Phase 2** — `document_type` is a flexible
  string until then (ADR-053).
- **The storage backend is LP-35** — this ticket only stores `storage_path`; no
  upload, no byte handling.
- **Processing tasks are Epic 5** — classification/extraction write `category`,
  `document_type`, `classification_confidence`, and advance `status`.
- **`storage_path` is `VARCHAR(1024)`** (a deliberate widening from the spec's
  `MediumStr`/256) to fit S3 keys and nested paths; `document_type` is
  `VARCHAR(64)` (a slug/label).

  `documentstatus`, `uploadsource`), CASCADE FK to loan_files, RESTRICT FK to
  users, indexes on `loan_file_id`/`document_type`/`status`/`uploaded_by_user_id`.

  works; autogenerate drift check is empty (model and migration agree).
- `tests/models/test_document.py` — 14 tests: creation/all fields, `PENDING`
  default, flexible `document_type`, all 8 categories accepted, CHECK rejection
  for all three enums, `uploaded_by` set for `USER_UPLOAD` and null for
  `BORROWER_INBOX`, relationships, multiple docs per file, soft delete +
  `only_active`, **cascade delete with the loan file**, and **tenant isolation**
  through the file.
- Full suite: **96 passed** (82 → +14). `ruff` + `mypy app/` clean.

## Notes

- The CHECK-rejection tests bypass the Python enum with a raw SQL `UPDATE` inside
  a `begin_nested()` savepoint, proving the database (not just the app) enforces
  the constraint.
- `uploaded_by` is one-directional — `User` gets no `documents` collection (not
  needed in V1); add `foreign_keys=` if a second user FK is ever introduced.
