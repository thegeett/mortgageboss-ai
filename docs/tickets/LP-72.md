# LP-72 — Document Detail Extensions + Standard Naming (package groundwork)

- **Ticket:** LP-72 — Tier-aware detail + standard naming + package qualification
- **Epic:** Phase 2 — Document Handling (last feature ticket)
- **Status:** Completed
- **Date:** 2026-06-25

## Summary

The last Phase-2 feature: it SURFACES the tier work (LP-58..66) in the UI and adds the
two pieces that make documents **package-ready** — a consistent derived name and a
computed package fitness. Three related pieces, all making a document readable +
identifiable + assessable. See **ADR-185**.

## What Was Built

- **Tier-aware document detail** (`document-drawer.tsx` → `TierDetail`/`TierBody` +
  `generic-analysis-view.tsx`): the detail view branches on `tier`.
- **Standard naming** (`app/documents/naming.py`): a derived display name + the per-type
  rules; exposed as `standard_name` on the document response.
- **Package qualification** (`app/documents/staleness.py` → `package_qualification`):
  computed per document; exposed as `package_qualification` on the response. A subtle
  "Package-ready" indicator in the list + drawer.

## Tier-Aware Detail

The detail view ADAPTS to the tier — the proportional-investment philosophy made visible:

- **Tier 1** → the structured EXTRACTED FIELDS (deep, type-specific — the existing
  `ExtractionView`: employer/pay/gross/net, balances, tax-return lines).
- **Tier 2** → the recognition SUMMARY (LP-65) + the category (light).
- **Tier 3** → the generic analyzer's FINDINGS (parties / dates / amounts / findings,
  LP-66) + summary (flexible).

Extends the LP-43 drawer (branches on `tier`), not a rebuild; pending/failed/loading
states degrade gracefully; PII stays masked.

## Standard Naming

A derived `{Type}_{KeyIdentifier}_{Date}` name (no spaces) from the type + extracted
data — e.g. `Pay-Stub_Thermofisher-PPD_2026-05-22`,
`Bank-Statement_Bank-of-America_2026-04-30`, `Tax-Return-1040_Mahesh-Chhotala_2024`. The
identifier + date come from the typed-core extraction (per-type rules). A sensible
`{Type}_{UploadDate}` fallback covers sparse data (Tier 2/3, extraction pending, missing
identifier). It is a **display/derived** name — **the stored file is untouched**. Only
non-PII fields feed it (never SSN, account number, or DOB). Surfaced as the scannable
primary label in the list + drawer (the raw filename shown as secondary). The rules are
a plain config (`NAME_RULES`) — they refine with use / Priya.

## Package-Qualification Groundwork

`package_qualified` = CURRENT (LP-71 versioning) + FRESH (LP-71 staleness) + TYPED
(recognized) + EXTRACTED (processing succeeded — terminal `COMPLETED`). It consumes
LP-71's signals + the extraction state and reports the first failing criterion
(superseded / stale / untyped / not_extracted). **Groundwork** — LP-72 makes each
document KNOW its readiness; **Phase 6** assembles the package from qualified documents.
A subtle "Package-ready" indicator surfaces it (informational); nothing assembles or
renders a package.

## Surfaces — Doesn't Re-Do

Presents the existing tier extraction (LP-60..66) tier-appropriately + consumes LP-71's
current/fresh signals + the extraction state. **No re-extraction, no package assembly,
no Phase-3 cross-verification.** The standard name is display-only.
