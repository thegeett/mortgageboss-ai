# LP-79.5 — Soft-delete a loan file (processor)

- **Ticket:** LP-79.5 — soft-delete a loan file from the UI (processor) + confirmation
- **Epic:** Phase 3 — Verification Engine & Calculators (usability gap)
- **Status:** Completed
- **Date:** 2026-06-29

## Summary

There was no way to delete a loan file from the UI — clutter (test/duplicate files) accumulated with no
way to remove it. This adds **soft-delete for processors**: a processor can delete a loan file in their own
company; it disappears from their views while the row + its children + the audit trail are **preserved and
recoverable**. **Hard-delete (permanent purge) is deferred, admin-only future work** — nothing is built for
it now. The backend already supported soft-delete end-to-end; LP-79.5 is mostly the **UI affordance + a
named confirmation**, plus two hardening tests. See **ADR-195** and [`api.md`](../api.md) (loan-files DELETE).

## Soft vs. Hard (and the deferral)

- **Soft-delete (this ticket)** — `DELETE /loan-files/{id}` sets `deleted_at`; the row, its children
  (documents, findings, verification, needs, communications), and the `FILE_DELETED` audit entry all
  survive. The file just leaves the processor's views. Recoverable.
- **Hard-delete (deferred)** — permanent purge is a deliberate, privileged action. A processor must not be
  able to truly destroy a mortgage record (compliance / audit / recovery). **Built nothing for it now**; the
  soft-delete design leaves room for a future admin-only purge.
- **No restore/trash view yet** — deferred; soft-delete preserves the data so a restore can come later.

## What Was Built

- **Backend (confirmed already complete, hardened):** `DELETE /loan-files/{id}` →
  `soft_delete_loan_file_with_activity` (sets `deleted_at`, logs `FILE_DELETED`), `only_active` exclusion in
  the `_scoped` base query (dashboard/list/get/search/counts all exclude soft-deleted), `204` on success,
  `404` cross-company / already-deleted. Added two tests: **already-deleted → 404 (graceful, not a crash)**
  and **soft-delete preserves the row + children (not a hard cascade)**.
- **Frontend (new):** `useDeleteLoanFile()` (invalidates the list queries); a reusable **`DeleteFileDialog`**
  (names the file + impact + recoverability; confirm fires the DELETE, toasts, closes; cancel is a no-op;
  loading/error); a **dashboard row overflow menu** ("⋯" → Delete file) that does not trigger row
  navigation; a **file-header overflow menu** (Delete file → redirects to the dashboard on success).

## The Soft-Delete Endpoint (confirmed)

`DELETE /api/v1/loan-files/{id_or_display_id}` — `204`. Sets `deleted_at` (never removes the row/children),
logs `FILE_DELETED` with the actor. Tenant-scoped: a cross-company id is a `404` (existence never revealed).
Any processor (or admin) in the owning company may delete (soft-delete is **not** gated to admin). Deleting
an already-deleted file resolves nothing (it's invisible via `only_active`) → a clean `404`, never a crash.

## Views Exclude Soft-Deleted Files (confirmed)

Every read goes through `_scoped(company_id)`, which applies `scope_to_company` + `only_active`
(`deleted_at IS NULL`). So `list_loan_files` (dashboard, search, filters, counts/total) and `get_loan_file`
(the detail route) both exclude soft-deleted files. `GET` a soft-deleted file → `404`.

## The UI Affordance + Confirmation

- **Two entry points:** the dashboard table's per-row overflow menu (`components/dashboard/file-table.tsx`)
  and the file-workspace header's overflow menu (`components/file/file-header-actions.tsx`). The row click
  still navigates; the menu trigger `stopPropagation`s so opening/using it never navigates.
- **The confirmation dialog** (`components/file/delete-file-dialog.tsx`) names the file (borrower +
  `display_id`) and what's affected ("…and all its documents, extracted data, and findings from your
  dashboard. The file is preserved — this can be undone by an admin."). Confirm → `DELETE` → success toast +
  the list refetches (the row disappears); on the detail page it redirects to the dashboard. Cancel does
  nothing. A deleting state disables both buttons. PII: the borrower name uses the same value the dashboard
  already shows (no new exposure).

## Decisions Made

- **ADR-195** — soft-delete for processors now; hard-delete (purge) admin-only future work; why soft
  (compliance / audit / recovery); the named-confirmation UI; no restore/trash view yet.

## Assumptions

- Hard-delete + a restore/trash view are future work; soft-delete preserves everything they'd need.
- Naming "its documents" without a hard count is accurate and avoids an extra fetch; the dialog enumerates
  documents + extracted data + findings as what leaves the dashboard.
- The cascade model is unchanged — children are scoped through the file, so a soft-deleted file's children
  are simply unreachable (sufficient; not destroyed).

## Verification Performed

- **Soft-delete sets `deleted_at`; row + children preserved (not hard-deleted)** — service test.
- **Already-deleted → `404` (graceful, not a crash)** — endpoint test.
- **Tenant-scoped (`404` cross-company); audit-logged (`FILE_DELETED` with actor); excluded from list/get** —
  existing tests (confirmed).
- **Frontend:** the dialog names the file + impact + recoverability; confirm triggers the delete + toasts +
  closes + `onDeleted`; error toasts and does not close; cancel is a no-op; the deleting state disables the
  buttons; the table renders a per-row action that does not navigate.
- **No regression:** backend **1205 passed**; frontend **193 passed**; `ruff`/`mypy --strict` clean;
  `pnpm lint`/`typecheck`/`build` pass.

## Notes

  LP-79.5 is the UI + the two hardening tests + the soft-vs-hard decision recorded.
- Soft-delete is **not** gated to admin — any processor in the company can do it (it's reversible).
