## What Was Built

### Seed script (`backend/app/scripts/seed_dev.py`)

Async, idempotent, standalone. `_get_or_create_company` (keyed on the unique `slug`) and
`_get_or_create_user` (keyed on the globally-unique `email`) skip existing rows; `seed()`
creates the company + admin + processor, **commits its own transaction** (a script, not a
request handler), then prints the credentials with a DEV-ONLY notice. Passwords come from
`SEED_*` env vars with dev defaults and are hashed via `hash_password` (bcrypt).

### Onboarding documentation (`docs/onboarding-and-tenancy.md`)

The full model: actors (platform → companies; admin → processors; processor → loan work),
the tenancy model (isolation via `company_id` from the token, **not** subdomains), the
intended future flow (platform onboards company → admin invites processor → processor sets
own password via a capability-token email link), the subdomain model (Phase 7; `slug` is
the future label; branding/UX not security), the staged build plan with dependencies, the
security rationale (no self-registration → controlled tenant assignment), and a prominent
**current-vs-future status table**.

## The Onboarding Model (summary)

Companies are tenants onboarded by the platform; a company admin provisions processors;
invited users inherit the admin's company; there is no public self-registration. Tenant
isolation is enforced from the authenticated user's `company_id` (LP-24), so it works
today — subdomains (`slug`-based) are future branding/UX, not the security boundary.
Invitation and password-reset will be capability-token flows (ADR-036), deferred until
email (Phase 4). Full detail: [`docs/onboarding-and-tenancy.md`](../onboarding-and-tenancy.md).

## Decisions Made

- **ADR-088** — Company-centric, invite-only tenancy (no public self-registration).
- **ADR-089** — Minimal dev seed now; staged onboarding build.
- **ADR-090** — Invitation and password-reset links are capability tokens (deferred).

> **ADR numbering:** LP-25 used 084–087; the next free numbers were **088–090**.

## Assumptions

- **Full onboarding flow is deferred** — depends on email (Phase 4), capability-token
  infra, and DNS/TLS (Phase 7).
- **Comprehensive seed is LP-48** — this is the minimal slice.
- **Subdomain routing is Phase 7** — `slug` is the future identifier; isolation already
  works via `company_id`.
- **Dev-default passwords are conveniences, not secrets** — env-overridable, DEV-ONLY.
- **`.com` (not `.test`) seed emails** — the login endpoint's `EmailStr` rejects reserved
  TLDs (`.test`), so reserved-TLD accounts couldn't log in; using `.com` keeps the seeded
  accounts usable through the real auth flow. (Deviation from the ticket's example
  defaults, for correctness.)

## Verification Performed

- **First run:** created the company + admin + processor; printed credentials.
- **Second run:** idempotent — all three reported "already existed", no duplicates, no
  crash.
- **psql:** the `demo` company and both users exist with `hashed_password` populated and
  bcrypt-prefixed (`$2b$12$`) — never plaintext; exactly **2** demo users after two runs.
- **Login sanity (real endpoint):** `POST /api/v1/auth/login` succeeds for both
  `admin@demo.com` / `adminpass123` (role `admin`) and `processor@demo.com` /
  `processorpass123` (role `processor`), returning an access token + the correct user.
- **Checks:** `ruff check`, `ruff format --check`, and `mypy app/` (strict, 62 files) all
  pass. `pre-commit run --all-files` clean.

## Notes

- **DEV-ONLY default passwords** (`adminpass123` / `processorpass123`) — local
  conveniences, overridable via `SEED_*` env vars; production users are provisioned per
  the onboarding doc. Carry `# pragma: allowlist secret` on the default literals so
  detect-secrets passes.
- The script **commits** (standalone), unlike services which `flush` and let the request
  handler commit.
