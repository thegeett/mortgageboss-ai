# LP-75 — Findings Model Extension: Confidence, Resolution States & Blocking

- **Ticket:** LP-75 — Findings model extension (the verification finding)
- **Epic:** Phase 3 — Verification Engine & Calculators (Arc A)
- **Status:** Completed
- **Date:** 2026-06-27

## Summary

Turns the shared LP-66 `Finding` into the full **verification finding** by **extending the
existing model in place** — adding the four dimensions verification needs (confidence,
resolution states, blocking, source location), confirming the uniform shape across all three
generators, and building the APPLY→recompute hook. Backend only (the dial is LP-79, the UI is
LP-81). See **ADR-189** and [`phase-3-verification-design.md`](../phase-3-verification-design.md)
§9.

## Extends LP-66 — Not A New Model

The locked architecture rests on **one** findings model that both generators (LP-74
deterministic, LP-78 AI cross-source) feed and the human resolves uniformly — plus the Phase-2
document findings (LP-66). So LP-75 **extends** that uniform-feedstock model rather than forking
it. Concretely, it **reconciles** with the existing shape instead of duplicating:

- the existing `resolution_status` enum is **extended** with `APPLIED`/`OVERRIDDEN` (not a
  parallel second resolution field);
- the existing `resolution_note` is **reused** as the override reason (required for OVERRIDDEN);
- the existing `status` (red/yellow/green) **is** the severity;
- the LP-74 `origin` field is **extended** with `document_analysis` (not re-added).

Only genuinely new dimensions became new columns: `confidence`, `source_page`,
`source_snippet`, `applied_record`.

## The Four Extensions

1. **Confidence** (`confidence: float` ∈ [0, 1], DB CHECK `ck_findings_confidence_range`) — how
   sure the system is the finding is real; the **aggression dial**'s substrate (LP-79) and the
   blocking input. Deterministic threshold findings are **certain** (`DETERMINISTIC_CONFIDENCE =
   1.0` — the math is exact); AI cross-source findings (LP-78) **vary**. Defaults to 1.0.
2. **Resolution states** — `OPEN` (default) → `APPLIED` | `OVERRIDDEN`. APPLIED *incorporates the
   finding into the structured data*; OVERRIDDEN *dismisses it with a recorded reason* (required,
   enforced in the service). The legacy LP-17 states remain for the document flow; any non-OPEN
   state is resolved for blocking. **No finding is silently ignored**; resolutions are
   activity-logged.
3. **Blocking** — `is_file_blocked`: a file is blocked from ready-to-submit while it has any
   **open in-scope** finding (actionable red/yellow, open, confidence ≥ the active cutoff). Green
   findings (passes) never block; resolving (apply/override) unblocks; a finding below the cutoff
   is out of scope. LP-75 owns the computation + a Balanced default; **LP-79's dial sets the
   cutoff**.
4. **Source location** — `source_page` + `source_snippet` (page + **verbatim** snippet): the
   trust/audit anchor (click a finding → the exact document line), building on extraction's
   per-field source. Bounding boxes deferred; page + snippet is V1.

## The Uniform Shape

`FindingOrigin` = `deterministic_rule` / `ai_cross_source` / `document_analysis` — the three
generators share **one** shape: type, amount, source document, page, snippet, confidence,
reasoning, severity (`status`), resolution, and the origin provenance marker. The dial, the UI
(LP-81), and the resolution flow treat findings **uniformly** — they don't care *how* a finding
was generated, only that it's a finding with confidence/severity/source/resolution. "Two
generators, one findings model" made concrete in the data. The source/origin distinguishes
**provenance** (for transparency); handling is uniform.

## The APPLY → Recompute Hook

APPLYING a finding **changes the structured data**: `apply_finding` performs the change the
finding declares (`details["apply"]`) — the canonical case is `add_liability` (an undisclosed
obligation **added to liabilities**) — records it on `applied_record`, and calls
`mark_recompute_needed` (the explicit seam). That structured-data change is the trigger of the
**AI↔deterministic interlock** (AI surfaces → human applies → deterministic recomputes). LP-75
builds the hook + the observable change; the **full** wiring is LP-78 (the loop) + the
calculators (LP-76/77).

## Verification Performed

- **Confidence** — every finding carries it; LP-74's deterministic findings are certain (1.0);
  the field accepts varying values (0.4 / 0.95) for AI findings.
- **Resolution** — open → APPLIED; open → OVERRIDDEN (blank reason rejected); exactly
  open/applied/overridden (no third "ignore"); override reasons activity-logged.
- **Blocking** — an open in-scope finding blocks; resolving unblocks; a finding below the cutoff
  doesn't block (and Thorough's cutoff brings it back into scope); green findings don't block;
  the ready-to-submit PATCH returns 409 while blocked; per-file (no leakage).
- **Source location** — a finding carries page + verbatim snippet.
- **Uniform shape** — deterministic / AI / document findings fit one shape with the origin marker.
- **APPLY→recompute hook** — applying an obligation finding adds a `StatedLiability` (the
  observable structured-data change) and records `applied_record`.
- **No regression** — full suite **1108 passed**; `ruff`/`mypy --strict` clean; the migration
  applies + reverses (constraint/index names converge with `create_all`).

## Notes

- **The uniform model is the architectural through-line** — one shape both generators feed and the
  human resolves uniformly; extend it, never fork it.
- **APPLIED-changes-structured-data is the interlock** — the point where an AI-surfaced correction
  feeds the deterministic recompute (the human-in-the-loop loop).
