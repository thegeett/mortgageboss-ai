# LP-16 — Extraction Model

- **Ticket:** LP-16 — Extraction model with versioning and JSON data storage
- **Epic:** Epic 2 — Database & Models (eighth ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Created the **`Extraction`** model — the structured data AI pulls out of a
document. Its defining features:

1. **JSON storage typed at the app layer** — extracted fields live in a single
   `extracted_data` JSON column whose structure is governed by document-type-
   specific Pydantic schemas at the application layer (Phase 2), **not** a
   generic field bag (ADR-057).
2. **Versioning** — a document can be re-extracted; each run is a `version`, with
   exactly one `is_current` per document enforced by a **partial unique index**
   (`UNIQUE (document_id) WHERE is_current`), history preserved (ADR-058).

Also added the `create_extraction_version` service that performs the version flip
safely, and `Document.current_extraction` convenience.

This ticket builds the **record + versioning mechanism**, not the extraction
logic (LP-39 / Phase 2).

Implemented in **two reviewed stages**: (1) the model + enum + partial unique
index + migration, (2) the versioning helper + tests, then docs.

## What Was Built

### Model (`app/models/extraction.py`)

`ExtractionStatus` enum + the `Extraction` model (Base + UUID/Timestamp/SoftDelete
mixins).

| Column | Type | Notes |
| --- | --- | --- |
| `document_id` | UUID FK → documents | required, indexed, `ondelete=CASCADE` (owned child) |
| `version` | int | required, sequential per document from 1 |
| `is_current` | bool | required, default True |
| `extracted_data` | JSON | required, default `{}`; document-type-specific, typed in Python |
| `extraction_status` | `ExtractionStatus` enum | required, VARCHAR + CHECK |
| `model_used` | `VARCHAR(64)` | nullable, e.g. `"claude-sonnet-4-5"` |
| `tokens_used` | int | nullable, cost tracking |
| `cost_estimate` | float | nullable; sub-cent estimate, so `Float` not `Money` |
| `error_detail` | text | nullable, set on `FAILED` |

Partial unique index `uq_extractions_document_id_current` via `__table_args__`
(`Index(..., unique=True, postgresql_where=text("is_current"))`).

### Versioning mechanism

`version` + `is_current` + the partial unique index together give: monotonic
per-document versions, exactly one current row (DB-enforced), and full history.
`create_extraction_version` (`app/services/extractions.py`):

1. Loads the document's current extraction (if any) and sets `is_current = False`.
2. **Flushes** that demotion — frees the partial index.
3. Computes `next_version = max(version) + 1` (over all rows, so numbers never
   repeat), or 1 for the first.
4. Inserts the new row with `is_current = True` and flushes.

The demote-then-flush-then-insert order is what keeps the partial unique index
from being momentarily violated.

### Storage approach (JSON typed at the app layer)

`extracted_data` is a single JSON column, validated by Phase 2 Pydantic schemas
in Python — **not** the POC's generic `ExtractedField` rows. One flexible table,
type safety in Python, no schema churn as the ~100 document types evolve. Bank-
statement transactions live as a nested list inside `extracted_data` (no separate
table in V1).

### Migration (`aa3e537ea73f_create_extractions.py`)

Creates `extractions` with the CASCADE FK to documents, the `extraction_status`
CHECK, an index on `document_id`, and the partial unique index (the WHERE clause
emitted correctly by autogenerate). Hand-cleaned to the repo style.

## Decisions Made

- **ADR-057** — Extracted data as JSON, typed at the application layer (Pydantic,
  Phase 2); deliberately not the POC's generic field bag.
- **ADR-058** — Extraction versioning with one current per document, enforced by a
  partial unique index; new versions demote the old current first.
- **ADR-059** — Bank-statement transactions stored in `extracted_data` JSON; no
  separate transactions table in V1.

> **ADR numbering:** the ticket referenced ADR-059…061. The actual next free
> numbers were **057–059** (LP-15 used 053–056). Used 057–059; references to
> scoping/cascade point at the real **ADR-052**.

## Assumptions

- **Document-type-specific Pydantic schemas are Phase 2** — they validate
  `extracted_data`; this ticket stores it as JSON.
- **Extraction logic is LP-39 / Phase 2** — `create_extraction_version` is the
  persistence entry point the task will call.
- **Transactions live in JSON** — no separate table until cross-transaction
  querying is needed (ADR-059).
- **`cost_estimate` is `Float`** (sub-cent estimates), a deliberate departure
  from the repo's Money-as-Decimal rule for ledger amounts.

## Notes

- The one-current invariant is proven at the DB level: the test directly inserts
  a second `is_current=True` row and asserts the partial unique index rejects it
  (not just app convention).
- `current_extraction` is a Python property over the loaded `extractions`
  collection (the simpler Option A), not a second filtered relationship — load
  `extractions` (e.g. `selectinload`) before accessing it.
