# LP-90 — Expose valuation_amount on the Overview (editable) + make the LTV "Appraised value" source explicit

- **Ticket:** LP-90
- **Epic:** Phase 3 — Verification Engine & Calculators
- **Status:** Completed
- **Date:** 2026-06-30

## Summary

A focused full-stack fix for a **hidden-field bug**: the LTV calculator's appraised-value basis reads
`appraised = valuation_amount or estimated_value` (`valuation_amount` wins), but `valuation_amount` (the MISMO
`PropertyValuationAmount`) was never exposed in the property read schemas or the Overview editor — only
`PropertyUpdate` accepted it. So it silently shadowed the editable `estimated_value`: editing "Estimated value"
on the Overview didn't move the LTV whenever `valuation_amount` was non-null (i.e. on any MISMO-imported file),
with no on-screen explanation. See **ADR-215**.

## The bug (traced read-only first)

For the real MISMO file: subject property had `SalesContractAmount` $1,380,000 → `purchase_price` and
`PropertyValuationAmount` $1,380,000 → `valuation_amount`, but no appraised value. The LTV's "Appraised value"
basis resolved to `valuation_amount` ($1,380,000). The Overview only let you edit `estimated_value` and
`purchase_price` — so an edit to "Estimated value" changed a field the LTV's appraised basis **does not read when
`valuation_amount` is set**. The number that drove the basis was invisible and uneditable.

## What changed

### PART 1 — Backend: expose `valuation_amount` (read)

- `schemas/property.py` — added `valuation_amount` to `PropertyResponse` (the PATCH response).
- `schemas/loan_file.py` — added `valuation_amount` to `PropertyPublic` (the loan-file detail the Overview reads).
- The PATCH/audit/mark-stale path is already generic (LP-80.5): editing `valuation_amount` records the from→to
  values in the activity log and marks verification stale, with no new endpoint code.

### PART 2 — Frontend: make it editable on the Overview (the core fix)

- `lib/types/loan-file.ts` — `valuation_amount` added to `PropertyPublic`.
- `components/file/overview/overview-cards.tsx` — a read-only "Valuation amount" row in the Subject Property card.
- `components/file/overview/property-editor.tsx` — a "Valuation amount" `money` field in the inline editor.
- The existing `useUpdateProperty` mutation already invalidates the `dti`/`ltv`/`verification` query keys, so the
  edit flows straight through to the LTV. **The field the LTV reads is now the field the processor can edit.**

### PART 3 — Frontend: make the LTV basis source explicit

- `schemas/ltv.py` + `lib/types/ltv.ts` — added `appraised_value_source` to `LtvCalculation`
  (`"valuation_amount"` | `"estimated_value"` | `null`).
- `services/ltv.py` — `_auto_value_lines` now also computes the source (`valuation_amount` if non-null, else
  `estimated_value` if non-null, else `None`); `build_ltv_calculation` returns it.
- `components/file/ltv/ltv-calculator.tsx` — under the value basis, a plain-language sub-line ("Appraised value
  *from valuation amount*" / "*from estimated value*") with the literal logic
  `appraised = valuation_amount or estimated_value` in a tooltip. Omitted when there is no appraised basis.

## Explicitly NOT done — flagged for Priya / deferred

- **Did not rename "Appraised value."** Whether `valuation_amount` (an AVM/stated valuation) should be labeled
  "appraised value" is a domain-naming question for Priya. The basis label is unchanged.
- **Did not collapse `valuation_amount` + `estimated_value`** into one field — a data-model decision deferred +
  flagged for Priya.
- **Did not change the LTV computation** (lesser-of basis, the `valuation_amount or estimated_value` precedence)
  or the MISMO parser (it correctly scopes `valuation_amount` to the subject property).

## Tests

**Backend** (`tests/services/test_ltv.py`, `tests/api/test_lp805_baseline_edits.py`):
- `appraised_value_source` reports `valuation_amount` / falls back to `estimated_value` / is `None` with no value.
- Regression: editing `valuation_amount` moves the appraised basis (the LTV recomputes off the new valuation).
- The PATCH on `valuation_amount` is exposed in the response, audited from→to, and marks verification stale.
- The loan-file detail exposes `valuation_amount`.

**Frontend** (`property-editor.test.tsx`, `ltv-calculator.test.tsx`):
- `valuation_amount` renders + is editable; editing it sends only that field.
- The source label + literal-logic tooltip render (valuation case), the fallback label (estimated case), and the
  sub-line is omitted when there is no appraised basis.

## Follow-up — LP-90.1

A screenshot review found the source transparency was only half-wired: the Value basis "(?)" tooltip was dead (a
native `title` on an `aria-hidden` glyph), the *editable* PROPERTY VALUE "Appraised value" row had no source
label/tooltip, and that row's sublabel mislabeled the source as "Stated". Fixed in **LP-90.1**
(`docs/tickets/LP-90.1.md`, ADR-215 addendum): a real shadcn/Radix Tooltip primitive (logic + plain explanation)
in both rows, the source label added to the editable row, and the sublabel corrected to the real provenance.
