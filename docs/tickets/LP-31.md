## The Backend Extension (small, scoped)

`GET /api/v1/loan-files` (and `list_loan_files`) gained:

- **`search`** — case-insensitive, matches `display_id` OR a borrower's name (via a
  scoped borrower sub-query), **always company-scoped** so it can't reach another
  company's files.
- **`lender_name`** + **`property_address`** on `LoanFileSummary` — resolved from the
  eager-loaded `lender`/`property` (null when absent; the detail endpoint also eager-loads
  `lender` now). No N+1: borrowers/lender/property are `selectinload`ed.
- **`status` is repeatable** (`?status=draft&status=submitted`) → filters to any of them,
  so grouped pills paginate correctly (the service takes `statuses: list[...]`).

Unchanged: tenant scoping (company from `current_user`), pagination, and the
no-`inbox_token`/no-raw-`ssn` exclusions. (ADR-103.)

## Filter Groupings (ADR-104)

One source of truth in `lib/loan-files/status.ts`, also holding the single status →
label/badge-colour map:

| Pill | Statuses |
| --- | --- |
| All | (no filter) |
| Active | draft, in_processing, ready_to_submit, submitted, **clear_to_close** |
| Action needed | in_conditions (V1 proxy — later: + outstanding blocking needs) |
| Completed | closed, withdrawn |

The four non-"All" groups are **disjoint and cover all eight statuses** (test). The plan's
example Active set (4) is extended by `clear_to_close` so no status is orphaned.

## Stats Counts (honest)

Each of the four cards runs a tiny list query (`page_size=1`) for its grouping and reads
the server-side **`total`** — exact, real counts (no fabricated numbers, no separate
counts endpoint). Cost: four small extra requests on load, cached by TanStack Query.

## Assumptions

- **Intake form is LP-32** ("New file" → `/loan-files/new` stub).
- **File detail is LP-33** (a row → `/loan-files/{display_id}` stub).
- **"Action needed" = IN_CONDITIONS** for V1 (refine later with blocking-needs surfacing).
- **No lender CRUD endpoint yet** — `lender_name` resolves when a lender is set (via DB /
  future endpoint); the dashboard degrades to "—" gracefully.
- **Stats counts via per-group total queries** (honest, reuses the list endpoint).

## Notes

- The dashboard fires four small count queries + one table query on load (the honest-counts
  tradeoff). A dedicated counts endpoint could replace them later if needed.
- `search` uses `ilike %term%` on `display_id` and the borrower full name; wildcard chars in
  user input act as wildcards (acceptable for V1 same-company search).
- Playwright was a throwaway verification tool (mocked the API for deterministic UI checks);
  not committed.
