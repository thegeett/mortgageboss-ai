# LP-44 — Manual Document Type Override (PATCH endpoint + drawer control)

- **Ticket:** LP-44 — manual document type override
- **Epic:** Epic 5 — Document Upload & Processing
- **Status:** Completed
- **Date:** 2026-06-12

## Summary

The **human-correction** half of human-in-the-loop. The AI classifies (sometimes wrong, or
unsure → `NEEDS_REVIEW`); LP-43 **shows** that; LP-44 lets the processor **fix** it. A single
`PATCH /api/v1/documents/{id}` sets the authoritative document type, re-derives the category,
marks the classification human-overridden, clears any stale error, audits the change, and
**re-runs extraction** for the corrected type — reusing the LP-39c re-extraction core. A drawer
control surfaces this in the UI and notes which types re-extract vs relabel-only.

> **Ordering note:** LP-39c intentionally built the re-extraction core
> (`reprocess_document_extraction`) ahead of this ticket — "the function LP-44's override calls."
> LP-44 reuses it **unchanged** and only adds a thin Celery task wrapper so it can be enqueued.

## What Was Built

**Backend**

- **`PATCH /api/v1/documents/{document_id}`** (`override_document_type`, `app/api/documents.py`)
  — tenant-scoped via `get_document_for_company` (out-of-company → `404`). Sets `document_type`,
  re-derives `category` (`category_for_type`), pins `classification_confidence = 1.0`
  (human-overridden), clears `processing_error`, logs `DOCUMENT_TYPE_OVERRIDDEN`, commits, then
  fire-and-forget enqueues re-extraction (`_enqueue_reprocess` → `reprocess_document.delay`).
- **`DocumentTypeOverrideRequest`** schema (`document_type: str`, `min_length=1, max_length=64`).
- **`category_for_type(document_type)`** — factored out of `_process_document` (the type→category
  map), now shared by the override endpoint.
- **`reprocess_document` Celery task** (`app/tasks/document_processing.py`) — thin wrapper:
  `_run_reprocess` loads the doc and calls the **existing, unchanged**
  `reprocess_document_extraction(db, document)` (registry-based, skips classification, new
  version, resilient). Missing doc → log + no-op.
- **`ActivityType.DOCUMENT_TYPE_OVERRIDDEN`** + Alembic migration `8a153b35129b` — constraint-swap
  on the `ck_activity_logs_activitytype` CHECK (raw-SQL drop/add, per the LP-30 pattern).

**Frontend**

- **`useOverrideDocumentType(fileId, documentId)`** mutation (`lib/api/documents.ts`) — PATCHes
  the type; on success invalidates the documents list **and** this document's detail so the live
  poll shows the re-processing.
- **`TypeOverride`** control in the LP-43 drawer (`components/file/documents/document-drawer.tsx`)
  — a "Correct type" section: a type `<select>` (`OVERRIDE_TYPE_OPTIONS`, the current type kept
  selectable even if non-standard) + an **Apply** button, a warning callout when the doc is
  `needs_review`, and a hint line that says whether saving **re-extracts** or **relabels only**
  (`typeReExtracts`). Success/relabel toasts.
- **`EXTRACTABLE_TYPES` / `OVERRIDE_TYPE_OPTIONS` / `typeReExtracts`** (`lib/loan-files/documents.ts`).

## Re-extract vs relabel-only

`typeReExtracts` is `true` only for `pay_stub` / `w2` / `bank_statement` (the registered
extractors). Override **to** one of those re-runs extraction for that type; override to any other
type (driver's license, credit report, gift letter, "other", …) relabels the document
**classified-only** — the reused core's registry miss → `COMPLETED` with no extraction and **no
API call**. The drawer states which will happen before the processor saves.

## Security / correctness

- **Tenant isolation:** `company_id` always from `current_user`; another company's document →
  `404` (tested for PATCH alongside get/download/delete).
- **Server-owned mapping:** the category is **re-derived**, never taken from the client.
- **Human-authoritative:** `classification_confidence = 1.0` so the re-extraction isn't
  immediately re-flagged `NEEDS_REVIEW` for low classification confidence.
- **Resilience:** the override is committed before enqueue; a broker hiccup is logged
  (`reprocess_enqueue_failed`, metadata-only) and can't lose the change.
- **PII:** no bytes/values logged; the activity summary is metadata only ("Type changed to …").

## Manual verification (live worker + broker)

Seeded a `needs_review` doc (`pay_stub`, conf `0.41`, with a `processing_error`), applied the
endpoint's mutation (relabel to `drivers_license`), and enqueued the **real** task against the
running Redis broker + a live Celery worker. The worker picked up `documents.reprocess_document`
and the doc settled to: `status=completed`, `type=drivers_license`, `category=borrower_info`,
`confidence=1.0`, `processing_error=None` — relabel-only, **no API call**, task succeeded in
~0.06s. (Confirms the reused core + new wrapper end-to-end; extractable-type re-extraction uses
the same path with a registered extractor.)
