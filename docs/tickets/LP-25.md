## The Hybrid Transport (client rules)

- **Access token** — in memory only (Zustand). Never localStorage/sessionStorage/JS
  cookie. Volatile by design; restored by silent refresh on reload (ADR-084).
- **Refresh token** — the browser's httpOnly cookie; JS never reads/writes it. All
  calls that need it (`refresh`, `logout`) just set `withCredentials`.
- **On load** — one `/auth/refresh`; success → store; failure → unauthenticated; a
  loader shows meanwhile so nothing flashes.
- **Interceptors** — attach Bearer on requests; on 401 do a single-flight refresh and
  retry once; exempt `/auth/login` + `/auth/refresh`; on refresh failure clear + go to
  `/login` (ADR-085).

## Security Properties

- **Minimal XSS exposure** — the only JS-held credential is a short-lived in-memory
  access token; the refresh token is httpOnly.
- **Anti-enumeration preserved** — the form shows the backend's generic "Invalid email
  or password"; it never distinguishes unknown-email from wrong-password, and never
  logs credentials/tokens.
- **Route protection is UX, not security** — the backend is the real boundary; client
  guards only prevent rendering authenticated chrome (ADR-086).
- **Loop protection** — auth endpoints are refresh-exempt and requests retry once, so a
  dead session ends in one clean redirect, not a refresh loop.

## Decisions Made

- **ADR-084** — Access token kept in memory only.
- **ADR-085** — Axios interceptors: single-flight auto-refresh with loop protection.
- **ADR-086** — Frontend route protection is UX, not security.
- **ADR-087** — Vitest for frontend unit tests.

## Verification Performed

- **Automated:** `pnpm test` → **9 passed** (auth store: setAuth/clearAuth/
  isAuthenticated/finishInitializing/token-without-user; schema: valid / bad email /
  empty email / empty password). `pnpm typecheck` clean. `pnpm lint` (Biome) clean (39
  files). `pnpm build` succeeds — `/`, `/login`, `/dashboard` all compile and prerender.
- **Backend contract (curl, against the running backend with a seeded user):**
  - login (valid) → 200, `access_token` in body, `Set-Cookie refresh_token` with
    `HttpOnly`, `Path=/api/v1/auth/refresh`, `SameSite=lax`, `Max-Age=2592000`, **no
    `Secure`** (dev/HTTP) — exactly the flags the client relies on.
  - `GET /me` with Bearer → 200, the `UserPublic` shape the store/types expect.
  - refresh with cookie (no body) → 200 + **rotated** cookie.
  - logout with cookie → 204 + cookie cleared (`Max-Age=0`).
  - wrong password and unknown email → **identical** generic `401`.
  - `/me` with no token → `401` + `WWW-Authenticate: Bearer`; refresh with no cookie → `401`.
- **Frontend serving (dev server):** `/`, `/login`, `/dashboard` all return `200`; the
  login HTML renders the form and the silent-refresh loader; no runtime/compile errors.

### Manual (in-browser) checklist for the reviewer

These need a real browser (cross-origin cookie + JS flow) and are recommended before
merge — backend on `:8000`, frontend `pnpm dev` on `:3000`, seed user
`processor@acme.com` / `Password123`:

1. Visit `/login`, sign in → lands on `/dashboard` showing name/email/role.
2. Reload `/dashboard` → brief loader, stays signed in (silent refresh; access token
   was wiped from memory and re-established).
3. Open DevTools → Application → Cookies: `refresh_token` is HttpOnly, scoped to
   `/api/v1/auth/refresh`; no access token in localStorage/sessionStorage.
4. Visit `/dashboard` in a fresh/incognito window (no cookie) → redirected to `/login`.
5. Click Sign out → cookie cleared, redirected to `/login`; `/dashboard` now redirects.
6. Bad credentials → generic "Invalid email or password."; no enumeration, nothing
   sensitive logged.

## Notes

- The whole app is gated behind the AuthProvider loader during the initial silent
  refresh (a brief loader on first load) — a deliberate tradeoff for a no-flash
  experience. Public pages render right after the attempt settles.
- The login UI reuses the LP-5 design tokens and the home page's visual language
  (gray-50, primary-blue radial accents, the `mortgageboss·ai` wordmark, shadcn/ui),
  rather than inventing a new look — appropriate for a trustworthy financial tool.
- Vitest tests cover pure logic; interceptor/flow behaviour is verified manually
  (above) rather than with brittle mocks.
