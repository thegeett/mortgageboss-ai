# LP-91 — DTI consumes the LP-87 MI calculator (fix MI omitted from PITI)

- **Ticket:** LP-91
- **Epic:** Phase 3 — Verification Engine & Calculators
- **Status:** Completed
- **Date:** 2026-06-30

## Summary

A correctness fix for the DTI calculator. The DTI's PITI **mortgage-insurance** line was
manual-only / default `$0`, so it silently **omitted mandatory MI** (FHA MIP always;
Conventional PMI when LTV > 80%), understating the front-end DTI in the **qualifying
(dangerous) direction** for every FHA file and every low-down Conventional file. LP-91 wires
the DTI to **consume** the LP-87 MI calculator's `monthly_premium` — the single source of
truth — so MI is now in PITI, program-aware, and overrideable. See **ADR-216**.

## The bug (confirmed from the trace)

- `app/services/dti.py` `_auto_housing_lines` seeded the MI line
  `_AutoLine(HOUSING_MORTGAGE_INSURANCE, "Mortgage insurance (MI)", None, "manual")` — auto
  `None`, source `"manual"`. Its siblings (P&I `computed`; taxes/insurance/HOA `extracted`)
  had data sources; only MI had none.
- `_to_items` sets `effective = override ?? (auto ?? 0)`, so with no override the MI line
  contributed **`$0`** to PITI — `compute_dti` built the front-end/housing payment with no MI.
- The DTI never consumed the LP-87 MI calculator (`build_mi_view` / `compute_*`); the correct,
  program-aware monthly premium existed but was unused — a two-source-of-truth gap (the same
  shape as the LP-90 appraised-value binding).
- Consequence: a borrower truly at 44% DTI could show ~41% (missing ~$300/mo MI) and appear to
  pass a lender ceiling they'd actually fail.

## The fix

### A single shared MI source — `app/services/mi.py` (new)

`compute_loan_mi(db, *, loan_file, confidence_cutoff)` is the program-aware MI computation:
sources LP-77's LTV, the base loan amount, the persisted MI overrides (the
`calculator_overrides` table, `mortgage_insurance` namespace), and LP-84's FHA UFMIP rule, and
dispatches to the pure `compute_conventional_pmi` / `compute_fha_mip`. It returns the
`MiResult` (incl. `monthly_premium`) plus the input lines + UFMIP rate the calculator view
needs. **It imports neither `dti` nor `calculators`**, so both consume it without an import
cycle.

- `app/services/calculators.py` — `build_mi_view` is refactored to **delegate** to
  `compute_loan_mi` (output unchanged; the MI-specific starter constants + the UFMIP helper
  moved to `mi.py`). The presentation (steps / headline / methodology) stays in the view.
- `app/services/dti.py` — `_auto_housing_lines` now consumes `compute_loan_mi(...)` and uses
  `result.monthly_premium` as the MI line's auto value (source `manual` → `computed`).
- `frontend/lib/api/calculators.ts` — the MI-calculator override mutations now also invalidate
  the DTI query (the DTI consumes MI → the panel must refresh on an MI change).

### Properties of the fix

- **Program-aware** (from the MI calculator): Conventional → PMI when LTV > 80% (`$0` at ≤ 80%);
  FHA → monthly annual-MIP always. No PMI/MIP math duplicated in the DTI.
- **Auto-populated but overrideable:** the consumed premium is the auto value; a `DtiOverride`
  on `housing.mortgage_insurance` still wins (the real MI quote).
- **Upfront MIP stays financed:** only `monthly_premium` enters PITI; the FHA UFMIP (1.75%) is
  financed into the loan — never a monthly DTI item.
- **Recompute on MI change:** the DTI reads MI live (LTV change → PMI on/off; program change; MI
  override); the frontend MI-override mutation invalidates the DTI query.
- **Single source of truth:** the DTI's MI equals the MI calculator's `monthly_premium` — they
  can't diverge, and the DTI doesn't recompute MI independently.

## Domain flag (validate with Priya)

The **mechanism** (the DTI must include mandatory MI) is not in question. But the Conventional
**PMI rate** is a grounded-starter (varies by credit / LTV / MI provider — a rate card, not a
clean formula): the auto-computed PMI is a *starting point* the processor overrides with the
real quote, surfaced via the MI calculator's `methodology.starter` note (the single source). The
FHA MIP rates (HUD via LP-84) are more deterministic. Do not treat the computed PMI as
authoritative.

## Tests

`backend/tests/services/test_dti_mi.py` (new, 8 tests):
- FHA file → monthly MIP in PITI (no longer omitted); MI line auto is `computed`, `$137.50`
  (300k × 55 bps / 12), front-end DTI reflects it.
- DTI MI == the MI calculator's `monthly_premium` (single source of truth).
- Conventional program-aware: LTV > 80% → PMI in PITI; LTV ≤ 80% → no PMI (MI auto `None`,
  excluded from PITI).
- Upfront MIP (financed `$5,250`) is NOT in monthly DTI — only the monthly premium.
- A `DtiOverride` on the MI line wins over the consumed auto.
- Recompute: raising the property value (LTV ≤ 80%) drops PMI off; an MI-rate override
  (`mi.pmi_rate_bps`) flows into the DTI.
- Tenant-scoped.

No regression on LP-76/77/84/87 (the MI calculator tests still pass — `build_mi_view` output is
unchanged).

## Not done (out of scope)

- The MI calculator (LP-87) computation is unchanged — the DTI consumes it.
- The UFMIP is not added to monthly DTI (it's financed).
- The DTI does not recompute MI inline (consume, not duplicate).
- The computed Conventional PMI is not treated as authoritative (grounded-starter; flagged).
- The LP-90/90.1 LTV/appraised-value work is untouched.
