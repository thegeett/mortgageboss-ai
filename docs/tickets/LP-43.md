# LP-43 — Documents Tab Frontend

- **Ticket:** LP-43 — Documents tab (dropzone upload, category grouping, live status polling, detail drawer)
- **Epic:** Epic 5 — Document Upload & Processing
- **Status:** Completed
- **Date:** 2026-06-12

## Summary

Built the **Documents tab** — the UI where processors see and use the document pipeline:
drag-and-drop upload, documents grouped by category, **live status via polling**, and a
detail drawer showing the extraction. Replaces the LP-33 Documents placeholder. Frontend
ticket consuming the existing LP-36 (documents) and LP-40 (dev text-layer) endpoints.

> **Prerequisite note (important):** The **LP-42 processing pipeline does not exist yet** in
> the codebase — the upload endpoint enqueues nothing and there is no `document_processing`
> task. So an uploaded document **stays `pending`** until LP-42 lands. The tab is built and
> correct (its polling logic is independent), but the headline "watch it process live →
> done" can only be demonstrated once LP-42 advances `Document.status`. This was verified, not
> assumed (see Verification).

## What Was Built

- **Types** (`lib/types/document.ts`) — `DocumentResponse`, `DocumentDetailResponse`,
  `ExtractionPublic` (`extracted_data` a flexible record), the `DocumentStatus` /
  `DocumentCategory` unions, `TextLayerExtraction`. No `storage_path`.
- **Helpers** (`lib/loan-files/documents.ts`) — `DOCUMENT_STATUS_META` (design-token
  treatment + `inProgress`), `isTerminalStatus` / `hasInProgressDocuments`,
  `groupDocumentsByCategory`, `validateUploadFile`, `formatFileSize` / `formatConfidence`,
  `extractionFields`. Unit-tested (`documents.test.ts`, 13 tests).
- **Hooks** (`lib/api/documents.ts`) — `useLoanFileDocuments` (live polling),
  `useDocumentDetail` (enabled when the drawer opens), `useUploadDocuments`,
  `useDeleteDocument`, `useDevTextLayer`, and `downloadDocument` (authed blob).
- **Components** (`components/file/documents/`) — `document-dropzone`, `document-status`
  (live badge), `document-list` (grouped + rows + states), `document-drawer` (sheet); a new
  shadcn `components/ui/sheet.tsx`. The tab page replaces the LP-33 placeholder.

## Live Polling (the signature)

`useLoanFileDocuments` uses a **function `refetchInterval`**: ~2500ms while any document is
non-terminal, `false` once all are terminal (`completed`/`needs_review`/`failed`). The list
updates during processing and **stops** when settled. After upload, the documents query is
invalidated so new PENDING docs appear and polling resumes. `Document.status` is the source of
truth. (ADR-134.)

## Status Treatment

`DocumentStatusBadge`: a **spinning** loader for in-progress (`pending`/`classifying`/
`classified`/`extracting`, "Processing" / brief "Classified"); **green Completed**; **amber
Needs review** (honest AI-uncertainty signal); **red Failed**. Colours from the single
`DOCUMENT_STATUS_META` map (design tokens). Type **correction** is LP-44 — LP-43 only displays
the state. (ADR-135.)

## Category Grouping

`groupDocumentsByCategory` → the eight `DocumentCategory` labels in order (only non-empty),
then a **"Processing / uncategorized"** group for documents without a category yet (e.g.
`pending`). Newest-first within a group.

## The Drawer

A shadcn `sheet` that fetches detail on open (`useDocumentDetail`): metadata (type, category,
confidence, size, uploaded), the **extraction** rendered as labelled key/values
(`extractionFields` — known pay-stub fields first, money as currency, nulls as "—"; "No
extraction — classified only" for non-pay-stub types), an authed **Download**, and a soft-
delete with confirm.

## Dev-Only Text-Layer Button

Rendered **only in non-production** (`process.env.NODE_ENV !== "production"`, statically
eliminated from a prod build) — calls the LP-40 dev endpoint and shows the deterministic text
(+ has_text / page_count) for comparison. Absent in production (and the endpoint 404s there).
(ADR-136.)

## Decisions Made

- **ADR-134** — live status via poll-while-non-terminal.
- **ADR-135** — grouped by category; NEEDS_REVIEW surfaced honestly; override deferred to LP-44.
- **ADR-136** — dev-only text-layer comparison button (non-production).

> **ADR numbering:** LP-41 used 132–133; the next free numbers were **134–136**.

## Assumptions

- **Type override** (correcting needs-review/misclassified) is **LP-44**.
- **Non-pay-stub** documents are "classified only" (no extraction) in V1 (Phase 2 adds types).
- **Verification/findings** are Phase 3; other tabs stay placeholders.
- Consumes the existing **LP-36 / LP-40** endpoints — **no backend changes** (the
  `DocumentResponse` shape already carries everything the tab needs;
  `processing_error` was not needed — FAILED shows a clean state).
- The **LP-42 pipeline** is required for documents to actually progress (see the note above).

## Verification Performed

- **`pnpm lint` (Biome), `pnpm typecheck` (tsc strict), `pnpm build`** — all pass. **13
  vitest** logic tests pass (poll/stop terminal rule, grouping order + uncategorized bucket,
  client validation, extraction field ordering/money/null, file-size formatting).
- **End-to-end API check against the running backend** (real token, `processor@demo.com`) —
  the exact endpoints the tab consumes, confirming the data shapes match the types:
  - Upload a PDF → `201`, `status: "pending"`, `category: null`, **no `storage_path`** (→ it
    lands in "Processing / uncategorized" with a spinning "Processing" badge).
  - `GET …/documents` → the document listed, `pending`.
  - `GET /documents/{id}` → `current_extraction: null` (→ drawer shows "classified only"),
    no `storage_path`.
  - `GET /documents/{id}/download` → `application/pdf`, `Content-Disposition: attachment;
    filename="…"`, exact bytes.
  - `POST /dev/documents/{id}/extract-text-layer` → `extraction_ok: true`, `has_text: true`,
    1 page, the real text (→ the dev button renders this). Test data cleaned up afterward.
- **Not demonstrable yet (honest):** the live status **transition** PENDING → processing →
  done — because **LP-42 isn't built**, so nothing advances the status; the document stays
  `pending` and the list polls without settling. The polling logic itself is unit-tested and
  correct; it will light up once LP-42 lands. A browser-level visual pass (Playwright) was
  not run (not installed); the integration risk — endpoint/shape correctness — was instead
  verified directly via the API as above.

## Notes

- To see live processing during manual testing, **LP-42 must be built and the Celery worker
  (LP-41) running**.
- Reused/extended the shared status→badge idea (a dedicated `DOCUMENT_STATUS_META`); design
  tokens only, no ad-hoc colours; the drawer is a focus-trapped shadcn `sheet`.
