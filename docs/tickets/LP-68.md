# LP-68 ‚Äî Needs-List Engine: Models, States & Per-File Serialization

## Summary

The deterministic backbone of the needs list ‚Äî the file's living checklist of
required documents (the highest-value differentiator). The five states, the
auto-update/satisfaction logic, the **per-file serialization** (the race-condition
fix), and a thin deterministic floor from the stated MISMO data. **Deterministic ‚Äî
no AI** (the case-by-case intelligence is LP-69; the UI is LP-70). Backend only.

## The Five States + Transitions

`PENDING` (needs this; not arrived ‚Äî the default) ‚Üí `RECEIVED` (a matching doc
arrived, not yet verified) ‚Üí `VERIFIED` (passed ‚Äî extraction succeeded) | `REJECTED`
(arrived but failed; still open, with a reason); any ‚Üí `WAIVED` (processor). Driven
by document arrivals + processor actions ‚Äî **deterministic, not AI**. A
valid-transition map guards invalid transitions (they raise `InvalidNeedTransition`).

> The LP-19 `REQUESTED` (borrower-outreach) state is **preserved** as an orthogonal
> pre-existing value ‚Äî a need awaiting arrival may be `PENDING` or `REQUESTED`, and
> both are satisfiable. `OUTSTANDING` was renamed to `PENDING`.

## Deterministic Satisfaction-Matching

When a document reaches a terminal status, the engine advances the oldest open need
whose `needs_type` equals the document's `document_type`: Received ‚Üí Verified (the
doc `COMPLETED`) | Rejected (it `NEEDS_REVIEW`/`FAILED`). **Type-level** now; no
false matches; no AI. Quantity/recency-granular matching ("2 pay stubs", "within 30
days") is a documented **future refinement**. `"Verified" = extraction passed`;
Phase 3 adds cross-source rules later.

## Per-File Serialization (the RACE FIX)

Real processing dumps batches of documents for a file; a naive inline "doc arrives ‚Üí
update needs" would race on the file's shared needs state (lost updates,
double-satisfaction). **The fix:** the needs update runs as a **separate Celery
task** (`needs.update_for_document`, enqueued after a document is terminal) that
acquires a **per-loan-file Redis lock** (`needs-lock:{loan_file_id}`) before applying
the matching.

- **Same file ‚Üí serial:** concurrent arrivals for one file apply one at a time.
- **Different files ‚Üí parallel:** different lock keys ‚Üí independent.
- The lock **auto-expires** (`timeout`) so a crashed worker never deadlocks a file.

**The concurrency test** proves it two ways: (1) the lock is exclusive for the same
file (a second acquire fails while held) and independent across files; (2) under the
serialized order the matching is correct ‚Äî two pay-stub documents do not
double-satisfy/clobber (the first verifies the need; the second no-ops).

## The Thin Deterministic Floor

`seed_floor_needs` generates near-certain needs from the **stated MISMO data**:
employment income ‚Üí `pay_stub` + `w2`; a purchase ‚Üí `purchase_agreement`; stated
assets ‚Üí `bank_statement`. Wired into the MISMO import (seeds when the stated data
lands). Floor needs are `origin=FLOOR`, `disposition=CONFIRMED` (near-certain); the
seeder is **idempotent**. **Thin by design** ‚Äî the reliable baseline LP-69's AI
augments.

## Source-Agnostic Engine

A need carries its `origin` ‚Äî the source-agnostic provenance (`floor` /
`suggestion` / `ai_reasoning` / `manual` / ‚Ä¶) ‚Äî plus a `disposition` (the
human-confirmation lifecycle) + `reasoning` + `source_finding_id`.
`ingest_suggested_need` turns an LP-67 `SuggestedNeed` into a need
(`origin=SUGGESTION`, `disposition=PROPOSED`, carrying the reasoning + the
source-finding link; idempotent per finding). **LP-69's AI proposals ingest the same
way** (source `ai_reasoning`).

## Disposition Groundwork (for LP-69/70)

`disposition` ‚àà {proposed, confirmed, waived, dismissed} models the AI-proposes /
processor-confirms lifecycle, orthogonal to the arrival `status`. The deterministic
floor is `CONFIRMED`; a suggestion / AI proposal starts `PROPOSED` (the processor
confirms/dismisses in LP-70).

## Deterministic ‚Äî No AI

LP-68 is the engine (states, matching, serialization, floor) ‚Äî **no AI**. LP-69 is
the intelligence (holistic, case-by-case needs reasoning, ingesting via the same
source-agnostic path); LP-70 is the UI.

## Verification Performed

- **States/transitions** ‚Äî valid sequence (Pending‚ÜíReceived‚ÜíVerified); an invalid
  transition (Pending‚ÜíVerified) is guarded; waive from any state (+ disposition).
- **Satisfaction-matching** ‚Äî a passing pay-stub document verifies the matching
  need; a failed document rejects it (with a reason); a non-matching type does NOT
  satisfy; an untyped document is a no-op. Scoped to the document's own file.
- **The CONCURRENCY test** ‚Äî the per-file lock is exclusive for the same file +
  independent across files (real Redis); under serialized order two documents don't
  double-satisfy (no lost update).
- **The floor** ‚Äî stated employment ‚Üí pay_stub + w2; purchase ‚Üí purchase_agreement;
  assets ‚Üí bank_statement; idempotent; thin when there's no stated data.
- **LP-67 ingestion** ‚Äî a `SuggestedNeed` ‚Üí a need carrying the reasoning + the
  source-finding link; idempotent per finding.
- **Migration** ‚Äî applies (status rename + new states/origins/disposition/reasoning/
  source columns + FK + indexes) and reverses; verified against dev Postgres.
- **No regression** ‚Äî the pipeline / extractors / findings unchanged (the needs
  update moved to the serialized task; the pipeline now enqueues it).
- **Suite** ‚Äî `uv run pytest` green (**986 passed**, +23 over LP-67); `ruff` +
  `ruff format` clean; `mypy` clean (139 source files).
