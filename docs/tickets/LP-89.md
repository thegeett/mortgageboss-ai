# LP-89 ‚Äî Phase 3 Testing, Polish & Hardening (capstone) + the Rule/Calculator Validation Aid

- **Ticket:** LP-89 ‚Äî the Phase-3 capstone
- **Epic:** Phase 3 ‚Äî Verification Engine & Calculators
- **Status:** Completed
- **Date:** 2026-06-29

## Summary

The capstone: HARDEN + POLISH the verification system (LP-74..88) for a real demo, make the demo proofs
runnable, and build a VALIDATION AID that captures the domain expert's (Priya's) verdicts on every
grounded-starter rule + calculator methodology ‚Äî without fabricating validation. See **ADR-212** (hardening),
**ADR-213** (the validation aid + the grounded_starter‚Üívalidated state), **ADR-214** (the V2 deferrals).

## The honest framing (the heart of this ticket)

Every rule (LP-82..86) + calculator methodology (LP-87) is **grounded-starter** ‚Äî researched against the real
sources but **NOT validated** by Priya. Her session is the validation. **This ticket does not (cannot) do that
validation** ‚Äî it hardens the system + builds an aid that CAPTURES her verdicts. Nothing is claimed "validated";
`validation_status` defaults to `grounded_starter`; a corrected value applies because Priya said so (recorded),
not because the system decided.

## PART 1 ‚Äî Engineering hardening

- **The stuck-RUNNING watchdog** ‚Äî a read-time reconcile in `GET ‚Ä¶/verification`: a run RUNNING past a 5-minute
  timeout (> the 180s task hard limit + slack) is marked FAILED with a legible error, so the UI never spins
  forever + can re-run. (Tests: a 10-min-old RUNNING run ‚Üí FAILED; a fresh one is left alone.)
- **Real-stack worker integration test** ‚Äî invokes the actual task body (`app.tasks.cross_source._run`)
  end-to-end (the AI stubbed, the session at the test DB) ‚Üí run COMPLETED + findings persist; paired with the
  task-registration guard. Closes the worker-seam lesson. (`run_cross_source` now resolves its reasoner at call
  time so the worker path is stubbable.)
- **Performance** ‚Äî a test bounds the deterministic engine under the full ~120-rule load (< 3s); the deterministic
  pass is sub-second (the AI pass is the async/expected-slow part).
- **Error-path robustness** ‚Äî a file with no data / no docs / FHA / partial doesn't crash (calculators show "‚Äî";
  the engine records absent-fact rules not-evaluated).

## PART 3 ‚Äî The validation aid (Option B)

- **The starter inventory** (`/admin/validation`, admin-gated) ‚Äî every grounded-starter item (~123: the
  Conventional + FHA + cross-source rules + the 6 calculator methodologies) with its program, category,
  description, value/op/unit, citation, source type, and `to_verify` marker; grouped/filterable (program /
  category / status).
- **The verdict capture** ‚Äî a new `validation_verdicts` table (company-scoped, self-audited) records the verdict
  per item: **Validate / Correct (+ value + note) / Flag remove (+ why) / Add-new (a missing rule)**. Upsert per
  item; ADD_NEW proposals accumulate.
- **The validation_status state** ‚Äî derived: `grounded_starter` (DEFAULT) / `validated` / `corrected` /
  `flagged_remove`. The grounded-starter‚Üívalidated transition is explicit + queryable.
- **`docs/validation-session-guide.md`** ‚Äî how to walk the inventory by category + record verdicts.
- **HONEST** ‚Äî the aid captures Priya's verdicts; it does not validate. A corrected value applies because she
  said so (recorded with attribution).
