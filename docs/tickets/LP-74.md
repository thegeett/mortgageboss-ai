# LP-74 — Verification Rule Engine: Uniform Rule Structure + Three-Layer Composition

- **Ticket:** LP-74 — Verification rule engine (the spine of Phase 3)
- **Epic:** Phase 3 — Verification Engine & Calculators (first ticket, Arc A)
- **Status:** Completed
- **Date:** 2026-06-27

## Summary

The **engine before the rule content** — the LP-68 of verification. Builds the uniform rule
structure, the three-layer composition, and the deterministic evaluation loop, proven with a
**few sample rules + a sample overlay**. The real ~60 Conventional + ~50 FHA rules are
LP-82..85; the real lender overlays are LP-80. Backend only. See **ADR-188** and
[`phase-3-verification-design.md`](../phase-3-verification-design.md) §8.

## What Was Built

- **The uniform rule structure** (`backend/app/verification/rules/schema.py`) — one
  `VerificationRule` shape for all three layers: stable `rule_id`, `layer`, `applicability`,
  typed `reads`, `condition` (threshold-as-data), `severity`, `category`, `description`,
  structured `source`. Plus the fixed comparison primitive `satisfies()`.
- **The three-layer composition** (`backend/app/verification/registry.py`) — `RuleRegistry`
  resolves a file's effective rule set: base (regulatory + investor[program]) patched by the
  lender overlay as a diff. `apply_overlay()` does override-by-id + add-custom.
- **The overlay structure** (`backend/app/verification/overlays/schema.py`) — `LenderOverlay`
  = a diff (`ThresholdOverride`s by id + custom rules).
- **The deterministic evaluation loop** (`backend/app/verification/engine.py`) — `evaluate()`
  reads a `FileFacts` snapshot (`facts.py`), compares each rule's typed field to its threshold,
  emits a pass/fail `EngineFinding`. Pure, no AI.
- **The DB-facing service** (`backend/app/services/verification_engine.py`) —
  `run_verification()`: builds facts from the file's stated/extracted data, resolves rules by
  program + lender, evaluates, and persists findings into the shared model attached to a
  `Verification` run. Per-file, tenant-scoped.
- **Sample rules** (`rules/samples.py`) + **a sample overlay** (`overlays/samples.py`).
- **The two-generator seam** — a minimal `origin` field on the `Finding` model (migration
  `f4a9c1e7b203`).
- **Tests** — `backend/tests/verification/` (structure, composition, evaluation) +
  `backend/tests/services/test_verification_engine.py` (per-file/tenant, two-generator, no
  regression).

## The Two Linchpins (why they make overlays possible)

1. **Stable `rule_id`** (e.g. `conv.dti.back_end_max`) — a rule's identity. An overlay
   overrides a rule *by this id*. Without a stable id, an overlay has nothing to reference and
   overlays are an impossible retrofit.
2. **Threshold-as-data** — the threshold is a `Condition` (`{op, value, unit}`) the fixed
   `satisfies()` logic reads, **not hardcoded**. Because it is data, an overlay supplies a
   different value and the *same* logic evaluates against it.

**Rule logic is fixed; thresholds are data.** `VerificationRule.with_condition()` produces an
overridden copy where only the `condition` changes — `rule_id`, `reads`, `layer`, and the
comparison logic are all preserved (the original frozen definition is untouched).

## Three-Layer Composition (base + overlay-diff → effective set)

Given a file's **program + lender**:

1. **Base** = all regulatory rules (Layer 1, every file) + the investor rules for the file's
   program (Layer 2 — Conventional **or** FHA, never both).
2. **Patch** with the lender overlay (Layer 3) as a **diff**: override → replace the base
   rule's threshold by `rule_id`; custom → append.
3. **Output** = a flat effective set with final thresholds.

**The investor rule is the default.** An un-overridden rule falls through to its investor
value; no overlay → all investor defaults; the overlay value wins where specified. **Overlays
are diffs, not full per-lender copies** — the sample overlay is one override + one custom rule,
far smaller than the catalog.

## Deterministic Evaluation (read typed field → compare → emit finding)

For each rule in the effective set: read the file's typed field from `FileFacts` → compare to
the (possibly overlay-patched) threshold with the fixed `satisfies()` → emit a pass/fail
finding (green on pass, red/yellow on fail). **Pure deterministic judgment, no AI** — the
"AI surfaces, deterministic code judges" principle. The handoff is **structured data** (rules
read typed field paths, never prose). A datum the file does not carry yet → the rule is **not
evaluated** (the engine never invents a verdict). The overlay-patched threshold (45) produces a
finding where the investor default (50) would not — the end-to-end proof the patch reaches
evaluation. This is **auditable**: correct by construction, not "probably right per the AI".

## Two-Generator Accommodation (the shared findings model)

The engine emits into the shared LP-66 `Finding` model in a **uniform shape** (rule_id /
origin, observed value, severity-derived status, condition, structured source, source-location
placeholder, reasoning). A new minimal `FindingOrigin` field marks engine findings
`deterministic_rule`; the AI cross-source layer (LP-78) feeds the **same** model as
`ai_cross_source`. The findings path is **not** engine-exclusive. LP-75 does the fuller
findings-model extension (confidence / resolution / blocking / source-location); `origin` is
the minimal field needed to emit in the uniform shape now.

## Engine Before Content (sample rules + a sample overlay)

The samples prove the mechanism, not the content (LP-68 pattern):

- **Regulatory** (`reg.aml.large_deposit_review`) — AML large-deposit review (all loans).
- **Documentation** (`doc.income.paystub_recency`) — pay stub within 30 days (all loans).
- **Investor** — `conv.dti.back_end_max` (≤ 50) and `fha.dti.back_end_max` (≤ 57), so
  program-selection is observable (a Conventional file gets the Conventional rule, never FHA).
- **Sample overlay** (`sample-overlay-bank`) — overrides `conv.dti.back_end_max` to ≤ 45 (by
  id) and adds a reserves custom rule (add-custom).

The real content is LP-82..85; the real overlays LP-80.

  rules need them.
- Verification is **per-file** (shared rule definitions, per-file runs), tenant-scoped via
  loan_file → company. PII is metadata-only in logs.

## Verification Performed

- **The linchpins** — every sample rule has a stable, unique `rule_id`; changing a
  `Condition`'s data flips the verdict (the threshold is not hardcoded); `with_condition`
  preserves id/reads/layer/logic.
- **Composition** — Conventional gets Conventional investor rules (not FHA) and vice versa; no
  program → only regulatory; no overlay → all investor defaults; the overlay overrides the DTI
  threshold (45 not 50) by id; the custom rule is added; an un-overridden rule falls through;
  the overlay is a diff (one override + one custom), not a copy.
- **Deterministic evaluation** — over threshold → finding; under → pass; missing datum → not
  evaluated; the overlay-patched threshold (45) produces a finding the default (50) would not;
  source-location carries through.
- **Two-generator** — findings persist in the uniform shape with `origin=deterministic_rule`;
  an `ai_cross_source` finding coexists in the same model (not engine-exclusive).
- **Per-file / tenant** — the run is rejected for a file outside the requesting company; findings
  link to the run; the run's counts match.
- **No regression** — full suite **1093 passed**; a plain `Finding` still constructs and
  defaults `origin`. `ruff`/`mypy --strict` clean; the migration applies + reverses (verified
  against Postgres; constraint/index names converge with `create_all`).

## Notes

- **The rule structure determines whether Phase 3 composes** — stable ids let overlays
  *reference* rules, thresholds-as-data let overlays *override* them, so an overlay is a clean
  patch rather than an impossible retrofit.
- **AI surfaces, deterministic code judges** — the engine's judgment is pure and auditable; the
  AI's role is upstream extraction. The handoff is always structured data, never prose.
