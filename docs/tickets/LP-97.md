# LP-97 — View fix: detailed itemized before/after impact preview before Apply

- **Ticket:** LP-97 (sixth ticket of the finding-presentation epic, LP-92..98)
- **Epic:** Phase 3.5 — Finding presentation
- **Status:** Completed
- **Date:** 2026-07-01

## Summary

Findings that carry an **apply-spec** now show **View fix** — a detailed, itemized before/after
impact preview — instead of a bare Apply. The preview is a **dry-run** computed by REUSING the real
apply→recompute in a rolled-back savepoint, so it **matches** exactly what Apply does (one source of
truth). "Apply fix" commits the real apply; Cancel is a no-op. See **ADR-222**.

## The flow

- Findings **with** an apply-spec (`details.apply`; currently ~`xsrc.liability.undisclosed_debt`'s
  `add_liability` → DTI-recompute) → **View fix** (a `Wrench` button) → the impact dialog.
- Findings **without** one → unchanged: Override / Accept-risk / Request-docs / Add-note (they move
  no numbers — nothing to preview).

## The dry-run (one source of truth)

`app/services/finding_impact.py` `preview_finding_apply`:

1. Snapshot the DTI / LTV **before**.
2. Open a **savepoint**, run the **real** `apply_finding` (the structured-data change + recompute),
   snapshot the DTI / LTV **after**, then **roll the savepoint back** + `refresh` the objects → nothing
   persists.
3. Return the itemized before/after (the existing `DtiCalculation` schema — already line-itemized) +
   `affects` (which calculators moved). Only the moved calculator(s) are populated.

Because it reuses the same `apply_finding`, the preview can't drift from the real apply. Served at
`GET /loan-files/{id}/findings/{finding_id}/apply-preview` — read-only, **never commits** (a 400 for a
finding with no apply-spec; tenant-scoped 404 cross-company).

## The itemized dialog

`components/file/verification/view-fix-dialog.tsx` renders the new math:

- **The change** (+ its holder/type) in a callout.
- **Before / after, line-by-line**: Housing (PITI), each existing debt, the **NEW** debt line
  (highlighted with a `NEW` badge), the **total monthly debts** (with delta), and the **qualifying
  income**.
- The recomputed **back-end DTI** (before → after), and — highlighted — the **limit-status crossing**
  ("Within limit → Over limit").
- **Apply fix** (confirm → the existing real apply) / **Cancel** (no-op).

`finding-card.tsx`: apply-spec findings render "View fix" (needs the `fileId`, now threaded from
`findings-list.tsx`) opening the dialog; the dialog's "Apply fix" calls the existing `onApply`.

## Reversibility (for LP-98)

The real apply already records enough before-state to reverse: `applied_record` carries the created
`liability_id` (add_liability) / the `from`→`to` (correct_income). Verified here; **Undo is LP-98**.

## Tests

- Backend (`tests/services/test_finding_impact.py`): the preview is itemized (new line, +$500 total,
  higher back-end DTI, income unchanged); it **persists nothing** (file/DTI/liabilities/finding
  unchanged after a preview); it **matches the real apply** (apply for real → the numbers equal the
  prediction); a big debt flips the limit status pass → over; `has_apply_spec` gates it.
- Backend (`tests/api/test_finding_resolution_endpoints.py`): the endpoint returns the itemized impact
  without persisting; a non-apply-spec finding → 400; cross-company → 404.
- Frontend (`view-fix-dialog.test.tsx`, `finding-card.test.tsx`, `findings-list.test.tsx`): the dialog
  renders the detailed before/after + status crossing; Apply fix confirms (calls onApply) + closes;
  Cancel no-ops; error state; apply-spec findings show View fix (not a bare Apply).

## Scope boundary

Does **not** author new apply-specs for other rules (the flow is general; more specs are separate/future),
build a parallel impact computation (reuses the real apply — one source of truth), or build **LP-98**
(Undo + the Resolved-section placement). Apply-specs are currently rare (~undisclosed-debt).
