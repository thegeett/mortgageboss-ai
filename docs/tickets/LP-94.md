# LP-94 ‚Äî Re-run behavior: merge currently-detected, DROP no-longer-detected (open)

- **Ticket:** LP-94 (third ticket of the finding-presentation epic, LP-92..98)
- **Epic:** Phase 3.5 ‚Äî Finding presentation
- **Status:** Completed
- **Date:** 2026-07-01

## Summary

On a verification re-run, findings are now **reconciled in place** against the file's existing
findings by LP-93's normalized identity: a still-detected finding is **merged** (its row ‚Äî
id/notes/resolution ‚Äî kept), a no-longer-detected **open** finding is **dropped**, a genuinely-new
one is **added**, and a **resolved** finding is **retained** even when no longer detected (Undo/
audit depend on it). See **ADR-219**.

## Trace correction (report before the change)

The plan framed this as "reverse LP-81's *mark as no-longer-detected*." The trace found **no mark
mechanism**:

- The live re-run path is `run_cross_source` (the single-source `run_verification` engine has **no
  callers** in `app/`). It *superseded* (soft-deleted) every OPEN cross-source finding and
  re-emitted the fresh set (`_supersede_open_*` + emit loops).
- So it **already dropped** no-longer-detected open findings and **retained** resolved ones
  (supersede was open-only); LP-93 handled within-run dedup + preserve-on-re-detect.
- The LP-78.1 fingerprint cache (`api/verification.py:151`) short-circuits **before** the pass on
  unchanged inputs ‚Üí a no-op re-run touches nothing.

The **real gap** vs the locked Q4 design was the opposite: supersede+recreate **churned
still-detected OPEN findings** ‚Äî deleting + re-creating them as new rows each run, losing their id
and `details["notes"]`/history. So "merge keeps history" was not actually true.

## The fix

`app/services/finding_reconcile.py` (new) ‚Äî `reconcile_findings`, compared by LP-93 normalized
identity, replaces supersede+recreate in **both** cross-source paths:

| Case | Behavior |
|---|---|
| still-detected OPEN | **MERGE** ‚Äî keep the existing row (id, notes/history, resolution) |
| no-longer-detected OPEN | **DROP** (soft-delete) ‚Äî the list stays honest to the current state |
| still-detected RESOLVED | resolution preserved (not reopened/duplicated) |
| no-longer-detected RESOLVED | **RETAINED** ‚Äî `resolution_status`/`applied_record`/trail survive |
| genuinely-new | **ADD** |

- `app/services/cross_source_deterministic.py` ‚Äî `run_cross_source_deterministic` loads its live
  `xsrc.*` findings (`_live_deterministic_findings`), builds the fresh detections, and reconciles.
  `_supersede_open_deterministic_findings` is removed.
- `app/services/cross_source.py` ‚Äî the AI loop builds fresh AI findings (after the `fired_types`
  defer) and reconciles against its live AI findings, deduping against the deterministic set via
  `external_identities` (never dropping findings it doesn't own). `_supersede_open_cross_source_findings`
  is removed.

Within-run duplicates collapse (LP-93). An unchanged re-run drops nothing (the LP-78.1 cache
returns the prior run without a pass). No UI marker existed to remove ‚Äî dropped findings simply
leave the tab query (`only_active` + red/yellow).

## The careful case (why resolved findings are retained)

A resolved finding is a **completed processor action**, not clutter. Dropping it would erase the
record ‚Äî and an **APPLIED** finding's `applied_record` (the data change it made) plus **LP-98's
Undo** and the audit trail depend on it surviving. So "drop no-longer-detected" targets **OPEN**
findings only; resolved findings persist regardless of re-detection. This is confirmed against the
model (`resolution_status`, `applied_record`) and documented in ADR-219.

## Tests

`backend/tests/services/test_finding_reconcile.py` (9): still-detected keeps the same row; an open
finding's notes survive a re-run (true merge, not churn); new added; no-longer-detected open
dropped (gone + soft-deleted, not marked); still-detected resolved preserved; no-longer-detected
resolved retained; a no-longer-detected APPLIED finding keeps its `applied_record` (Undo-safe);
unchanged re-run drops nothing; tenant-scoped (dropping A's finding never touches B's).

## Verification

Backend-only. `ruff` / `mypy` (strict) / `pytest` ‚Äî **1399 passed** (+9). No regression on LP-81
merge, LP-93 dedup, LP-78.1 cache, or LP-92 labels.

## Scope boundary

Depends on LP-93 (normalized identity ‚Äî the comparison basis). Does **not** build the Resolved-
section UI (LP-95) or Undo (LP-98) ‚Äî it only ensures resolved findings are **retained** in the
data so those later tickets can build on them. No fuzzy matching (LP-93's deterministic identity).
