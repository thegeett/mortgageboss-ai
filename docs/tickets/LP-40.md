# LP-40 — Dev-Only PDF Text-Layer Extraction Tool (Reframed)

- **Ticket:** LP-40 — Dev-only PDF text-layer extraction tool (gated endpoint + deterministic utility)
- **Epic:** Epic 5 — Document Upload & Processing
- **Status:** Completed
- **Date:** 2026-06-12

## Summary

The production pipeline now reads documents with **AI directly** (full-document native
reading, LP-38/39), so deterministic PDF text extraction is **no longer a pipeline step**.
LP-40 (reframed) builds it as a **dev-only comparison tool**: a deterministic PDF text-layer
extractor exposed through a **production-gated**, **tenant-scoped** endpoint, so a developer
can inspect text-layer output on real documents and compare it against the AI's reading —
informing a possible future hybrid. Backend only.

## What Was Built

- **Extractor** (`app/services/pdf_utils.py`) — `PdfTextExtractionResult` (`text`,
  `page_count`, `has_text`, `extraction_ok`, `error_reason`) and async
  `extract_text_from_pdf(content: bytes)`: deterministic (PyMuPDF, no AI, no OCR), multi-page
  (joined with form feeds), async via `asyncio.to_thread`, graceful on bad input,
  metadata-only logging.
- **Dev router** (`app/api/dev.py`) — `POST /api/v1/dev/documents/{document_id}/extract-text-layer`
  + `TextLayerExtractionResponse`, mounted in `main.py` **only when `not settings.is_production`**.

## Purpose (a dev tool, NOT a pipeline step)

It does **not** feed classification/extraction (the AI reads documents directly), does **not**
update the `Document`, and does **not** route anything to `NEEDS_REVIEW`. It returns the
deterministic text-layer output for the developer to compare against the AI's reading,
informing the open question of whether to adopt a **hybrid** later (cheap deterministic text
for easy cases, AI for the rest).

## Production Gating

The dev router is included in `main.py` only under `if not settings.is_production:`. In
production it is **never mounted** → its routes are **absent (404)**. Gating is at the router
level (no production code path to the route), not a flag check inside the handler.

## Tenant Scoping (even for a dev tool)

The endpoint resolves the document with `get_document_for_company(db, document_id=...,
company_id=current_user.company_id)` → **404** if it isn't the caller's. A Company A user
cannot extract a Company B document. `company_id` comes from the authenticated user, never the
request. Unauthenticated → `401`.

## Text-Layer-Only + has_text (informational)

PDF only — a non-PDF document returns a clear `extraction_ok=False`,
`error_reason="text-layer extraction supports PDF only"` (no crash, no attempt). `has_text`
(stripped text ≥ 20 chars) is **informational** — an empty text layer suggests a scanned
image — **not** a routing signal (scans are the AI's job). No OCR.

## Library Choice

**PyMuPDF** (`pymupdf` / fitz) — fast, reads from an in-memory byte stream, exposes
`page_count` / `needs_pass` for encrypted handling, simple per-page `get_text()`. Wrapped
behind `pdf_utils.py`. Test PDFs are generated with PyMuPDF itself, so no extra fixture
dependency. (ADR-131.)

## Graceful Failure

Corrupt / not-a-PDF / truncated / empty bytes → `extraction_ok=False`, `has_text=False`, an
`error_reason`; password-protected → `error_reason="encrypted"`. The function **never raises**.

## Privacy

The extracted **text is returned** in the endpoint response (that is its purpose — for the
developer to read) but is **never logged**. `pdf_text_layer_extracted` logs only metadata
(`page_count`, `has_text`, `extraction_ok`, `error_reason`, `size_bytes`). A test seeds an SSN
into a PDF and asserts it appears in the returned text but not in any log line.

## Decisions Made

- **ADR-129** — deterministic PDF text extraction repositioned as a dev-only comparison tool
  (not a pipeline step).
- **ADR-130** — dev-gated endpoints: present only in non-production; still auth'd + tenant-scoped.
- **ADR-131** — PDF library: PyMuPDF.

> **ADR numbering:** LP-39 (modification) used 128; the next free numbers were **129–131**.

## Assumptions

- The **AI reads documents** in the pipeline (LP-38/39); this tool is for comparison only.
- The **dev-only UI button** that calls this endpoint is **LP-43**.
- **OCR / scanned handling** is the AI's job; `has_text` is informational.
- Whether to **promote a hybrid** into the pipeline is an open question this tool informs.
- Tests are AI-free (deterministic), so **no real API key** is needed.

## Verification Performed

`uv run pytest` → **458 passed** (23 new); ruff check + format, `mypy app/` (strict),
pre-commit all clean.

- **Extractor** — a text-layer PDF (text present, `page_count` correct, `has_text` True);
  multi-page (text from all pages); blank PDF → `has_text` False (`extraction_ok` True);
  short-text below threshold → `has_text` False; corrupt / truncated / empty bytes → graceful
  (`extraction_ok` False, no crash); the extracted text is **not logged**.
- **Endpoint** — success returns text + `has_text` + `page_count`; a non-PDF document → the
  "PDF only" response; **cross-tenant → 404**; **unauthenticated → 401**; **production gating**
  — applying `main.py`'s mount condition with `is_production` forced true leaves the route
  absent, and a production-configured app **404s** the dev route (present in dev, absent in
  prod).

## Notes

- **Developer use:** compare this endpoint's text-layer output against the AI's reading on real
  documents to decide whether a hybrid is worth adopting later.
- PyMuPDF needs a few narrowly-scoped, documented `# type: ignore[no-untyped-call]` (incomplete
  hints); its SWIG `DeprecationWarning`s are filtered in pytest config.
