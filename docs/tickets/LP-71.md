# LP-71 — Document Versioning + AI Staleness Detection

- **Ticket:** LP-71 — Document versioning (Model C) + date-driven staleness
- **Epic:** Phase 2 — Document Handling
- **Status:** Completed
- **Date:** 2026-06-25

## Summary

Two paired capabilities for documents changing over a file's life: **versioning**
(a newer document supersedes an older one, with the audit trail) and **staleness**
(flagging documents that have aged out), so the file's current, valid documents stay
clear and the stale/superseded ones are surfaced — not silently included. Full-stack.
See **ADR-183**.

## Versioning — Model C

New uploads are normal (current + standalone) — **multiples are normal** (sets of
statements/stubs are not replacements), so a same-type upload is never auto-treated as
a replacement (**no over-prompting**). **Explicit replace** supersedes a specific
current document: old → historical, new → current, **both kept** for audit, the need
**re-evaluates** against the new current version (LP-68 serialized). **Gentle
surfacing** is informational (client-side "N other pay stubs"), not blocking. The
**email-ingest `possible_duplicate` flag** is the mechanism for auto-ingested docs
(ingestion itself is later).

## Staleness Detection

Deterministic — **the extracted date (LP-60..64) vs. a recency window** (pay stub
~30d, bank statement ~60d) **or an expiration** (ID / insurance past its date) → flagged
with a reason. A **threshold, like DTI** — auditable, NOT a new heavy AI call (the AI
extracts the date; the rule judges). The processor **resolves** (replace / waive /
accept); auto-resolution is **V2**.

## Recency Windows — Refine With Priya

The windows are sensible industry-standard starters in a plain config dict
(`RECENCY_WINDOWS` / `EXPIRATION_RULES` in `app/documents/staleness.py`). Priya's
lenders (UWM, Sun-West) have exact windows that vary by program — editing the dict is
the whole knob.

## Package-Fitness Groundwork

Versioning (current vs. historical) + staleness (fresh vs. stale) → a document's
fitness: current + fresh → fit; historical (superseded) or stale-unresolved → flagged.
The package itself is **Phase 6**; the qualified status is partly **LP-72** — this is
the data.

## Helpful-Not-Blocking Warnings

Version history, the Replace control, calm staleness warnings (reason + resolve), and
the gentle duplicate hint — clear-but-calm; the processor decides.
