# LP-63 — Tier 1 Borrower-Info & Legal Documents

- **Ticket:** LP-63 — Tier 1 borrower-info & legal extractors
- **Epic:** Phase 2 — Document Handling (Tier 1 extractors)
- **Status:** Completed
- **Date:** 2026-06-18

## Summary

The fourth Tier-1 **extractor** batch: the borrower-info & legal cluster — the
**driver's license / government ID** (identity), the **divorce decree** (legal
obligations/awards), and the general **Letter of Explanation** (reused from
LP-60). Same established pattern, with **heightened PII care** (the ID) and the
**findings** dimension (the divorce decree). Backend only.

## What Was Built

- `drivers_license.py` + `prompts/extraction/drivers_license.txt` →
  `extract_drivers_license` (heightened PII).
- `divorce_decree.py` + `prompts/extraction/divorce_decree.txt` →
  `extract_divorce_decree` (with first-class `support_obligations` /
  `property_awards` lists).
- Registered `drivers_license` + `divorce_decree` in `EXTRACTORS`.
- **Reused** the LP-60 `letter_of_explanation` extractor for the general variant
  (no duplication) — only the prompt was lightly broadened to enumerate general
  subjects.

## The Established Pattern (followed, not reinvented)

Each follows **LP-39a**, with the **W-2** extractor (its SSN discipline) as the
template for the ID's PII: a typed core of `TypedField`s (coerced `value` +
`SourceLocation`) + an `additional_sections` catch-all; the `.failed()` result
interface; the shared tolerant parser; Decimals/dates; honest nulls; metadata-only
logging. The divorce decree adds first-class typed **lists** (the same
structured-rows extension the bank statement uses for transactions, ADR-061) —
**not** a new shape.

## The Typed Cores (sensible starters — refine with Priya)

- **Driver's license / ID** — `full_name`, `date_of_birth` (sensitive), `address`,
  `id_number_masked` (sensitive — masked), `issuing_state`, `issuing_authority`,
  `expiration_date`.
- **Divorce decree** — `party_1_name`, `party_2_name`, `effective_date`; plus the
  lists `support_obligations` (`obligation_type`, `amount`, `frequency`, `payer`)
  and `property_awards` (`description`, `awarded_to`).
- **LOE (general)** — `subject`, `explanation_summary`, referenced facts (LP-60,
  reused).

## Heightened ID PII (the critical discipline)

The driver's license is the most identity-dense document in the product. Applying
the **W-2 SSN discipline (ADR-147)** at its strictest:

- `id_number_masked` is captured **masked** (last 4; the model masks it).
- `date_of_birth` is captured for the Phase 3 identity cross-check.
- **No extracted value is ever logged** — only status / confidence / counts. The
  whole ID (name, DOB, ID number, address) is withheld from logs.
- The raw values live only in the tenant-scoped extraction JSON (ID number masked,
  DOB masked in display).
- A dedicated test asserts the DOB, the ID number, the name, and the address never
  appear in logs. **All ID test data is synthetic** — never a real identity
  document.

## Divorce Decree + Findings Sequencing

The decree's support obligations (alimony / child support) are the canonical
**undisclosed-obligation feedstock** Phase 3 cross-checks against the stated
liabilities. Because a decree can set more than one obligation, they are captured
as a **first-class typed list** (`support_obligations`, each with amount +
frequency + payer + source), alongside a `property_awards` list.

**Sequencing:** the obligations are **captured now**. Surfacing them as formal
**findings** — the structured observations the implications engine + Phase 3 read
— is **wired when the findings infrastructure exists (LP-66/67)**. This ticket
does **not** build findings infrastructure prematurely; nothing is lost (the
obligations are in the typed list today). A minimal findings hook was considered
and deliberately deferred — capture-now / wire-later is cleaner.

## LOE Reuse (not duplicated)

LP-60 already built `letter_of_explanation` with a general `subject` +
`explanation_summary` + referenced facts, and the catalog files it under
`borrower_info`. It already serves the general variant, so this ticket **reuses**
it — one LOE extractor, not two. The only change was a light prompt broadening to
enumerate general subjects (income / credit / address / name / large-deposit /
other); the schema, registry entry, and tests are unchanged.

## ID Expiration (validity / staleness)

The ID `expiration_date` is captured — an expired ID is invalid, so it feeds
validity / staleness checks (LP-71).

## Shaped for Phase 3

The ID feeds the **identity cross-check** (name/DOB/ID vs. the borrower record);
the divorce-decree obligations feed the **undisclosed-obligation cross-check**
(vs. the stated MISMO liabilities). The catch-all + the AI cross-source layer have
the rest of each document.

## Decisions Made

- **ADR-172** — the borrower-info/legal extractors: heightened ID PII (ID number +
  DOB captured but masked + never logged, like the SSN); the divorce-decree
  obligations captured as first-class lists now, formal findings sequenced to
  LP-66/67; the LOE reused; the ID expiration captured for staleness; accuracy
  honestly scoped.

## Assumptions

- Field sets are sensible starters **refined with Priya**.
- **Accuracy** is validated as real (synthetic / redacted — never real) documents
  flow through; **no samples were available** when this was built.
- **LP-66/67** build the findings infrastructure that surfaces the decree's
  obligations as findings + cross-checks them (Phase 3).
- The ID expiration feeds staleness (**LP-71**).
- **LP-64** completes Tier 1 (tax returns).

## Verification Performed

- **No sample documents** present (`/mnt/user-data/uploads/` absent), so the tests
  verify the **mechanism/shape**, not per-document accuracy (stated honestly). All
  ID test data is synthetic.
- **Each extractor** — returns the typed-core (+ list) + catch-all shape; per-field
  source locations; types coerced; honest nulls; all-null → `FAILED`; unparseable
  → `None`; `extract_*` handles success / AI failure / unsupported / empty
  gracefully.
- **PII (the critical check)** — the ID's DOB, ID number, name, and address never
  appear in logs; only a count is logged; the raw values remain available to the
  caller.
- **Divorce obligations** — each captured separately (child support AND alimony,
  not merged) with amount + frequency + payer + source; property awards captured;
  obligations count as content for status; empty/junk rows dropped.
- **ID expiration** — captured (for staleness).
- **LOE** — the general variant is the reused LP-60 extractor (registry asserts
  identity).
- **Registry routing** — `drivers_license` + `divorce_decree` + the reused
  `letter_of_explanation` route to their extractors (Tier 1, category Borrower
  Info); an extraction persisted.
- **No regression** — the existing extractors (incl. LP-60/61/62) unchanged and
  their suites pass.
- **Suite** — `uv run pytest` green (**901 passed**, +29 over LP-62); `ruff` +
  `ruff format` clean; `mypy` clean (129 source files).

## Notes
