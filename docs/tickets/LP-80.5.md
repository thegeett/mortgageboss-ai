# LP-80.5 — Editable Subject Property & Loan + target lender, value-recording audit, staleness on baseline edits

- **Ticket:** LP-80.5 — editable property/loan + target lender, from→to audit, verification staleness on baseline edits
- **Epic:** Phase 3 — Verification Engine & Calculators (unblocks LP-80)
- **Status:** Completed
- **Date:** 2026-06-29

## Summary

Makes the Overview's **Subject Property** and **Loan** sections editable (reusing the stated-financials
editing pattern), surfaces a **target lender** on the file (the LP-80 prerequisite — which wholesale lender
selects the overlay), and closes two gaps a code investigation found: property edits weren't audited, and no
stated/loan/property edit marked verification stale even though it changes the verification baseline. Edits
now record **from→to values** in the audit (superseding the LP-56 value-free posture) and **mark verification
stale** so the "re-run" prompt fires. See **ADR-197 / 198 / 199**.

## What Was Built

- **Editable Property + Loan** — the shared inline `EditableRow` (extracted to `editable-row.tsx`, given a
  `select` kind) backs a `PropertyEditor` and `LoanEditor`; the Property/Loan cards gain an Edit/Done toggle.
  They PATCH the existing `PATCH /loan-files/{id}/property` and `PATCH /loan-files/{id}` endpoints.
- **Target lender** — the file's existing `lender_id`, now processor-editable via a lender picker (the
  company's `lenders`), displayed on the Loan card with a "Set lender" affordance when unset. MISMO does not
  carry it (verified) → set on the Overview.
- **From→to value-recording audit** — a shared `audit_value` / `field_changes` helper records
  `{section, action, changes: [{field, from, to}]}` for stated/loan/property edits. Property edits, previously
  silent, are now audited with values.
- **Verification staleness on baseline edits** — stated/loan/property/lender edits call
  `mark_verification_stale`; the frontend invalidates `dti`/`ltv`/`verification` on these edits; the stale
  banner copy generalized to "The file changed".
- **Program-change confirmation** — changing Conv ↔ FHA prompts a confirmation (it swaps the rule set +
  overlay); other fields save inline.

## Reuse, Not Reinvent

`EditableRow` (dirty-tracking, save-only-changed-fields, `text`/`money`/`int`/`bool` + new `select`) is the
one editing mechanism across stated financials, property, and loan. `useStatedFinancialsEdit` and the new
`useUpdateProperty`/`useUpdateLoanFile` share the same invalidation set. No new editing pattern, no new
backend endpoints (property + loan PATCH already existed).

## The Target Lender (unblocks LP-80)

`lender_id` already existed on the model + `LoanFileUpdate` + `LoanFileDetail`. LP-80.5 makes it editable
(picker), displays it ("Target lender" row + a "Set lender" affordance), and marks verification stale on
change (it changes which overlay applies). **MISMO** does not carry a target wholesale lender — the 3.4
application export has only a LoanOriginationCompany (broker) + LoanOriginator party (verified against the
real file) — so the import leaves it null and it is a processing decision.

## From→To Value-Recording Audit (supersedes LP-56 value-free)

`detail` now carries `{section, action, changes: [{field, from, to}]}` (values encoded exactly — Decimals as
strings, enums as values, dates ISO). Stated + loan edits upgraded from value-free; **property edits, before
silent, are now audited with values**. The generic `FILE_UPDATED` type is kept (the `section` distinguishes
the kind) — no new enum, no migration.

**PII consequence (handled):** the activity_log now holds financial/PII-adjacent values, so it inherits the
stated data's posture — exposed only through the same auth + tenant-scoped surfaces that already show that
data. SSNs are out of scope (borrower editing excluded), so no raw SSN enters the log.

## Verification Staleness — Both Sides Now Trigger It

`mark_verification_stale` fired only on document changes; now stated/loan/property/lender edits trigger it
too — a baseline change on **either** side of the cross-source comparison sets the flag. Lifecycle/contact
fields (status, loan officer) do not. The LP-78.1 fingerprint already makes the re-run *real*; this adds the
*prompt*. DTI/LTV are read-time derived (already correct on next GET); the frontend invalidation just
refreshes already-open panels.

## Decisions Made

- **ADR-197** — editable property/loan + the target lender (reuse the pattern; program-change confirmed).
- **ADR-198** — audit posture change: record from→to values for stated/loan/property edits (supersedes LP-56
  value-free); the PII consequence (activity_log inherits the stated-data posture).
- **ADR-199** — verification staleness on baseline edits (both sides of the comparison; the LP-78.1
  fingerprint already makes the re-run real; the frontend invalidation fix).

## Assumptions

- Borrower editing remains out of scope (still display-only; also silent today — a future ticket).
- The note rate / amortization stay in the stated-data editor (not duplicated on the Loan card).
- DTI/LTV need no recompute trigger (read-time derived); the invalidation only refreshes open panels.

## Verification Performed

- **Stated edit** records `{field, from, to}` + marks stale; **add** records the added values + marks stale.
- **Property edit** is audited (`section=property`) with from→to values + marks stale (previously silent).
- **Target lender** set marks stale; **program** change marks stale; a **contact** field (loan officer)
  records its from→to but does **not** mark stale.
- **Tenant-scoped** — a cross-company property edit is 404.
- **Frontend** — Property/Loan editors render + save the changed field; the lender picker sets `lender_id`;
  the **program-change confirmation** fires (and cancel does not save); a property/loan edit **invalidates**
  the `dti`/`ltv`/`verification` queries (real QueryClient spy).
- **No regression** — backend **1227 passed**; frontend **202 passed**; `ruff`/`mypy --strict` clean;
  `pnpm lint`/`typecheck`/`build` pass. No migration (no schema change).
