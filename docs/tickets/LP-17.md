# LP-17 — Finding Model

- **Ticket:** LP-17 — Finding model with red/yellow/green status and resolution lifecycle
- **Epic:** Epic 2 — Database & Models (ninth ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Created the **`Finding`** model — one verification result (a red/yellow/green
flag) against a loan file. A finding records the **rule** that produced it
(`rule_id`, a flexible dotted string), a **severity** (`status`), a **category**,
a human `message`, structured `details` JSON, and a **resolution lifecycle** with
an audit trail (who/when/why). Findings belong to the **durable loan file**, so
their resolution state **persists across verification runs** (ADR-061).

This ticket builds the finding **record** + resolution helper; the rules that
produce findings are Phase 3, and the `Verification` run model is LP-18.

Run straight through (sequential, no mid-ticket gates) per the ticket.

## What Was Built

### Model (`app/models/finding.py`)

Three enums + the `Finding` model (Base + UUID/Timestamp/SoftDelete mixins).

| Column | Type | Notes |
| --- | --- | --- |
| `loan_file_id` | UUID FK → loan_files | required, indexed, `ondelete=CASCADE` (owned child) |
| `verification_id` | UUID | nullable, indexed, **no FK** (added LP-18, ADR-063) |
| `source_document_id` | UUID FK → documents | nullable, indexed, `ondelete=SET NULL` |
| `rule_id` | `VARCHAR(256)` | required, **indexed**, flexible dotted string (no CHECK) |
| `status` | `FindingStatus` enum | required, indexed, VARCHAR + CHECK |
| `category` | `FindingCategory` enum | required, indexed, VARCHAR + CHECK |
| `message` | text | required, human-readable |
| `details` | JSON | required, default `{}`, structured supporting data |
| `resolution_status` | `FindingResolutionStatus` enum | default `OPEN`, indexed, VARCHAR + CHECK |
| `resolution_note` | text | nullable |
| `resolved_by_user_id` | UUID FK → users | nullable, `ondelete=SET NULL` |
| `resolved_at` | timestamptz | nullable, timezone-aware |

Relationships: `loan_file` (back_populates `findings`), `source_document` and
`resolved_by` (one-directional, nullable). `LoanFile` gained `findings`
(`cascade="all, delete-orphan"`).

### Status model & resolution lifecycle

- **Severity:** `red` (blocking) / `yellow` (review) / `green` (passed).
- **Resolution:** `open` → `resolved` / `accepted_risk` / `waived`, with a trail
  (`resolved_by_user_id`, `resolved_at`, `resolution_note`). `accepted_risk`
  captures accepting a yellow flag with a compensating factor.
- **Persistence:** findings belong to the durable loan file, so resolution state
  survives re-verification (ADR-061); cross-run matching is Phase 3.

### Resolution helper (`app/services/findings.py`)

`resolve_finding(db, *, finding, resolution_status, user_id, note=None)` writes
the status + trail together so the audit record is always consistent. For a
resolution it sets `resolution_status`, `resolution_note`, `resolved_by_user_id`,
and `resolved_at = utcnow()` (timezone-aware). For **re-opening**
(`resolution_status=OPEN`) it clears the trail (note/resolved_by/resolved_at →
None) — a re-opened finding is no longer resolved. Uses `flush` (caller owns the
transaction).

### Migration (`2b2266d15127_create_findings.py`)

Creates `findings` with: CASCADE FK to loan_files; SET NULL FKs to documents and
users; `verification_id` as a nullable UUID with **no FK constraint**; CHECK
constraints on `status`/`category`/`resolution_status`; and seven indexes
(`loan_file_id`, `verification_id`, `source_document_id`, `rule_id`, `status`,
`category`, `resolution_status`). Hand-cleaned to the repo style.

## Decisions Made

- **ADR-060** — Finding status (red/yellow/green) + resolution lifecycle with a
  trail.
- **ADR-061** — Findings belong to the loan file; resolution persists across runs
  (cross-run matching is Phase 3).
- **ADR-062** — `rule_id` as a flexible dotted-namespace string (app-layer
  registry, not a DB enum).
- **ADR-063** — `verification_id` column added before its FK target; LP-18 adds
  the constraint.

> **ADR numbering:** the ticket referenced ADR-062…065. The actual next free
> numbers were **060–063** (LP-16 used 057–059). Used 060–063; references to
> scoping/cascade point at the real **ADR-052**.

## Assumptions

- **Verification rules are Phase 3** — this ticket stores findings; nothing
  produces them yet.
- **The `Verification` run model is LP-18** — which also adds the
  `verification_id` FK constraint to `findings`.
- **Cross-run finding matching is Phase 3** — resolution state persists on the
  finding; carrying it forward between runs is later logic.
- **`rule_id` is `VARCHAR(256)`** (MediumStr) — dotted slugs are short, but the
  width leaves room; no DB constraint on values.

## Notes

- `verification_id` deliberately has **no FK constraint** until LP-18 — verified
  in psql (only three FK constraints exist). LP-18 must add
  `fk_findings_verification_id_verifications`.
- `source_document_id` / `resolved_by_user_id` use `SET NULL` so a finding
  outlives a removed document or user (the note still records the decision).
- `resolve_finding` supports re-opening (`OPEN` clears the trail); "who re-opened"
  is left to the future activity log, not stored on the finding.
