# LP-10 — Model Layer Foundations

- **Ticket:** LP-10 — Model layer foundations (test infra, shared types, enum pattern, query helpers)
- **Epic:** Epic 2 — Database & Models (second ticket)
- **Status:** Completed
- **Date:** 2026-06-10

## Summary

Built the remaining model-layer foundations that every subsequent model and
model-test depends on: reusable column types (`Money`, string lengths), a
consistent `StrEnum` pattern with a `str_enum()` helper, a `only_active()`
soft-delete query helper, and the **test database infrastructure** — an
isolated, auto-created test database with per-test transaction-rollback
isolation. Proved it all with nine infrastructure tests against a throwaway
model that is deliberately confined to the test module.

> **Scope note (LP-9 / LP-10 consolidation):** LP-10's original phase-plan
> scope was "core SQLAlchemy base and mixins." That content (`Base`, mixins,
> naming convention, `utcnow`) was delivered in **LP-9** alongside Alembic, so
> LP-10 was repurposed to the *remaining* model-layer foundations. The base
> classes were **not** recreated here.

Implemented in three reviewed stages: (1) shared types + enums + helper,
(2) test-DB fixtures, (3) infrastructure tests.

## What Was Built

### Foundations added

| Piece | File | Purpose |
| --- | --- | --- |
| Shared column types | `app/models/types.py` | `Money` (`Numeric(14,2)`→Decimal); `ShortStr`/`MediumStr`/`LongStr` + length constants |
| Enum pattern | `app/models/enums.py` | `StrEnum` convention, `RecordStatus` example, `str_enum()` helper |
| Soft-delete helper | `app/models/helpers.py` | `only_active(stmt, model)` → adds `WHERE deleted_at IS NULL` |
| Test infrastructure | `tests/conftest.py` | `test_engine` (session) + `db_session` (function, rollback-isolated) |

`app/models/__init__.py` re-exports all of the above.

### Test infrastructure

- **`test_engine`** (session-scoped): derives `<dev_db>_test` from
  `settings.database_url` (or honors `TEST_DATABASE_URL`), auto-creates the
  database via the `postgres` maintenance DB with AUTOCOMMIT, builds the schema
  with `create_all`, and drops it at session end. A safety assertion refuses to
  run if the resolved name equals the dev database.
- **`db_session`** (function-scoped): opens a connection, begins a transaction,
  binds an `AsyncSession`, yields it, and rolls back afterwards — tests never
  commit and leave no residue.
- Both fixture and test event-loop scopes are set to `session` in
  `pyproject.toml` so the async engine, sessions, and tests share one loop
  (asyncpg connections are loop-bound).

### Infrastructure tests (`tests/models/test_infrastructure.py`)

Nine tests against a throwaway `Widget` model: working session; create/query;
soft delete + `only_active`; `Money`/`Decimal` round-trip; enum round-trip
(incl. raw-value check); enum default; two isolation tests; and a no-leak test.

The throwaway model uses its **own `MetaData`** (a separate declarative base),
so it never registers on the app's `Base.metadata`. Its table is created/dropped
on the test engine by an autouse fixture; rows still roll back per test. Because
Alembic autogenerate reads `Base.metadata` (importing `app.models` only), the
throwaway can never reach a migration — and a test asserts that directly.

## Decisions Made

- **ADR-037** — Database-backed enums as VARCHAR with CHECK (`native_enum=False`),
  value-persisted via `str_enum()`.
- **ADR-038** — Money stored as `Numeric`/`Decimal`, never float.
- **ADR-039** — Test database isolation via transaction rollback.
- **ADR-040** — No generic repository/CRUD abstraction in V1.

## Assumptions

- PostgreSQL is running locally (dev DB `mortgageboss_dev`, superuser
  `mortgageboss` with `CREATEDB`); the `postgres` maintenance DB exists.
- Tests use `create_all`, not migrations (migrations verified separately).
- pytest-asyncio 1.4 with both loop scopes set to `session`.

## Notes

- **Money is always `Decimal`**, never float (ADR-038).
- **Enums use `str_enum()`** so the value (not the name) is stored — every enum
  column must use it (ADR-037).
- Soft-delete filtering is **explicit** (`only_active`), never a hidden global
  filter.
- No generic repository layer (ADR-040) — services write explicit queries.
