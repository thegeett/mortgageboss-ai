# LP-46 — Error Handling & User Feedback

- **Ticket:** LP-46 — error handling & user feedback
- **Epic:** Epic 6 — Testing, Polish & Phase 1 Completion
- **Status:** Completed
- **Date:** 2026-06-12

## Summary

Makes the system **fail gracefully and informatively** across the whole stack. Backend: one
consistent, SAFE error envelope for every API error + a global exception handler (no leaks/PII;
full detail logged server-side). Frontend: an axios error-normalization layer, a React error
boundary (no white screen), friendly/specific error states with **retry** across the main
surfaces, and consistent mutation feedback. This unifies error handling built piecemeal in Epics
1–5 into consistent *communication* — the underlying handling logic (pipeline resilience, AI
graceful failures, tenant 404s) is unchanged.

## What Was Built

**Backend — consistent SAFE envelope + global handler (`app/core/errors.py`)**

- The envelope: `{"error": {"type": str, "message": str, "details"?: [{"field", "message"}]}}`.
- Handlers (registered in `app/main.py` via `register_exception_handlers`):
  - **Unhandled `Exception` → 500** with a SAFE generic message
    (`"An unexpected error occurred. Please try again."`). Full detail logged as PII-safe
    **metadata only** (`error_type`, `path`, `method`) — never the body, a value, or a stack.
  - **`HTTPException` → the envelope** with its safe `detail` and a `type` derived from the status
    (`not_found`, `unauthorized`, `forbidden`, `conflict`, …).
  - **`RequestValidationError` → 422** with field-level `details` (`field` + Pydantic message).
- Audited every endpoint's `HTTPException` detail — all are already safe, generic strings ("Loan
  file not found", "Document not found"); nothing leaks.

**Frontend — axios layer + error boundary + states + feedback**

- **`lib/errors/api-error.ts`** — `normalizeError()` / `getErrorMessage()`: any throw → one
  `{ kind, status, message, details }`, reading the envelope (legacy `detail` fallback + safe
  generic default), classifying network/auth/not_found/validation/server.
- **Session expiry** — the axios refresh layer now redirects a dead session to
  `/login?...&reason=session_expired`; the login form shows a "your session expired" notice.
- **`components/error-boundary.tsx`** — a class `ErrorBoundary` + friendly `DefaultErrorFallback`
  (Try again / Reload). Mounted top-level in `Providers` (clears the query cache on reset) and
  around the app-shell content (a page crash keeps the sidebar/header). **No white screen.**
- **`components/ui/error-state.tsx`** — `ErrorState` (panel) + `InlineErrorState` (compact), each
  with an optional **Retry**. Applied to: the documents list, the drawer's extraction, and the
  overview sections (borrowers/property/loan/needs/activity) — each Retry re-runs its query.
- **Consistent feedback** — upload / override / delete / create show success and a safe normalized
  failure message via sonner; the dropzone's error mapper now uses `normalizeError`.

## Backend Error Contract

| Case | Status | Body (envelope) |
|---|---|---|
| No / bad token | 401 | `{"error":{"type":"unauthorized","message":"Not authenticated"}}` |
| Cross-company / missing | 404 | `{"error":{"type":"not_found","message":"…not found"}}` |
| Bad request body | 422 | `{"error":{"type":"validation_error","message":"Some fields need your attention.","details":[{"field","message"}]}}` |
| Bad credentials | 401 | `{"error":{"type":"unauthorized","message":"Invalid email or password"}}` |
| Unexpected failure | 500 | `{"error":{"type":"internal_error","message":"An unexpected error occurred. Please try again."}}` |

Server-side logging for a 500 is metadata-only (`error_type`, `path`, `method`) — never the
request body or any value.

## Frontend Error UX

- **Normalization:** one shape for the whole app; components display `message`, branch on `kind`.
- **401 / session:** redirect to login with `reason=session_expired` → the user is told why, not
  silently bounced.
- **Error boundary:** friendly fallback + Retry instead of a white screen; the raw error is
  console-only (never rendered).
- **Specific states + retry:** load failures show an inline panel with a Retry that refetches; the
  file 404 stays "doesn't exist or no access"; network errors read "couldn't connect".
- **Mutation feedback:** consistent sonner success/failure with safe messages.

## Safety (no leaks / no PII)

- Clients never receive a stack trace, internal path, DB error text, or PII — the 500 message is
  generic; the validation messages describe the *constraint*, not the submitted value.
- Logs carry metadata only (error type, path, method), never request bodies, extracted values,
  SSNs, or account numbers. The error boundary logs to the console (dev only) and never renders the
  error text.
- Tested explicitly: the envelope test asserts forbidden fragments (`Traceback`, `/app/`,
  `sqlalchemy`, `asyncpg`, `SELECT `, the secret exception text) never appear in any error body.

## Unifies Existing Patterns

- **Pipeline resilience / AI graceful failures** (LP-42/39) — already surface `processing_error` /
  `NEEDS_REVIEW` (shown by LP-43, with the override in LP-44); LP-46 keeps that communication
  consistent and adds normalized mutation feedback around it.
- **Tenant-isolation 404s** (verified LP-45) — the frontend already shows the file "no access" state
  (`FileError`); LP-46 adds a Retry to its transient-error variant.
- The underlying handling logic is unchanged — only the *communication* is now consistent + friendly.

## Decisions Made

- **ADR-154** — Consistent API error envelope + global exception handler (safe by default).
- **ADR-155** — Frontend error UX: axios normalization, error boundary, specific states + retry.

## Assumptions

- No observability stack yet (Sentry, etc.) — Phase 7; this is consistent envelopes + safe logging.
- Loading states & skeletons are LP-47 (siblings; this touches only the error side).
- Mechanisms over bespoke: the envelope/handler/normalizer/boundary/standard states are the
  deliverable; a few high-value specific messages (session expired, no access, network, processing
  failed) are crafted.
- The underlying error *handling* (pipeline resilience, AI graceful failures, tenant scoping) is
  unchanged — only the communication is unified.

## Notes

- The error mechanisms (`error_body` + handlers, `normalizeError`, `ErrorBoundary`, `ErrorState`)
  are reused everywhere — new surfaces get consistent error handling for free.
- The response-shape change (`{detail}` → envelope) is read by `normalizeError` with a legacy
  `detail` fallback, so any not-yet-migrated reader still degrades safely.
