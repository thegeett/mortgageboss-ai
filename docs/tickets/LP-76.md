# LP-76 — DTI Calculator (transparent, auto-populated, override-able)

- **Ticket:** LP-76 — DTI calculator (the headline "replace ChatGPT" win)
- **Epic:** Phase 3 — Verification Engine & Calculators (Arc A)
- **Status:** Completed
- **Date:** 2026-06-28

## Summary

The DTI calculator — THE central mortgage-qualification number, done **better than ChatGPT**:
**transparent** (every component + the explicit formula), **auto-populated** (from the file's
structured data — no re-entry), and **trustworthy** (deterministic math, override-able with an
audit log). Coupled to findings (the unresolved alert + recompute-on-apply). Full-stack,
frontend-prominent. See **ADR-190** and
[`phase-3-verification-design.md`](../phase-3-verification-design.md) §10.

## What Was Built

- **The deterministic DTI math** (`backend/app/verification/dti.py`, pure) — front/back ratios +
  amortized monthly P&I; `Decimal`; no AI.
- **The calculator service** (`backend/app/services/dti.py`) — auto-population, override
  application, the effective-limit resolution, and the findings coupling →
  `build_dti_calculation`; `set_dti_override` / `clear_dti_override` (audited).
- **The override model** (`backend/app/models/dti_override.py`) + migration `c1f7a4e9d250`
  (table + the `dti_overridden` activity type).
- **The schemas** (`backend/app/schemas/dti.py`) + **the API** (`backend/app/api/dti.py`):
  GET the calculation, PUT/DELETE an override (recompute in the response).
- **The frontend** — `lib/types/dti.ts`, `lib/api/dti.ts` (query + override mutations priming
  the cache), `components/file/dti/dti-calculator.tsx` (the transparent calculator), surfaced on
  the **verification tab**.
- **Tests** — backend math/service/api; a frontend component test.

## The Transparency (the feature)

Front-end DTI = housing ÷ income; back-end DTI = (housing + monthly debts) ÷ income. The value is
not the arithmetic — it's that the calculator **shows its work**:

- **Income** itemized (each stated income source → gross monthly income).
- **Housing** itemized — PITI (computed Principal & Interest + extracted Taxes & Insurance) + MI +
  HOA → the housing payment.
- **Each debt** itemized (every monthly obligation → total monthly debts).
- **The explicit formula** is shown verbatim, then with the file's actual numbers substituted.
- **Program limits side-by-side** — the computed back-end DTI against the **effective** limit
  (LP-74's investor rule patched by any lender overlay) with a pass/over status.

A black-box DTI (ChatGPT) is untrustworthy; one that shows every input and the formula is
trustworthy. The math is **deterministic** — correct by construction.

## Auto-Population (the "better than ChatGPT")

The calculator opens **already filled** from the file's structured data — income from
`StatedIncomeItem`, debts from `StatedLiability`, P&I amortized from the loan terms, and
taxes/insurance/HOA from the current document extractions. It reads the *same* structured data
the rules engine evaluates. Review + adjust, not enter from scratch.

## Override + Audit (the human controls)

Any field is override-able: a `DtiOverride` row (one per `(file, field_key)`, unique; clearing
soft-deletes) holds the override amount, **takes precedence** over the auto value, and
**persists**. Every set/clear is audited (`ActivityType.DTI_OVERRIDDEN`, recording the prior
value, by whom). The auto values are a trustworthy starting point, not a cage.

## Real-Time + Findings Coupling (the interlock)

- **Real-time recalc** — the override endpoints return the recomputed calculation in the response;
  the UI primes the query cache from one round-trip.
- **The unresolved-findings alert** — `build_dti_calculation` queries open in-scope findings
  (LP-75's `open_in_scope_findings`, Balanced default) and the UI warns when the numbers may be
  incomplete.
- **Recompute on applied findings** — because the calculation reads the structured data live,
  applying a finding (LP-75's hook adds a liability) makes the next calculation recompute higher.
  **LP-76 is a recompute consumer of LP-75's apply hook** — the AI↔deterministic interlock landing
  in the calculator.

## Deterministic

The DTI is pure Python arithmetic — **no AI computes it**. The AI's role was upstream (extracting
the income/debt values) and adjacent (findings that correct the inputs). The calculation is
auditable and correct by construction — the trust ChatGPT can't offer.

## Verification Performed

- **The math** — amortization (a textbook value + the zero-rate case + missing-input None);
  front/back ratios; breakdown sums; half-up rounding; income-zero → undefined.
- **Auto-population** — the calculator opens filled from stated income/liabilities + computed P&I.
- **Override** — takes precedence, recomputes, persists, clears to the auto value, and is audited
  with the prior value; an unknown field is rejected.
- **Effective limit** — the program default (50), and the sample overlay tightening it to 45.
- **Findings coupling** — the unresolved alert on an open in-scope finding; recompute-on-apply
  (applying an $800 obligation raises the back-end DTI).
- **Endpoints** — GET returns the itemized calc; PUT/DELETE recompute in the response; cross-company
  is 404.
- **Frontend** — the component renders the ratios, breakdown, formula, and limit; the unresolved
  alert; over-limit in red; the inline override triggers the mutation; loading/error.
- **No regression** — backend **1126 passed**; frontend **159 passed**; `ruff`/`mypy --strict`
  clean; `pnpm lint`/`typecheck`/`build` pass; the migration applies + reverses.

## Notes

  deterministic DTI trustworthy where ChatGPT's black box isn't.
- **The findings coupling is the interlock** — applying a finding changes the structured data →
  the DTI recomputes; LP-76 consumes LP-75's hook.
