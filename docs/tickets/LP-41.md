## Summary

Set up **Celery** вЂ” the background task queue so document processing runs **asynchronously**,
outside the upload request. **Infrastructure only:** the configured Celery app, the Redis
broker connection, a task base class with the **syncв†’async bridge**, validation tasks, a
worker run command + Compose service. LP-41 does **not** process documents вЂ” LP-42 adds the
real tasks (read bytes в†’ classify в†’ extract) that run on this. Backend only.

## What Was Built

- **Celery app** (`app/tasks/celery_app.py`) вЂ” `Celery("mortgageboss", broker=..., backend=...)`
  from settings; **JSON-only** serialization (no pickle), UTC, `task_track_started`, soft/hard
  time limits (120s/180s), `include=["app.tasks.health"]` (LP-42 appends its module). Imports
  with **no live broker**.
- **Task base** (`app/tasks/base.py`) вЂ” `run_async`, `task_session`, and `BaseTask`
  (logging context).
- **Validation tasks** (`app/tasks/health.py`) вЂ” `ping` (в†’ `"pong"`) and `db_ping` (async
  `SELECT 1` via the bridge в†’ `"db-ok"`).
- **Settings** (`app/core/config.py`) вЂ” `celery_broker_url` / `celery_result_backend`
  (computed, defaulting to `redis_url`; optional `CELERY_BROKER_URL` /
  `CELERY_RESULT_BACKEND` overrides). `.env.example` documents them.
- **Worker** вЂ” `backend/Dockerfile` (+ `.dockerignore`) and a `worker` service in
  `docker-compose.yml` (behind the `worker` profile); README run instructions.

## The Queue Model

Document processing (PDF read + up to two AI calls) is too slow for the upload request, so it
runs on a **Celery worker** вЂ” a **separate process** from the API, both backed by the same
Redis (broker/result) and Postgres. Upload returns fast; the UI polls. **Status is tracked via
`Document.status`** (the DB is the source of truth, LP-42); Celery's result backend is
available but secondary. Serialization is **JSON only** вЂ” pickle is disabled (an RCE vector if
the broker is compromised).

## The Async Bridge

Celery tasks are sync; this codebase is async (DB, AI wrapper, storage). `app/tasks/base.py`
bridges them:

- **`run_async(coro)`** в†’ `asyncio.run` вЂ” **a fresh event loop per task** (simplest correct
  V1).
- **`task_session()`** в†’ an async session from a **fresh `NullPool` engine created inside that
  per-task loop**, disposed when done. This is deliberate: the app's module-level `engine`
  (`app/core/database.py`) is bound to the loop that first used it, so reusing it across the
  per-task loops would raise *"attached to a different loop"* (asyncpg connections are
  loop-bound). A per-task engine with no pooling avoids that entirely.
- **`BaseTask`** binds `task_name` / `task_id` into the structlog context for each call
  (metadata only) and exposes `run_async`.

**Caveats (documented):** a new event loop and new DB connections per task вЂ” fine at V1 volume;
**revisit loop/pool reuse** if throughput grows. Tasks must do async work inside a
`run_async`-driven coroutine and use `task_session()` (not the request-scoped `get_db`), and
stay sync at the Celery boundary (`asyncio.run` can't run inside an already-running loop).

## Running the Worker

```bash
# Local (alongside the dev server)
cd backend
uv run celery -A app.tasks.celery_app worker --loglevel=info
# macOS/Colima: add --pool=solo if the default prefork pool misbehaves

# Container (behind the Compose "worker" profile; default `up` stays infra-only)
docker compose --profile worker up worker   # needs backend/.env
```

## Validation

With a worker running, from a Python shell:

```python
from app.tasks.health import ping, db_ping
ping.delay().get(timeout=10)     # -> "pong"
db_ping.delay().get(timeout=10)  # -> "db-ok"  (proves the async DB bridge end-to-end)
```

## Decisions Made

- **ADR-132** вЂ” Celery + Redis for background document processing.
- **ADR-133** вЂ” sync Celery tasks run async code via a per-task event loop.

## Assumptions

- **LP-42** adds the real document-processing tasks and enqueues them from upload; **status is
  tracked via `Document.status`** (DB).
- **No Flower/Beat** (monitoring/periodic) yet.
- Broker and result backend **share the configured Redis** in V1 (Celery namespaces its keys;
  a separate DB index is a later tuning).
- The **per-task event loop** is revisited if volume grows.
- The backend isn't otherwise containerized; the `Dockerfile` exists for the worker container
  (and future API containerization); the API/worker primarily run locally via uv.
