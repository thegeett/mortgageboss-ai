# LP-13 — Loan File Core Model

- **Ticket:** LP-13 — Loan file core model with three-identifier design (ADR-036)
- **Epic:** Epic 2 — Database & Models (fifth ticket)
- **Status:** Completed
- **Date:** 2026-06-10

## Summary

Created the **`LoanFile`** model — the **central entity** of the system, the
thing everything else (borrowers, properties, documents, verification findings,
conditions) hangs off. The headline of this ticket is the **three-identifier
design** from ADR-036: a UUID primary key (internal), a non-sequential readable
**display ID** (`LF-XXXX`, human reference), and a cryptographically unguessable
**inbox token** (the borrower email capability). Identifier generation is a
dedicated, security-sensitive service built and verified in isolation first.

Also added the `LoanFileStatus` and `LoanPurpose` enums, the nullable lender
relationship, a minimal `create_loan_file` service, and the migration.

Implemented in **three reviewed stages**: (1) ID generation service + its tests,
(2) `LoanFile` model + enums + migration, (3) `create_loan_file` service + model
tests + docs.

## What Was Built

### ID generation service (`app/services/loan_file_ids.py`)

The security-sensitive core, built and tested first in isolation:

- `generate_display_id()` — `"LF-"` + 4 chars from the unambiguous alphabet
  `23456789ABCDEFGHJKMNPQRSTUVWXYZ`, via `secrets.choice`.
- `generate_inbox_token()` — `secrets.token_urlsafe(12)` (~16 chars, ~96 bits).
- `generate_unique_display_id(db)` — collision-checked against existing files,
  regenerates on the rare hit, raises after `MAX_DISPLAY_ID_ATTEMPTS = 10`.
- Uses the `secrets` module only — **never `random`**. The inbox token is
  generated **independently** of the display ID (never derived from it).

### Model (`app/models/loan_file.py`)

- `LoanFile(Base, UUIDMixin, TimestampMixin, SoftDeleteMixin)`, table
  `loan_files`.
- Identifiers: `display_id` and `inbox_token` (both unique, indexed,
  non-nullable).
- `company_id` FK → companies (RESTRICT, required); `lender_id` FK → lenders
  (RESTRICT, **nullable**).
- Nullable loan attributes: `loan_program`, `loan_purpose`, `loan_amount`
  (`Money`).
- `status` (`LoanFileStatus`, default `DRAFT`); free-text `loan_officer_name`/
  `loan_officer_email`.
- `get_inbox_address()` → `lf-{inbox_token}@inbox.mortgageboss.ai`
  (`INBOX_DOMAIN` module constant).
- Enums: `LoanFileStatus` (8 states), `LoanPurpose` (PURCHASE, REFINANCE);
  `LoanProgram` reused from `app/models/lender.py`.
- Relationships added to `Company` (`loan_files`) and `Lender` (`loan_files`),
  non-destructive (RESTRICT FKs, ADR-044).

### Creation service (`app/services/loan_files.py`)

- `create_loan_file(db, *, company_id, lender_id=None, loan_program=None,
  loan_purpose=None, loan_officer_name=None, loan_officer_email=None)`.
- Generates a unique display ID + an independent inbox token, sets status
  `DRAFT`, `db.flush()` (not commit — caller owns the transaction).
- Minimal by design: no needs-list generation or activity logging yet (LP-30).

### Migration

`..._create_loan_files.py` creates `loan_files` with: `fk_loan_files_company_id_companies`
and `fk_loan_files_lender_id_lenders` (both RESTRICT), unique indexes on
`display_id` and `inbox_token`, indexes on `company_id`/`lender_id`, CHECK
constraints on all three enum columns, and `loan_amount` as `NUMERIC(14,2)`.

## The Three Identifiers (ADR-036)

| Identifier | Example | Purpose | Exposure | Generation | Security posture |
| --- | --- | --- | --- | --- | --- |
| `id` (UUID PK) | `7f3a8b2c-…` | FKs, joins, internal references | Never exposed | `uuid4` (UUIDMixin) | Internal only |
| `display_id` | `LF-7K3M` | Human reference (UI, conversation, email subjects) | Authenticated users only | Non-sequential random, unambiguous alphabet, collision-checked, globally unique | **Identifier** — predictability is low-risk; access gated by auth + company scoping |
| `inbox_token` | `a7k4nq2x9m3p` | Borrower inbox email address | Public (in the address) | `secrets.token_urlsafe(12)`, ~96 bits, **independent** of display ID | **Capability** — possession grants the ability to send documents in; must be unguessable |

The key principle: an **identifier** names a thing (access controlled
separately), a **capability** grants power by possession (must be unguessable).
The display ID is the former, the inbox token the latter — hence they are
generated separately, and the inbox token is never derived from the display ID.

## Status Lifecycle (ADR-049)

```
DRAFT → IN_PROCESSING → READY_TO_SUBMIT → SUBMITTED
      → IN_CONDITIONS → CLEAR_TO_CLOSE → CLOSED
```

plus `WITHDRAWN` (terminal, reachable from any earlier state). Default `DRAFT`.
Transitions are **not** state-machine-enforced in V1 (any-to-any at the model
level); workflow enforcement can come later. Stored as VARCHAR + CHECK (ADR-037).

## Decisions Made

- **ADR-048 — Display ID globally unique.** Random codes make global uniqueness
  cheap and avoid cross-company ambiguity; contrasts with per-company lender
  slugs (ADR-045).
- **ADR-049 — Loan file status lifecycle.** The 8-state lifecycle above; no state
  machine in V1; VARCHAR + CHECK so adding a status is a simple migration.
- **ADR-050 — ID generation in the service layer, not the model.** Secure
  randomness + collision checking is logic, not declarative column config; keeps
  the model clean and the generation independently testable.
- Implements **ADR-036** (loan file identifier strategy), written earlier.

## Assumptions

- The inbox domain (`inbox.mortgageboss.ai`) is a module constant for now; it may
  move to settings if it needs to vary per environment. Inbound email routing is
  Phase 4 — this ticket only generates the token.
- The originating loan officer is an **external party**, stored as free text, not
  a FK to `users`.
- `loan_amount`, `loan_program`, `loan_purpose`, and `lender_id` are nullable
  because they may arrive later (MISMO import in Phase 1.5, or processor action).
- `MAX_DISPLAY_ID_ATTEMPTS = 10` is ample given ~924k codes; the unique DB index
  is the final safety net regardless.

## Verification Performed

- **Security properties** (`test_loan_file_ids.py`, 10 tests):
  - Display IDs are `LF-` + 4 chars; **no ambiguous chars** (`0O1IL`) across 2000
    generations, and the alphabet constant excludes them.
  - Inbox tokens are ≥16 url-safe chars; **1000 generations all unique**.
  - **Independence** — over 1000 pairs, the token never contains the display ID
    or its code, and when a display code recurs the tokens still differ (so the
    token is not a function of the display ID).
  - Collision-checked generation **regenerates** on a forced collision and the
    guard **raises** after the max attempts (DB mocked to isolate the logic).
- **Model + service** (`test_loan_file.py`, 12 tests):
  - Creation via `create_loan_file`: identifiers populated/formatted, status
    defaults `DRAFT`, optional attributes null then settable.
  - `get_inbox_address()` returns the correct format.
  - Identifiers unique across 25 files; **unique constraint** rejects a forced
    duplicate `display_id`.
  - All four relationship directions load.
  - Soft delete + `only_active`; **tenant isolation** via `scope_to_company`.
  - **Enum CHECK rejection** — raw SQL `UPDATE … SET status='banana'` raises
    `IntegrityError` (confirms the LP-11 `str_enum` fix at the DB level).
  - `create_loan_file` **flushes, not commits** (row visible, transaction open).
- **Migration / DB** (psql against the dev database):
  - CHECK constraints present: `ck_loan_files_loanfilestatus`,
    `ck_loan_files_loanprogram`, `ck_loan_files_loanpurpose`.
  - FKs RESTRICT, unique indexes on `display_id`/`inbox_token`, `NUMERIC(14,2)`.
  - `upgrade head` applies; `downgrade -1` drops the table cleanly; re-upgrade
    works.
- **Quality gates:** full suite **58 passed**; `ruff check`, `ruff format
  --check`, `mypy app/` all clean.
