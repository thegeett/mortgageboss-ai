# LP-19 — Needs Item Model

- **Ticket:** LP-19 — NeedsItem model (needs list) with lifecycle and origin tracking
- **Epic:** Epic 2 — Database & Models (eleventh ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Created the **`NeedsItem`** model — items on a loan file's "needs list": the
outstanding requirements that move a file forward ("need 2023 W-2", "need LOE for
large deposit"). It is the processor's running checklist — a first-class entity
(not derived purely from findings) with a lifecycle, an origin, a priority, an
optional target borrower, and an optional satisfying document.

This ticket builds the **record + manual creation/transition helpers**. Generating
needs from findings (Phase 3), conditions (Phase 4.5), and templates (later) is
later-phase — the `origin` enum is ready for it.

Run straight through (sequential, no mid-ticket gates).

## What Was Built

### Model (`app/models/needs_item.py`)

Three enums + the reused `DocumentCategory` + the `NeedsItem` model (Base +
UUID/Timestamp/SoftDelete mixins).

| Column | Type | Notes |
| --- | --- | --- |
| `loan_file_id` | UUID FK → loan_files | required, indexed, `ondelete=CASCADE` (owned child) |
| `title` | `VARCHAR(256)` | required, short description |
| `description` | text | nullable |
| `category` | `DocumentCategory` enum | nullable, VARCHAR + CHECK (reused) |
| `needs_type` | `VARCHAR(64)` | nullable, **indexed**, flexible string (no CHECK) |
| `borrower_id` | UUID FK → borrowers | nullable, indexed, `ondelete=SET NULL` |
| `origin` | `NeedsItemOrigin` enum | default `MANUAL`, VARCHAR + CHECK |
| `priority` | `NeedsItemPriority` enum | default `STANDARD`, indexed, VARCHAR + CHECK |
| `status` | `NeedsItemStatus` enum | default `OUTSTANDING`, indexed, VARCHAR + CHECK |
| `satisfied_by_document_id` | UUID FK → documents | nullable, indexed, `ondelete=SET NULL` |
| `satisfied_at` / `requested_at` | timestamptz | nullable, timezone-aware |
| `notes` | text | nullable |

Relationships: `loan_file` (back_populates `needs_items`), `borrower` and
`satisfied_by_document` (one-directional, nullable). `LoanFile` gained
`needs_items` (cascade).

### Lifecycle & origin

- **Lifecycle:** `outstanding` → `requested` → `received`, or `waived`. Driven by
  the helpers so status + timestamps stay consistent.
- **Origin:** `manual` (wired now), `finding` / `condition` / `template` (later
  phases populate these; the enum is ready).
- **Priority:** `blocking` / `standard` / `low`.

### How it relates

- **documents** — `satisfied_by_document_id` (SET NULL): the document that
  fulfilled the need.
- **borrowers** — `borrower_id` (SET NULL): the target borrower (or null,
  file-level).
- **findings / conditions / templates** — via `origin` (generation is later-phase).

### Helpers (`app/services/needs_items.py`)

`create_needs_item(...)` → `OUTSTANDING` item; `request_needs_item(...)` →
`REQUESTED` + `requested_at`; `satisfy_needs_item(..., document_id)` → `RECEIVED`
+ `satisfied_by_document_id` + `satisfied_at`. All use `flush` (caller owns the
transaction).

### Migration (`4db01a03523e`)

Creates `needs_items` with the CASCADE FK to loan_files, SET NULL FKs to borrowers
and documents, **four** CHECK constraints (`category` reusing `documentcategory`,
plus `origin`/`priority`/`status`), and six indexes. Hand-cleaned to the repo
style.

## Decisions Made

- **ADR-067** — NeedsItem as the loan file's requirement checklist (first-class,
  lifecycle, origin, transition helpers).
- **ADR-068** — category reuses `DocumentCategory`; `needs_type` is a flexible
  string (mirrors ADR-053).
- **ADR-069** — document and borrower links use `SET NULL` (the durable item
  survives a removed document/borrower).

> **ADR numbering:** the ticket referenced ADR-069…071. The actual next free
> numbers were **067–069** (LP-18 used 064–066). Used 067–069; the type-string
> mirror points at the real ADR-053 and cascade/scoping at ADR-052.

## Assumptions

- **Generation from findings/conditions/templates is later-phase** — the `origin`
  enum supports all four; only `MANUAL` is created in V1.
- **Borrower communication is Phase 4** — `REQUESTED` + `requested_at` are
  recorded; actually sending the request is later.
- **Auto-matching a document to a need is Phase 2/4** — `satisfy_needs_item`
  links explicitly; matching is later.
- **`needs_type` is `VARCHAR(64)`** (a slug); `title` is `VARCHAR(256)`.

## Notes

- `category` **reuses** `DocumentCategory`, so the CHECK is named
  `ck_needs_items_documentcategory` (it shares the enum name) — intentional.
- After a satisfying document is hard-deleted, the item remains with a null
  `satisfied_by_document_id`; in V1 its `status` is left unchanged (re-opening a
  satisfied item is a later-phase concern).
