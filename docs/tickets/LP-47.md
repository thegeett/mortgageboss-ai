# LP-47 — Loading States Everywhere

- **Ticket:** LP-47 — loading states
- **Epic:** Epic 6 — Testing, Polish & Phase 1 Completion
- **Status:** Completed
- **Date:** 2026-06-12

## Summary

The sibling of LP-46 (error states): consistent, calm **loading** states everywhere data is
fetched or an action is in flight. Content loads show shape-matching skeletons (no blank-then-pop,
no layout shift); mutations show button spinners that disable (no double-submit); navigation shows
a route loading shell. Every async surface now resolves to exactly one clear state — **loading →
content | empty | error** — so the user is never looking at an ambiguous blank. Frontend only; no
backend; no new features. The document **processing** status (LP-43 polling) is left untouched.

## What Was Built

**Reusable loading primitives**

- `components/ui/skeleton.tsx` — `Skeleton` (base, now `aria-hidden`), `SkeletonText` (text-line
  blocks with optional per-line widths), `SkeletonRows` (row stacks; pass the real row height to
  shape-match). One set, reused instead of bespoke line stacks per surface.
- `components/ui/spinner.tsx` — `Spinner` (the one in-flight indicator, `aria-hidden`) and
  `LoadingRegion` (wraps a loading area with `aria-busy` + a visually-hidden status cue).

**Content skeletons (every data surface — shape-matched, no layout shift)**

- Dashboard **stat cards** — skeleton box now matches the number line exactly (`mt-1 h-9`,
  `w-14`), fixing the prior shift on multi-digit counts.
- Dashboard **file table** — per-column skeleton widths (ID short … address long) so columns don't
  resize when rows arrive; wrapped `aria-busy`.
- **File header** — title/subtitle skeleton (already matched); `aria-busy` added.
- **Overview cards** (borrowers/property/loan) — `CardSkeleton` → `SkeletonText` (4 rows);
  `aria-busy` on the card body.
- **Needs list** → `SkeletonRows` (`h-9`); **activity feed** → `SkeletonText`; both `aria-busy`.
- **Documents list** — `ListSkeleton` → `SkeletonRows` (`h-[58px]`, matching `DocumentRow`).
- **Document drawer** extraction — `SkeletonText`; `aria-busy`.

**Mutation/button loading + double-submit prevention (every mutation)**

- Standardized every in-flight button to the `Spinner` + `disabled` pattern: **login**, **create
  loan file**, **upload** (dropzone locked + button disabled), **type override**, **delete**
  (added spinner + "Removing…"), **download**, **logout** (added spinner), dev text-layer. Each is
  `disabled` while pending → cannot be double-fired.

**Route/navigation loading**

- `app/(protected)/dashboard/loading.tsx` and `app/(protected)/loan-files/[id]/loading.tsx` —
  shape-matching shells (title, stat cards, table / header, tabs, overview cards) shown during the
  navigation/code-split gap.

**Accessibility**

- Loading regions carry `aria-busy`; each has a visually-hidden `<output>` (role=status) cue
  ("Loading documents", "Loading your dashboard", …); the skeleton shapes are `aria-hidden`;
  disabled loading buttons convey state via their label ("Signing in…", "Uploading…", "Removing…").

## Skeletons vs Spinners (the rule)

- **Skeletons** for **content** loads (known shape): lists, cards, feeds, the drawer, route shells.
  They preserve layout and show structure.
- **Spinners** for **actions** in flight (mutations): the button disables + shows the spinner —
  signals work *and* prevents double-submit.
- **Route shell** for navigation: `loading.tsx` so a click reads as progress.

## No Layout Shift

Skeletons occupy the same box as the content they stand in for — dimensions were matched per
surface (stat-card number line, document rows `h-[58px]`, table per-column widths, header
title/subtitle, overview ~4 rows). Content arriving slots in without a jump.

## Double-Submit Prevention

Every mutation trigger is `disabled` while its mutation `isPending` (consuming TanStack Query's
mutation state), so a second click can't fire a duplicate upload/create/override/delete.

## Four-State Coordination

Every async surface resolves to exactly one state at a time:

| Surface | loading | content | empty | error (LP-46) |
|---|---|---|---|---|
| Dashboard table | skeleton rows | rows | "No loan files yet" / "No matching files" | error panel |
| Stat cards | number skeleton | the count | (n/a — 0 is valid) | "—" |
| Overview cards | `SkeletonText` | rows | "No … yet" | `CardError` + retry |
| Needs / activity | skeletons | list | "No outstanding needs" / "No activity yet" | `InlineErrorState` + retry |
| Documents list | row skeletons | grouped rows | "No documents yet" | `ErrorState` + retry |
| Drawer extraction | `SkeletonText` | extraction | "classified only" | `InlineErrorState` + retry |
| File workspace | header skeleton / route shell | header + tabs | — | `FileError` (404 / transient) |

Fast loads don't flicker (the natural query state + a brief 1-min `staleTime` cached client avoid
gratuitous skeleton flashes); we did not add artificial delays.

## Processing-Status Distinction (left as-is)

Two kinds of waiting: **data loading** (quick fetches — this ticket's skeletons/spinners) and
**document processing** (the seconds-long pipeline, status-driven polling — LP-43). They're
complementary and were kept separate: a document card shows a **load skeleton** (LP-47) while the
list loads, then a **processing** status badge (LP-43) if that doc is still processing. LP-43's
polling/status UI was not touched.

## Verification Performed

- **Component tests** confirm: skeleton-while-loading → content (DocumentList); double-submit
  prevention (dropzone disabled while pending); `aria-busy` + status cue on loading regions;
  primitive counts/decorative roles. `vitest` **91 passed**.
- **Build** compiles all surfaces + the two route `loading.tsx` shells.
- **No layout shift**: verified by matching skeleton dimensions to content per surface (stat-card
  number, document rows, table columns, header) — the skeleton occupies the content's box.
- **Four-state coordination**: each surface audited for loading/content/empty/error (table above);
  errors coordinate with LP-46 (a failed load → error state + retry, never an endless skeleton).
- `pnpm lint`, `pnpm tsc`, `pnpm build` all pass.
- *Note:* a live throttled-browser pass wasn't run in this headless environment; the loading→content
  behavior, `aria-busy`/disabled states, and no-shift dimensions are covered by the component tests
  + build, and the shape-matching was done by aligning skeleton boxes to the real content.

## Decisions Made

- **ADR-156** — Loading states: skeletons for content, spinners for actions, coordinated four
  states, consuming TanStack Query.

## Assumptions

- Frontend only; consume TanStack Query's `isPending` (no new machinery).
- The document **processing** status (LP-43) is a separate, longer, status-driven wait — left as-is.
- Coordinate with LP-46 error states (shared four-state model); don't redo errors.
- Mechanisms (primitives) over per-surface bespoke; calm, no artificial delays.

## Notes

- The primitives (`Skeleton`/`SkeletonText`/`SkeletonRows`, `Spinner`/`LoadingRegion`) are reusable
  — new surfaces get consistent loading for free.
