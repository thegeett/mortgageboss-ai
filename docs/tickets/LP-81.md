# LP-81 — Minimal Verification Tab UI (the Arc A demo surface)

- **Ticket:** LP-81 — compose the verification slice into one coherent, demonstrable screen
- **Epic:** Phase 3 — Verification Engine & Calculators (Arc A capstone)
- **Status:** Completed
- **Date:** 2026-06-29

## Summary

The capstone of Arc A: the Verification tab that **surfaces** everything built across LP-74…80.5 into the one
screen the domain expert (Priya) actually uses. It composes — the **DTI/LTV calculators prominent**, the
**interactive findings list** (severity / type / confidence / source location + resolution), the **aggression
dial** filtering it, the **cross-source trigger + staleness**, and the **needs-completeness indicator** — and
completes the **re-run stability** that keeps findings consistent across runs. It is **minimal by design**
(NOT the full Wireframe 5). After this the slice is whole + demonstrable end-to-end. See **ADR-200/201/202**.

## What Was Built

- **Backend (the genuine gap):** the per-finding resolution API — `POST …/findings/{id}/apply`,
  `…/override` (required reason), `…/note` — wrapping the existing LP-75 services (+ a new `add_finding_note`);
  the GET broadened to surface deterministic + cross-source findings uniformly; `resolution_note` exposed on
  `FindingPublic`. Each resolution returns the re-filtered status (the calculators refresh in one round-trip).
- **Frontend (the bulk):** the composed tab (calculators prominent + the cross-source panel); the interactive
  **FindingCard** (templated headline, AI description secondary, expandable **source location** = page +
  verbatim snippet, Apply / Override-with-reason / Add note, notes); the **FindingsList** (open dial-filtered +
  a **Resolved** history group); the **NeedsCompleteness** indicator; the templated-wording helper.

## The Composition

`verification/page.tsx` places the **DTI + LTV calculators** at the top (the headline "replace ChatGPT" win,
lender-specific limits via LP-80), then the **VerificationPanel** — the run trigger + staleness, the
needs-completeness guard, the dial, and the findings list. One coherent, scannable screen.

## The Findings List + Resolution

Each finding shows **severity** (red/yellow), the **type/label**, the **confidence**, and the **source
location** (click → the document page + verbatim snippet — the trust mechanism). Open findings carry the core
actions: **Apply** (incorporate into the structured data → the recompute interlock fires; DTI/LTV refresh),
**Override** (with a required, recorded reason), **Add note** (informational). Deterministic + cross-source
findings are handled uniformly (the origin distinguishes provenance).

## The Dial Filtering The List

The aggression dial (LP-79) filters the **open** findings by confidence (Conservative/Balanced/Thorough),
gating display + blocking; moving it re-filters **instantly** (no AI re-run) with the legible consequence.
Resolved findings are not dial-filtered — they always show in the Resolved group.

## The Cross-Source Trigger + Staleness

"Run verification" runs the pass; LP-78.1's cache returns existing findings on **unchanged** inputs (no AI
re-run) and re-runs on **changed**. The staleness banner ("The file changed — out of date, re-run") fires on
**document AND baseline edits** (LP-80.5). Progress shows while the pass runs (it's an AI call).

## The Needs-Completeness Indicator

The outstanding-needs count + a non-blocking message ("verification compares documents against the stated
application; N outstanding document needs — results may be incomplete until they're collected"). A sparse
result on an under-documented file is **understood, not mistaken for clean**. An **indicator, not a gate** —
the processor can still run verification (needs-first → verification-second is the ordering).

## The Re-Run Stability

- **Stable identity** — a cross-source finding's identity is its canonical type (`cross_source.<type>`).
- **Merge-not-replace** — a re-run supersedes only the prior pass's **open** findings; **resolved** ones
  (APPLIED/OVERRIDDEN) survive and stay visible in the Resolved group — a finding is never silently dropped,
  and resolutions are never lost.
- **Templated wording** — known types get a deterministic headline (reads identically every run); the AI's
  free-form wording is secondary detail; novel ("other") + deterministic findings keep their own message.

## Honest Minimality

The **CORE** that demonstrates the slice — NOT the full Wireframe 5. Deferred to **LP-88**: the stats row,
filter pills, version selector, and the full per-finding action set (Request docs / Accept risk). The deeper
"promote reliable checks to deterministic" consistency is **LP-86**; the overlay admin UI is **LP-87**.

## Verification Performed

- **Composition** — the tab renders the calculators + the cross-source panel (dial + trigger + staleness +
  needs indicator + findings list).
- **Findings list** — severity/type/confidence + source location (click → page + snippet); resolution actions.
- **Resolution** — Apply (→ incorporated, recompute), Override (requires reason: empty → 422, blank → 400),
  Add note (annotates, does not resolve); tenant-scoped (cross-company / unknown finding → 404).
- **Dial filtering** — low-confidence open findings hidden at Balanced, revealed at Thorough.
- **Re-run stability** — known-type wording is identical across differing AI descriptions; resolved findings
  survive a re-run (existing `test_rerun_preserves_resolved_findings`) and show in the Resolved group.
- **Needs indicator** — shows the count + message when outstanding; hidden when none / loading.
- **No regression** — backend **1232 passed**; frontend **220 passed**; `ruff`/`mypy --strict` clean;
  `pnpm lint`/`typecheck`/`build` pass. No migration.
