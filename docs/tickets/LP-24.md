# LP-24 — Auth Dependencies and Route Protection

- **Ticket:** LP-24 — Auth dependencies, role-based access, tenant context, `GET /auth/me`
- **Epic:** Epic 3 — Authentication (third ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Built the route-protection machinery that gates every protected endpoint from Epic 4
onward: the `get_current_user` dependency (Bearer extraction → token verify →
**live-user lookup** → active check), an optional non-raising variant, the
`require_role` authorization factory (403 vs 401), the request **tenant context**
(`current_user.company_id`), and `GET /auth/me` as the end-to-end proof.

Implemented as **per-route FastAPI dependencies, not global middleware** — see the
note below on the ticket title.

## What Was Built

### `app/api/dependencies.py`

- `get_current_user(db, credentials) -> User` — `HTTPBearer(auto_error=False)` →
  `verify_token(..., ACCESS)` → `get_user_by_id` → active check. Uniform `401`
  (`WWW-Authenticate: Bearer`) on any failure.
- `CurrentUser = Annotated[User, Depends(get_current_user)]` — the alias routes use.
- `get_current_user_optional` / `OptionalCurrentUser` — returns `User | None`, never
  raises; an invalid token is treated as no token (still runs the live/active checks).
- `require_role(*allowed_roles) -> Callable[[User], Awaitable[User]]` — depends on
  `get_current_user`, raises `403` if the live role isn't allowed.
- `get_current_company_id` / `CurrentCompanyId` — the non-forgeable tenant scope.

### `GET /auth/me` (`app/api/auth.py`)

Depends on `CurrentUser`, returns `UserPublic.model_validate(current_user)` — the
first protected endpoint, proving the full chain. No `hashed_password`.

## The Protection Model

- **Dependencies, not middleware** — a route is protected by declaring `CurrentUser`
  or a `require_role(...)`; public routes opt out by not declaring them. No global
  auth middleware (ADR-080).
- **Live lookup** — `role`/`company_id`/`is_active` are read from the current DB row,
  never the token (which carries only `sub`). Deactivation/role-change/delete take
  effect on the next request (ADR-081).
- **401 vs 403** — `401` = unauthenticated (no/invalid token); `403` = authenticated
  but wrong role. `require_role` builds on `get_current_user`, so authn precedes
  authz (ADR-083).

## Security Properties

- **Deactivation cutoff** — a valid token whose user is set `is_active=False` is
  rejected `401` on the next request (the V1 stateless-JWT cutoff; CRITICAL test).
- **No token-trust** — authorization data is never read from the token; a stale token
  cannot assert outdated role/company.
- **Non-forgeable tenant context** — `current_user.company_id` derives from the
  validated token + live user, so a caller can't scope to another company. This
  activates Epic 2 multi-tenancy at runtime (queries use `scope_to_company`).
- **Wrong-type rejection** — a refresh token presented as a Bearer access token → 401.

## Decisions Made

- **ADR-080** — Auth via per-route dependencies, not global middleware.
- **ADR-081** — Live-user lookup on every authenticated request (deactivation cutoff).
- **ADR-082** — Tenant context derives from the authenticated user.
- **ADR-083** — Role-based authorization via `require_role` (403 vs 401).

> **ADR numbering:** LP-23 used 076–079; the next free numbers were **080–083**.

## Assumptions

- **Business endpoints are Epic 4** — `GET /auth/me` is the only new endpoint here;
  it's where the tenant-scoping pattern gets exercised broadly.
- **Frontend is LP-25** — backend only.
- **No granular permissions** — role-level gating only (PROCESSOR/ADMIN); no
  per-resource ACLs in V1.
- **No revocation store** — `is_active` + live lookup is the V1 cutoff (ADR-075/081).
- **`get_user_by_id` does not filter soft-deletes** — adequate here (the deleted-user
  test hard-deletes the row); if soft-delete-aware auth is wanted later, filter there.

## Verification Performed

- **`get_current_user` / `GET /auth/me`** (9 tests): valid token → 200 + correct user
  (no `hashed_password`); no header → 401 (+ `WWW-Authenticate: Bearer`); malformed
  header (`Basic ...`, bare `Bearer`) → 401; refresh-as-access → 401; expired → 401;
  tampered/garbage → 401; **deactivated user → 401**; deleted user → 401; unknown
  subject → 401.
- **`require_role`** (4 tests): ADMIN passes admin-only; PROCESSOR → **403 (not 401)**;
  no token → 401; `require_role(ADMIN, PROCESSOR)` admits a PROCESSOR.
- **Tenant context** (1 test): `get_current_company_id` returns the user's `company_id`.
- Role/tenant routes are defined **in the test module** (a throwaway test app that also
  mounts the real auth router), never in `app/`.
- **Full suite:** `uv run pytest` → **227 passed** (213 → +14). `ruff check`,
  `ruff format --check`, `mypy app/` (60 files), and `pre-commit run --all-files` clean.

## Notes

- **Ticket title says "middleware"** but auth is implemented as **per-route
  dependencies** (ADR-080) — cleaner than global middleware that must exempt public
  routes, and explicit about which routes are protected. The phase plan line
  ("LP-24: Auth dependencies and middleware") is satisfied by the dependency approach;
  global request-logging/request-ID middleware remains a Phase 7 concern.
- The deactivation test relies on the dependency re-reading the user each request; with
  the rollback session the flushed `is_active=False` is visible to the next call.
