# LP-90.1 — Fix the dead source tooltip + add the source label/tooltip to the PROPERTY VALUE row + correct the "Stated" sublabel

- **Ticket:** LP-90.1 (follow-up to LP-90)
- **Epic:** Phase 3 — Verification Engine & Calculators
- **Status:** Completed
- **Date:** 2026-06-30

## Summary

A small UI/labeling follow-up to LP-90. A screenshot review of the LTV Calculator found the appraised-value
source transparency was only half-wired. Three fixes, frontend-only (plus tests/docs):

1. **The dead tooltip** in the "Value basis" row now works (a real, accessible tooltip).
2. The **editable PROPERTY VALUE "Appraised value" row** gets the same source label + working tooltip.
3. That row's **"Stated" sublabel** is corrected to its real source (valuation), which was a factual mislabel.

No change to the LTV computation, the lesser-of logic, the "Appraised value" main label, or the field bindings
(LP-90/LP-77 set those). The valuation/estimated model-collapse + the main-label naming remain flagged for Priya.
See the **ADR-215 addendum** in `decisions.md`.

## Diagnosis (read-only first)

- **The "tooltip" was a native `title` attribute.** In `ltv-calculator.tsx` → `ValueBasisCallout`, the literal
  logic lived only in `title="appraised = valuation_amount or estimated_value"` on the wrapper `<div>`. The "(?)"
  glyph was `aria-hidden` with **no handler** — purely decorative. There was **no tooltip primitive** in the repo
  (no `components/ui/tooltip.tsx`, no `@radix-ui/react-tooltip`). So the "(?)" rendered but did nothing; the native
  `title` is unreliable (~1.5s delay, never fires on touch/tap, unstyled) and held only the bare logic string — the
  plain explanation the LP-90 AC wanted was never present.
- **The PROPERTY VALUE "Appraised value" row** renders via `BreakdownSection` → `LineRow` over `data.value_items`
  (`item.key === "ltv.appraised_value"`). `LineRow` had no access to `appraised_value_source` and rendered no
  source label/tooltip.
- **The "Stated" sublabel** comes from `humanize(item.source)`; the backend (`ltv.py:135`) sets the appraised
  auto-line `source = "stated" if appraised is not None else "manual"`. "Stated" is wrong — the value is sourced
  from `valuation_amount`/`estimated_value`, not borrower-stated.

## What changed (frontend only)

- **`components/ui/tooltip.tsx` (new)** — the shadcn/Radix Tooltip primitive (`@radix-ui/react-tooltip`,
  added to `package.json`), matching the existing dialog/dropdown component style.
- **`components/file/ltv/ltv-calculator.tsx`:**
  - `LtvBody` is wrapped in a `TooltipProvider`.
  - New `AppraisedSourceTooltip` — the "(?)" is now a focusable, accessible `<button>` trigger whose tooltip
    content is the literal logic `appraised = valuation_amount or estimated_value` **plus** the plain explanation
    ("uses the property valuation amount; if absent, falls back to the estimated value; no appraisal document is on
    file yet").
  - `ValueBasisCallout` uses `AppraisedSourceTooltip` (the dead native `title`/`aria-hidden` glyph is gone).
  - `BreakdownSection` accepts `appraisedValueSource` and threads it to `LineRow`. The appraised row
    (`ltv.appraised_value`) now renders the `from valuation amount` / `from estimated value` source label + the
    same tooltip, and its sublabel is the real provenance instead of "Stated" (falling back to the humanized source
    — "Manual" — only when neither value is present). All other rows are unchanged ("Purchase price" stays
    "Stated"; it *is* stated from `SalesContractAmount`).

No backend change.
