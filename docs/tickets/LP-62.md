# LP-62 — Tier 1 Property Documents

- **Ticket:** LP-62 — Tier 1 property extractors
- **Epic:** Phase 2 — Document Handling (Tier 1 extractors)
- **Status:** Completed
- **Date:** 2026-06-18

## Summary

The third Tier-1 **extractor** batch: the property cluster — **purchase
agreement**, **homeowner's insurance**, **mortgage statement**, **property tax
bill**, and **HOA statement**. Same established pattern as LP-60/61. The cluster
spans both subject-property facts (LTV + housing expense) and other-property
obligations (DTI). Backend only.

## What Was Built

Five extractors under `app/ai/extraction/`, their prompts, and registry entries:

- `purchase_agreement.py` + prompt → `extract_purchase_agreement`
- `homeowners_insurance.py` + prompt → `extract_homeowners_insurance`
- `mortgage_statement.py` + prompt → `extract_mortgage_statement`
- `property_tax_bill.py` + prompt → `extract_property_tax_bill`
- `hoa_statement.py` + prompt → `extract_hoa_statement`
- Registered all five in `EXTRACTORS` (`app/ai/extraction/__init__.py`).

## The Established Pattern (followed, not reinvented)

Each follows **LP-39a**, using the existing extractors as templates: a typed core
of `TypedField`s (coerced `value` + `SourceLocation`) + an `additional_sections`
grouped catch-all; a `<Type>ExtractionResult` with `data`/`status`/`confidence`/
`reasoning`/tokens + `.failed()`; the shared tolerant parser; Decimals for money,
dates as `date`; honest nulls; metadata-only logging.

## The Typed Cores (sensible starters — refine with Priya)

- **Purchase agreement** — buyer/seller, `property_address`, `sales_price` (LTV
  basis), `closing_date`, `earnest_money_amount`. Contingencies → catch-all.
- **Homeowner's insurance** — carrier, policy number, `property_address`,
  `coverage_amount`, `annual_premium` (housing expense), effective/expiration.
- **Mortgage statement** — lender, **`property_address`**, `monthly_payment`
  (DTI), `unpaid_balance`, `escrow_amount`, `due_date`.
- **Property tax bill** — **`property_address`**, `assessed_value`,
  `annual_tax_amount` (housing/DTI), `due_dates` (string — often two
  installments), `taxing_authority`.
- **HOA statement** — association, **`property_address`**, `dues_amount` +
  `dues_frequency` (obligation), `balance`, `due_date`.

## Two Property Contexts

- **Subject-property facts → LTV + housing expense:** the purchase agreement
  (`sales_price`) and the insurance binder (`coverage_amount` + `annual_premium`).
- **Other-property obligations → DTI:** the mortgage statement (`monthly_payment`),
  the tax bill (`annual_tax_amount`), and the HOA statement (`dues_amount`) — for
  properties the borrower already owns.

The cluster spans both; one extractor set serves both purposes.

## Subject-vs-Other (the address is captured; Phase 3 matches)

A mortgage statement / tax bill / HOA statement may be for the subject property OR
another property the borrower owns. Each captures `property_address` in the typed
core, and the prompts are explicit that the model must **not** decide which
property it is — **Phase 3 matches the address** to the subject. This ticket
captures; it does not decide.

## Shaped for Phase 3 LTV/DTI Verification

These typed cores are the values Phase 3 verifies, cross-checked against the
**stated** MISMO property + liabilities (Phase 1.5 — e.g. the file's
`SalesContractAmount` and its mortgage liabilities). The catch-all + the AI
cross-source layer have the rest of each document.

## The Appraisal Note (flagged for the user)

The **appraisal**'s appraised value also feeds LTV, which might argue for Tier-1
extraction. But the catalog currently classifies `appraisal` as **Tier 2**
(recognized, not extracted). This ticket **honors the catalog and does not extract
it** — flagged here as a candidate for Tier-1 promotion in a future catalog
refinement with Priya. Noted, not acted on.

## Decisions Made

- **ADR-171** — five Tier-1 property extractors following the established pattern;
  the cluster spans subject-property facts (LTV/housing expense) and other-property
  obligations (DTI); the property address is captured but subject-vs-other is left
  to Phase 3; `due_dates` kept as a string; typed cores are V1 starters refined
  with Priya; the appraisal-is-Tier-2 promotion flagged; accuracy honestly scoped.

## Assumptions

- Field sets are sensible starters **refined with Priya**.
- **Accuracy** against real documents is validated as real documents flow through
  — **no sample property documents were available** when this was built.
- **Phase 3** does LTV/DTI verification and the subject-vs-other address matching.
- **LP-63/64** add the remaining Tier-1 clusters (borrower-info, tax returns).

## Verification Performed

- **No sample documents** present (`/mnt/user-data/uploads/` absent), so the tests
  verify the **mechanism/shape**, not per-document accuracy (stated honestly).
- **Each extractor** — returns the typed-core + catch-all shape; per-field source
  locations; types coerced (Decimals/dates); honest nulls; all-null core →
  `FAILED`; unparseable → `None`; `extract_*` handles success / AI failure /
  unsupported / empty gracefully.
- **Address captured** — the mortgage statement / tax bill / HOA extractors
  capture `property_address` (tested) for Phase 3 subject-vs-other matching.
- **Registry routing** — each of the five now routes to its extractor (Tier 1,
  category Property), no longer the classified-only fallback; an extraction
  persisted.
- **No regression** — the existing extractors (pay_stub/w2/bank_statement +
  LP-60/61) unchanged and their suites pass.
- **Suite** — `uv run pytest` green (**872 passed**, +55 over LP-61); `ruff` +
  `ruff format` clean; `mypy` clean (127 source files).
