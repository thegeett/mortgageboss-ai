LTV ratios, with the LTV-specific subtleties done correctly (the **lesser-of** basis, the **HELOC

limit). Full-stack. See **ADR-191** and

## The Three Ratios

- **LTV = first loan ÷ the LESSER OF (purchase price, appraised value)** — for a purchase, the lower
  of the two (the lender won't lend against a price above the appraisal). The basis (and which value
  won) is made **visible**. Tested both directions (price < appraised and appraised < price).
- **CLTV = (first + second + HELOC drawn balance) ÷ value** — combined liens.
- **HCLTV = (first + second + HELOC CREDIT LIMIT) ÷ value** — the most conservative measure: a
  HELOC at $0 balance with a $40k line still counts the full $40k (it could be drawn tomorrow), so
  HCLTV rises above CLTV. Tested with a $0-balance / nonzero-limit HELOC.

The lesser-of and credit-limit subtleties are the trust mechanism — exactly what a processor wants
verified deterministically, and what ChatGPT fumbles.

## Refinance-Aware

The loan purpose drives the **denominator** and the **limit**:

- **Purchase** → LTV uses the lesser-of (price, appraised); limit = purchase max.
- **Rate/term refinance** → appraised value (no purchase price); limit = purchase max.
- **Cash-out refinance** → appraised value + a **stricter** limit (cash-out pulls equity out).

A nullable `refinance_type` (rate_term / cash_out) on the loan file carries the cash-out distinction.

## Reuses The DTI Calculator Model

Not reinvented — the same patterns, applied to LTV: the transparent itemized breakdown + explicit
formulas; auto-population from the structured data; per-field override with an audit log
(`ltv_overridden`, the prior value recorded); real-time recalc (the override endpoints return the
recomputed result); the findings coupling (the unresolved-findings alert + recompute-consumer); pure
deterministic math. A parallel `LtvOverride` table mirrors `DtiOverride` (LP-76 untouched). The UI
matches the DTI calculator's style and quality.

## The Appraised Value

LTV needs the appraised value, which isn't a Tier-1 extraction yet. So it **auto-populates from the
MISMO valuation** (else the estimated value) and is **manual/override-able** where neither is present
— the override is the graceful fallback. The appraisal is **not** promoted to Tier 1 here.

## Verification Performed

- **The math** — LTV lesser-of (both directions); CLTV combined; HCLTV uses the credit limit not the
  balance (HELOC $0 balance + $40k line → HCLTV > CLTV); refi denominators (rate-term + cash-out use
  appraised); missing basis → None; half-up rounding.
- **Auto-population** — opens filled; the appraised value graceful when absent (manual/override-able).
- **Override** — recomputes, persists, clears to auto, audited with the prior value; HELOC-limit
  override drives HCLTV end-to-end; unknown field rejected.
- **Effective limit** — purchase 97 vs cash-out 80 (purpose-varying); program default; pass/over.
- **Findings coupling** — the unresolved alert on an open in-scope finding; recompute on a
  structured-data change (a corrected appraised value → LTV recomputes).
- **Endpoints** — GET returns the three ratios + breakdown; PUT/DELETE recompute; cross-company 404.
- **Frontend** — the three ratios, the lesser-of basis, the limit, and the formulas render; the
  refinance purpose + unresolved alert; over-limit in red; inline override triggers the mutation;
  loading/error.
- **No regression** — backend **1147 passed**; frontend **165 passed**; `ruff`/`mypy --strict` clean;
  `pnpm lint`/`typecheck`/`build` pass; the migration applies + reverses.
