# LP-65 — Tier 2: Recognized Documents (classify + categorize + summary)

- **Ticket:** LP-65 — Tier 2 shared summary path
- **Epic:** Phase 2 — Document Handling (Tier 2/3 handlers)
- **Status:** Completed
- **Date:** 2026-06-19

## Summary

Builds the **Tier 2 handling path** — the **one shared mechanism** for the ~60-80
*recognized* document types that get classified, categorized, and **summarized**
but NOT field-extracted. LP-58 stubbed this path; LP-65 fills the stub with the
real implementation: a single lightweight AI summary for any Tier 2 document.
Primarily backend (the summary path) + a minimal frontend touch (the summary is
visible). The full tier-aware detail view is LP-72.

## What Was Built

- **The shared Tier 2 path** — `_tier2_summarize` in
  `app/tasks/document_processing.py` (fills the LP-58 `_tier2_summarize_stub`).
- **The summarizer** — `app/ai/summarization.py` (`summarize_document`, a cheap
  Haiku call) + `prompts/summarization/document_summary.txt`.
- **Summary storage** — a nullable `summary` TEXT column on `documents`
  (migration `b344317498a5`, up/down) + exposed on `DocumentResponse`.
- **Minimal display** — the summary shows as a subtle line in the document list
  and a "Summary" block in the existing document drawer.

## One Shared Path (the tier-model efficiency)

Every Tier 2 document goes through the **same** path — **no per-type logic** (no
`flood_certification.py` / `credit_report.py`). This is the efficiency of the tier
model: **~18 extractors + 1 Tier-2 summary path + 1 Tier-3 analyzer**, not ~80
extractors. The document arrives already classified + categorized (LP-59); the
shared path adds a summary and finalizes. Tests run several different Tier 2 types
(`credit_report`, `flood_certification`, `closing_disclosure`,
`verification_of_deposit`) through the same path to prove there is no branching.

## The Summary (a gist for human reference, NOT extraction)

- A **1-2 sentence** human-readable gist — "what is this document, briefly?" (what
  it is + a key identifying detail). E.g. *"Flood zone determination for 60 North
  Street — Zone X, minimal risk."*
- **The Tier 1 contrast:** Tier 1 extracts precise structured values that *drive
  decisions* (income/asset/property figures, with source locations); Tier 2 just
  *summarizes for reference* — no structured fields, no source locations.
- **Cheap:** the Haiku-class (classification) model, capped at 256 tokens. Low
  cost-per-document is the point of Tier 2. A response cap guards a rambling answer
  without failing it.
- **Forgiving / low-stakes:** a slightly-off gist is fine (human reference, not a
  calculation) — accuracy is proportionately light and refine-able.

## Storage + Minimal Display

- A nullable `summary` TEXT column (`documents.summary`) holds the gist; the
  migration applies + reverses cleanly. `DocumentResponse.summary` exposes it.
- **Minimal frontend:** the summary appears as a subtle gray line under a
  document's row in the Documents tab (when present) and as a "Summary" block in
  the existing document drawer. The **full tier-aware detail view** (Tier 1 fields
  / Tier 2 summary / Tier 3 findings) is **LP-72** — this ticket only makes the
  summary visible.

## Tier 2 Docs Are Normal / Package-Eligible

A Tier 2 document is a first-class file document — it appears in the Documents tab
under its category with its summary, and is part of the file. Not second-class.
(Package groundwork is LP-72; package assembly is Phase 6 — not built here; Tier 2
docs are simply not treated as lesser.)

## Graceful Handling

`summarize_document` never raises and returns `None` on any failure (AI error,
empty/unsupported document, empty output). A failed summary still **finalizes** the
document (recognized + categorized, `summary` null) — never stuck, never a crash
(the resilience discipline). The summary text is **never logged** (it can quote
document PII) — only a length / presence flag.

## Decisions Made

- **ADR-174** — the Tier 2 shared summary path: one mechanism for ~60-80 recognized
  types (no per-type logic); a lightweight cheap Haiku gist, NOT extraction; graceful
  on failure; Tier 2 docs are normal package-eligible documents; the summary stored +
  minimally visible (full detail in LP-72); the low-stakes / forgiving stance.

## Assumptions

- The summary is **low-stakes / forgiving** and refine-able; best confirmed against
  real documents over time, but the stakes are low.
- The **full tier-aware detail view** + package groundwork is **LP-72**; package
  assembly is **Phase 6**.
- **LP-66** fills the Tier 3 stub (generic analyzer + findings).

## Verification Performed

- **Routing** — a Tier 2 doc (e.g. `credit_report`) routes to the shared path, gets
  a summary, and reaches a terminal status with no deep extraction.
- **One shared path** — four different Tier 2 types go through the same path (no
  per-type branching), all summarized + terminal.
- **Graceful** — summarization failure (`None`) → terminal status, `summary` null
  (not stuck / crash).
- **Stored + brief** — the summary is stored, retrievable, and asserted short (not
  a giant extraction blob).
- **Summarizer contract** — reads the document natively, uses the cheap
  (classification) model, returns a trimmed gist, `None` on empty/unsupported/AI
  failure/empty output, caps a rambling response, and never logs the document bytes
  or the summary text (PII).
- **Migration** — applies + reverses (column added/dropped); verified against dev
  Postgres.
- **No regression** — Tier 1 extraction (LP-60..64 + the original 3) unchanged; the
  Tier 3 stub (LP-58) still routes (LP-66 fills it).
- **Suite** — backend `uv run pytest` green (**938 passed**, +15 over LP-64);
  `ruff` + `ruff format` clean; `mypy` clean (131 source files). Frontend `pnpm
  lint` + `typecheck` + `test` (113 passed) + `build` all green.
