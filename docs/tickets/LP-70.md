# LP-70 — Needs-List Frontend (the processor's dashboard)

- **Ticket:** LP-70 — Needs-list frontend (the differentiator's face)
- **Epic:** Phase 2 — Document Handling → the needs list
- **Status:** Completed
- **Date:** 2026-06-19

## Summary

The **face of the differentiator** — the first major Phase-2 UI. A
**self-maintaining checklist** on the loan-file overview that surfaces the needs
ENGINE (LP-68) and the AI REASONING (LP-69): open the file → a tailored checklist
appears (the MISMO floor + the AI reasoning produce it; LP-70 displays it). The
processor sees at a glance what's outstanding, what's satisfied, what needs
attention — and **why each need is there** — with the disposition controls to
confirm/adjust/dismiss/waive/add. Full-stack, frontend-heavy. See **ADR-179**.

## What Was Built

**Backend (surfacing LP-68/69 — tenant-scoped, audited):**

- **Read** (`GET /loan-files/{id}/needs`) — now returns the dashboard's fields:
  `description`, `reasoning` (the "why"), `reason` (rejected/waived), `disposition`,
  `origin`, the satisfying document's `filename`, `satisfied_at`. Eager-loads the
  satisfying document; no raw PII.
- **Writes** (the disposition flow): `POST …/{need}/confirm`, `PATCH …/{need}`
  (adjust), `POST …/{need}/dismiss`, `POST …/{need}/waive`, and `POST …/needs`
  (add). Each updates the need, **captures the correction signal** (the disposition
  on the need — LP-69's improve-from-corrections), and is **audited**.
- New service helpers: `get_needs_item` (scoped, eager-loaded), `adjust_needs_item`;
  reuses LP-69's `record_need_correction` (confirm/dismiss) + `waive_need`.
- Four new `activity_type` values (`needs_item_confirmed` / `_adjusted` /
  `_dismissed` / `_waived`) + a CHECK-widening migration (`a7f4c2e1b9d0`). Add
  reuses `needs_item_created`.

**Frontend (the dashboard — frontend-design skill):**

- `components/file/needs/needs-dashboard.tsx` — the orchestrator (header + summary +
  the subtle updating cue + the Add control; grouped sections; loading/empty/error).
- `need-card.tsx` — one need: the visual state, the reasoning "why", the source tag,
  the satisfying document, the reason; a proposed need gets a left accent.
- `need-actions.tsx` — the disposition controls (Confirm + an overflow menu →
  Adjust / Waive / Dismiss, each with a dialog).
- `add-need-dialog.tsx` — add a need the AI missed.
- `lib/api/needs.ts` — the read hook (live polling) + the five disposition
  mutations; `lib/loan-files/needs.ts` — the visual language (state/group/source
  meta + grouping); `lib/types/needs-item.ts` — the current model; a `Textarea` ui
  primitive. The old provisional `NeedsSection` (LP-34) is replaced.

## The Self-Maintaining Checklist (the core experience)

Open the file → the tailored checklist appears. The system tracks "what am I still
waiting on?" instead of the processor holding it in their head. Needs are grouped
**action-first** so the chase pile is unmistakable, the reasoning is one glance
away, and the disposition controls are right there.

## The AI Reasoning Surfaced (the "why" — trust-making)

Every need shows its LP-69 reasoning in an inset note ("Needs tax returns because
the borrower has self-employment income…"). This is **explainability made visible** —
what makes the AI proposals trustworthy/evaluable rather than a mysterious checklist
item. It's the distinctive element vs. a dumb checklist, and the screen's signature.

## The Disposition Flow (the human disposes; feeds the correction signal)

The AI proposes (LP-69), the processor disposes. **Confirm** (proposed → confirmed),
**Adjust** (edit — confirms), **Dismiss** (with a reason — set aside), **Waive**
(with a reason — not required), **Add** (a CONFIRMED manual need the AI missed).
Every action calls an audited write and feeds LP-69's correction-capture. The
processor controls; the AI did the heavy lifting.

## The Visual States (action-oriented)

The five LP-68 states made visual and grouped: **Needs action** (pending / requested
/ rejected) is distinct from **In review** (received), **Complete** (verified), and
**Set aside** (waived). The processor sees "what to do next" at a glance.

## Live Updates + The Subtle Updating State

The dashboard reads the (already-polling) documents query to know when any document
is in-flight and feeds that to the needs query as a `live` flag — so the list polls
while a document processes and settles once done (a satisfied need visibly moves
Pending → Received → Verified, no manual refresh; a backstop stops a stalled poll).
While the list settles, a **subtle "Updating…" cue** shows the OUTCOME — deliberately
**NOT** a queue-depth / "engine running" / "N files queued" meter. The per-file
serialization is a fast internal mechanism, not a user-facing batch job, so it stays
invisible (the prior decision).

## The APIs (tenant-scoped + audited)

All routes nest under the loan file (the LP-29 file gate → `404` cross-company); a
per-need action additionally `404`s a need not in the path file. The needs response
carries only the need's own fields (no raw borrower PII). Each disposition is
activity-logged with the acting user.
