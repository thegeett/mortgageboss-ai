# LP-96 — AI-generated "why it matters" + "suggested fix" (generated-once, stored, grounded, warned, validate-with-Priya)

- **Ticket:** LP-96 (fifth ticket of the finding-presentation epic, LP-92..98)
- **Epic:** Phase 3.5 — Finding presentation
- **Status:** Completed
- **Date:** 2026-07-01

## Summary

Findings now carry an AI-authored **why it matters** (the consequence) + **suggested fix** (the
remediation), filling LP-95's slots. This is the **one deliberate, guard-railed relaxation** of the
project's "AI never touches authoritative output" principle — the AI adds *explanation*; the
deterministic core (what + source + citation) and the processor's judgment still decide. See
**ADR-221**.

## The guardrails (all implemented)

1. **Generated once, stored, NEVER per-run.** Guidance is keyed **per canonical finding type**
   (`income_variance`, `employer_mismatch`, … — the key the `xsrc.*` rules and the AI findings
   share) in a grounded-starter store, and resolved by a **plain dict lookup at read time**
   (`FindingPublic.from_model` merges it into `details`). Novel findings generate once at discovery
   (stored on the finding); LP-94's reconcile keeps the row on re-run, so it is never regenerated.
   **No model call on render / re-run** — identical text every time (no flicker, no per-view cost).
2. **Grounded in the rule's facts.** `generate_guidance` (reusing the app's `complete()` at
   temperature 0.0) is given the type + category + description + threshold and asked to EXPLAIN them
   — not to invent facts or cite ungiven regulations.
3. **Grounded-starter, validate-with-Priya** — `starter=True`; not authoritative.
4. **Warned** — a clear-but-calm "AI-generated — verify before relying on this; it may be wrong."
5. **Visually distinct** — a tinted + bordered + iconned (`Sparkles`, amber) block, set apart from
   the deterministic core.
6. **Overrideable** — the Override action still wins; the guidance is advisory.

## What changed

**Backend**
- `app/verification/finding_guidance.py` (new): `RuleGuidance`; the grounded-starter `GUIDANCE_BY_TYPE`
  (per canonical type) + a per-category fallback; `guidance_for` / `resolve_guidance` (deterministic
  read-time resolution, prefers finding-stored guidance for novel findings); and the AI-authoring
  mechanism `generate_guidance` (grounded prompt via `build_guidance_prompt`, `complete()` at temp
  0.0, defensive parse, graceful `None` on failure).
- `app/schemas/verification.py`: `FindingPublic.from_model` merges the resolved guidance into
  `details` (no model call) so LP-95's slots render it; absent → the card degrades gracefully.
- `app/services/cross_source.py`: novel ("other" / unmapped) findings get their guidance generated
  **once at discovery** via an injectable `guidance_fn` (best-effort; stored on the finding). Known
  types resolve from the store at read.
- `app/scripts/generate_finding_guidance.py` (new): the one-time, **idempotent** generation pass —
  per canonical type, grounded in the type's facts, writes a review JSON (never auto-applied; Priya
  validates). Not per-request.

**Frontend**
- `finding-card.tsx`: LP-95's why/fix slots are now one **visually distinct** amber `Sparkles` block
  carrying the **"AI-generated — verify"** warning + the **grounded-starter** marker. Renders only
  when populated (LP-95 graceful degradation intact); the deterministic What-we-found + Source are
  unchanged.

## Honesty note

Given no API key in this environment, the committed `GUIDANCE_BY_TYPE` is the **grounded-starter**
content (authored deterministically from each type's meaning, marked `starter`). The AI-authoring
*mechanism* (`generate_guidance` + the script) produces the richer, lender-specific prose when run
with a key — reviewed + validated by Priya before it lands, the same grounded-starter → validated
posture as the rule content.

  no warning / slots when absent (graceful); the finding is still overrideable.
