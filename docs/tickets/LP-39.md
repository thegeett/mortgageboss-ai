# LP-39 — Document Extraction Module (Pay Stub, Phase 1)

- **Ticket:** LP-39 — Pay stub extraction module (typed PayStubExtraction, prompt-as-file, honest nulls, graceful failure)
- **Epic:** Epic 5 — Document Upload & Processing
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Built the document **extraction** module, scoped to the **pay stub only** for Phase 1.
Where classification (LP-38) answered "what kind of document is this?", `extract_pay_stub(text)`
answers "what does it say?" — reading the structured values out of a pay stub into a typed,
mostly-nullable schema. One type end-to-end establishes the per-type pattern (schema + prompt
+ module) that Phase 2 replicates. Backend only; tests mock the wrapper (no real calls, no key).

## What Was Built

- **`PayStubExtraction`** (`app/ai/extraction/pay_stub.py`) — a typed Pydantic schema, **V1
  starter marked "refine with Priya"**, all fields nullable: `employer_name`,
  `employee_name`, `pay_period_start/end`, `pay_date`, `gross_pay`, `net_pay`, `ytd_gross`,
  `pay_frequency`, `hours`, `rate` (money as `Decimal`, dates as `date`).
- **`PayStubExtractionResult`** — `data`, `status` (reuses LP-16 `ExtractionStatus`),
  `confidence`, `reasoning`, plus the `.failed(reason)` factory (all-null data, `FAILED`, 0.0).
- **Prompt file** — `app/ai/prompts/extraction/pay_stub.txt`, a clearly marked **STARTER**
  that instructs JSON output, **null-for-missing**, and **never guess**.
- **`extract_pay_stub(text)`** — empty-text short-circuit, prompt load, LP-37 `complete` with
  the **Sonnet-class** model, defensive+tolerant parse, graceful failure, metadata-only logging.
- **`_parse_pay_stub_json` + tolerant coercers** (`_coerce_decimal` / `_coerce_date` /
  `_coerce_str`).
- **Shared parsing helpers** — factored `extract_json_object` + `coerce_confidence` into
  `app/ai/parsing.py`; **classification (LP-38) was refactored to use them** (no behavior
  change), so both type-parsers share one defensive primitive.

## The Contract

`extract_pay_stub(text: str) -> PayStubExtractionResult` — always returns a typed result,
never raises. Text is the **user** message; the file-based prompt is the **system** prompt.
This module **returns** a result; persisting/versioning the `Extraction` record is LP-42's
job. The pipeline treats `FAILED` / low-confidence / many-nulls as `NEEDS_REVIEW`.

## Typed, Not a Field Bag

`PayStubExtraction` is document-specific and typed — the deliberate departure from the POC's
generic `ExtractedField` bag (LP-16, ADR-057). Typed values are what make the data
**verifiable** downstream (Phase 3 checks `gross_pay` / `pay_period_end` as `Decimal` /
`date`). It serializes to JSON for `Extraction.extracted_data`.

## Pay-Stub-Only + the Per-Type Pattern

Phase 1 builds **one** type end-to-end to prove the pipeline shape (upload → text → classify
→ extract) and establish the reusable pattern (schema + prompt + module + the shared
`app/ai/parsing.py` helpers). Phase 2 replicates it for the other ~100 types.

## Honest Nulls / No Hallucination / Reads, Doesn't Judge

- Missing/illegible values are `None`, **never fabricated** — the prompt forbids guessing. A
  hallucinated income figure could falsely pass verification; a missing one just routes to
  review.
- Extraction **reads** faithfully (including absences); it does **not** verify, compute, or
  judge plausibility — that's Phase 3.
- **Tolerant coercion:** a single uncoercible field drops to `None` and marks the run
  `PARTIAL` (data loss), rather than failing the whole extraction. Status is derived: no
  fields read → `FAILED`; a coercion loss → `PARTIAL`; else `SUCCEEDED`.

## Reuses LP-38 Patterns

`load_prompt` (file-based prompt), the shared defensive JSON parser (`app/ai/parsing.py`),
graceful failure, and metadata-only logging — plus the Sonnet-class
`settings.anthropic_model_extraction` (vs classification's cheaper Haiku-class model),
`max_tokens=1024`.

## Privacy

The document text, the raw model response, and the extracted **values** (income, employer,
names) are **never** logged. Only metadata — `paystub_extraction_done` carries `status`,
`confidence`, and `fields_present` (a count) — no values. A test seeds PII into the text and
the extracted values and asserts none of it appears in captured logs.

## Decisions Made

- **ADR-123** — typed document-specific extraction, not a generic field bag.
- **ADR-124** — pay stub only for Phase 1; the per-type pattern is the deliverable.
- **ADR-125** — honest nulls, no hallucination; extraction reads, it does not judge.

> **ADR numbering:** LP-38 used 120–122; the next free numbers were **123–125**.

## Assumptions

- **Classification** is LP-38 (this assumes the doc is already a pay stub); **PDF text
  extraction** is LP-40 (takes text); the **pipeline that calls this and persists/versions
  the Extraction** is LP-42; **verify/compute income** is Phase 3.
- The **prompt is a starter** to replace with the POC pay stub prompt; the **field set is a
  starter** to refine with Priya.
- Reused the existing `anthropic_model_extraction` setting (the spec's `ai_extraction_model`).
- Tests **mock** the wrapper — no real calls, no key (CI safe).

## Verification Performed

`uv run pytest` → **424 passed** (27 new), and `tests/ai/` passes **without a real key**.
ruff check + format, `mypy app/` (strict), pre-commit all clean.

- **Success** — full JSON → correctly typed `Decimal`/`date` values; uses the extraction
  model; text as the user message, prompt as `system`.
- **Fenced/preambled JSON** — still parses; **nested `data`/`fields` key** accepted.
- **Honest nulls** — absent fields are `None`, status `SUCCEEDED` (absent ≠ loss); an

- **Coercion** — `"$4,200.00"`/`"4,200.50"`/numbers → `Decimal`; ISO/US/"June 15, 2024"

  **empty/short text** → `failed` **without** calling the API (`call_count == 0`).
- **Confidence clamp** — `1.7 → 1.0`.
- **Privacy** — PII text + extracted values (`SECRETCORP`, `9999.99`, SSN) absent from logs;
  the metadata log carries only `status` / `confidence` / `fields_present`.

# LP-39 (Modification) — Extract From the Full Document (not text)

- **Date:** 2026-06-11
- **Status:** Completed
- **ADR:** ADR-128 (revises ADR-123/124/125; follows the LP-37 revision / ADR-126 and LP-38 / ADR-127)

## What Changed

Following the full-document AI decision (ADR-126) and the matching classification change
(ADR-127), extraction now reads the **full document** (PDF/image bytes) natively with the
Sonnet-class model — no OCR, no pre-extracted text — using the LP-37 document/image content
block. Pay stub remains the only type in Phase 1.

- **Signature:** `extract_pay_stub(text: str)` → **`extract_pay_stub(content: bytes,
  media_type: str)`** (async).
- **Call construction:** the loaded prompt stays the `system` instruction; the document is
  sent as the `user` message via `build_document_message(content=..., media_type=...)`
  (LP-37 revision). The model is unchanged (`settings.anthropic_model_extraction`,
  Sonnet-class), `max_tokens=1024`.
- **Short-circuit:** an **empty** (`len 0`) document or an **unsupported** `media_type`
  (outside `application/pdf` / `image/jpeg` / `image/png` / `image/jpg`) returns
  `PayStubExtractionResult.failed("empty or unsupported document")` **without an API call**
  (plus a defensive `try/except ValueError` around `build_document_message`).

## What Stayed (no regression)

`PayStubExtraction` + `PayStubExtractionResult` + `.failed(...)`; **honest nulls / no
hallucination** (missing → `None`, never fabricated); **tolerant coercion** (currency →
`Decimal`, date strings → `date`; a single bad field → `None` marking `PARTIAL`, not a
whole-extraction failure); the **defensive JSON parser** (`_parse_pay_stub_json`, unchanged);
the **graceful-failure contract** (`AIClientError` → `failed("AI call failed")`, unparseable →
`failed("could not parse extraction")`, **never raises**); the file-based **starter prompt**;
the **Sonnet model**; and **metadata-only logging**. Extraction still **reads, doesn't judge**
(Phase 3 verifies/computes).

## Privacy (reinforced — strict)

The document **bytes**, their **base64**, the message content, the raw response, **and the
extracted values** (income, employer, names) are **never** logged — only `status`,
`confidence`, and `fields_present` (a count). The privacy test seeds PII into the **bytes**
and asserts neither the raw byte content, the base64 payload, nor the extracted values
(`SECRETCORP`, `9999.99`) appear in captured logs.

## Supported Media Types

`application/pdf`, `image/jpeg`, `image/png` (`image/jpg` normalized). Matches the LP-36
upload allowlist and the LP-37 document-block support.

## Verification

`uv run pytest` → **443 passed**, `tests/ai/` passes **without a real key** (wrapper mocked).
ruff check + format, `mypy app/` (strict), pre-commit all clean. The `extract_pay_stub` tests
were adapted to bytes + media type (the parser-level `_parse_pay_stub_json` tests are
unchanged):

- **Success (PDF)** — typed `Decimal`/`date` values; asserts `complete` was called with a
  **document block** whose base64 round-trips to the original bytes.
- **Image input (PNG)** — uses an **image block** (`media_type` `image/png`).
- **Malformed → failed**; **AIClientError → failed** (not propagated; never raises).
- **Empty document → failed without calling**; **unsupported media type
  (`text/plain` / `application/zip` / `image/gif` / `""`) → failed without calling**.
- **Honest nulls, currency/date coercion, junk-field → None + PARTIAL, confidence clamp** —
  all still verified by the unchanged parser tests.
- **Privacy** — raw byte content + base64 + extracted values absent from logs; the metadata
  `paystub_extraction_done` log carries only `status` / `confidence` / `fields_present`.

## What's Next

- The AI features (classification + extraction) now both read full documents; the **LP-42
  pipeline** wires upload → classify → extract → persist (and supplies the bytes from
  storage). The schema + prompt remain starters (Priya / POC).
