# LP-20 — Communication and ActivityLog Models

- **Ticket:** LP-20 — Communication and ActivityLog models with event-record foundations
- **Epic:** Epic 2 — Database & Models (twelfth ticket)
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Created **two** related-but-distinct event-record models:

- **`Communication`** — one message in/out of a loan file (borrower document
  requests, lender condition responses, inbound borrower replies). Direction,
  channel, status, sender/recipient/subject/body, an optional needs-item link, and
  provenance. The data foundation for the Phase 4 communication module.
- **`ActivityLog`** — one recorded event on a file (file created, status changed,
  document uploaded, finding resolved, …). The audit trail / timeline, with an
  optional actor (null = system), a summary, and JSON detail. Append-only in
  spirit.

This ticket builds the **records** and minimal create helpers. Email sending and
inbound routing (Phase 4) and pervasive activity instrumentation (incremental) are
later.

Run straight through (sequential, no mid-ticket gates).

Creates both tables: CASCADE FK to loan_files on each; `communications` SET NULL

`loan_file_id` + `direction`/`status`/`needs_item_id` (communications) and
`loan_file_id` + `activity_type` (activity_logs). Hand-cleaned to the repo style.

## Communication vs ActivityLog (the distinction)

A **communication** is a *message* — it has sender/recipient/subject/body and a
send lifecycle. An **activity-log** entry is *any event* — including non-message
ones (status changed, document uploaded, verification run) that have no
sender/recipient. They overlap at exactly one point: a sent communication may also
produce a `COMMUNICATION_SENT` activity entry. One combined table would be
mostly-null and semantically muddy, so they are two tables (**ADR-070**).

## Decisions Made

- **ADR-070** — Communication and ActivityLog as separate models.
- **ADR-071** — ActivityLog is append-only in spirit; `log_activity` is the
  standard helper; pervasive instrumentation is incremental.
- **ADR-072** — `CommunicationChannel` is EMAIL-only in V1.

> **ADR numbering:** the ticket referenced ADR-072…074. The actual next free
> numbers were **070–072** (LP-19 used 067–069). Used 070–072; cascade/scoping
> point at the real ADR-052.

## Assumptions

- **Email sending and inbound routing are Phase 4** — `create_communication` only
  persists the record.
- **Pervasive activity instrumentation is incremental** — `log_activity` exists;
  wiring it into every operation happens as operations are built (ADR-071).
- **EMAIL is the only channel in V1** (ADR-072).
- **`sender`/`recipient`/`subject`/`external_message_id` are `VARCHAR(256)`**;
  `summary` is `VARCHAR(256)`.
