# LP-67 — Implications Engine (surface findings → suggest needs)

- **Ticket:** LP-67 — Implications engine
- **Epic:** Phase 2 — Document Handling → the needs bridge
- **Status:** Completed
- **Date:** 2026-06-19

## Summary

The **implications engine** — the first **consumer** of findings (LP-66). It reads
findings ("this court order mentions a $500/mo obligation") and **suggests needs**
("→ consider a need for payment history"). It is the bridge from findings (what
documents say) to the needs list (what the file still requires). Backend only.

## What Was Built

- **`SuggestedNeed`** (`app/services/implications.py`) — the intermediate: a
  `need_description`, a provisional `need_type`, the `reasoning` (the why), and the
  traceable links `source_finding_id` + `source_document_id`. Produced **on-demand**
  (a pure projection over the persisted findings — no table, no migration).
- **The implications engine** — `suggest_needs_for_finding(finding)` (pure) +
  `suggest_needs_for_loan_file(db, loan_file_id)` (read-only): findings → suggested
  needs with reasoning.

## Surface + Suggest, Not Act (the locked constraint)

The engine produces `SuggestedNeed`s the processor disposes of — it **never**
mutates the financial picture (no silent debt-adding, no DTI change), **never
persists anything**, and **never creates a needs-list item**. Acting on findings is
Phase 3 (human-confirmed); disposing of suggestions is the LP-68/70 needs flow. The
functions are pure (`suggest_needs_for_finding`) or read-only
(`suggest_needs_for_loan_file` does a single `SELECT`). **A test asserts** running
the engine creates no `NeedsItem` and mutates no finding — the critical constraint.

## The Findings-Scoped Mapping

A bounded, **deterministic** (no AI) mapping of each `DocumentFindingType` → a
sensible suggested need:

| Finding type | Suggested need (`need_type`) |
| ------------ | ---------------------------- |
| `obligation` | Payment history / obligation documentation (`obligation_documentation`) |
| `income_related` | VOE / income explanation (`income_verification`) |
| `property_interest` | Property documentation review (`property_documentation`) |
| `discrepancy_candidate` | Review / explanation (`discrepancy_review`) — Phase 3 does the cross-check |
| `other` | **none** (a sensible "none", not a noisy generic) |

The heavy holistic reasoning is LP-69; LP-67 is the focused "this finding implies
this need" mapper. (A small AI call to phrase suggestions could be added later, but
the core is a bounded mapping.)

## Explainable + Traceable

Every `SuggestedNeed` carries `reasoning` — the human-readable *why* (e.g. *"Because
document X asserts a $500.00/monthly child support obligation, the file should
document this recurring obligation (e.g. request payment history)."*) — plus
`source_finding_id` + `source_document_id`, the machine-traceable chain
**suggestion → finding → document**. Trustworthy, not mysterious.

## The Seam to LP-68/69

`SuggestedNeed` is a clean **intermediate** produced on-demand. **LP-68** (the needs
engine) and **LP-69** (the AI needs reasoning) ingest these suggestions as ONE input
source and decide how/whether each becomes a real needs-list item. LP-67 does not
build the needs model — it produces suggestions ready for ingestion. (Producing them
on-demand, rather than persisting, avoids a premature schema LP-68 would reshape and
avoids staleness — suggestions are recomputed from the live findings.)

## Findings-Scoped vs File-Scoped (the LP-69 line)

- **LP-67 is findings-scoped:** one finding → its implied need(s). Bounded,
  composable.
- **LP-69 is file-scoped:** the whole file (stated data + documents + findings + the
  LP-67 suggestions) → the complete needs list (holistic AI reasoning).

LP-67 **feeds** LP-69; it does **not** duplicate the holistic reasoning.

## Decisions Made

- **ADR-176** — the implications engine: findings → suggested needs with explainable
  reasoning (a deterministic findings-scoped mapping); surface + suggest, **not act**
  (no financial mutation / persistence / needs-item creation); traceable
  (suggestion → finding → document); an on-demand intermediate feeding LP-68/69;
  findings-scoped, not file-scoped (the LP-69 line).

## Assumptions

- The **needs model / engine** is **LP-68**; the **holistic AI needs reasoning** is
  **LP-69** — both consume these suggestions.
- **Phase 3** acts on findings (cross-source, human-confirmed).
- The mapping refines as the needs work + Priya input land.

## Verification Performed

- **The mapping** — each finding type → a sensible suggested need (with the right
  `need_type`); `other` → no suggestion.
- **Surface-not-act** (the critical assertion) — running the engine creates **no
  `NeedsItem`** and mutates **no finding** / financial data; it returns suggestions
  only.
- **Explainable + traceable** — each suggestion's `reasoning` starts with "Because",
  names the source document, surfaces the obligation amount, and carries
  `source_finding_id` + `source_document_id` (matching the finding).
- **End-to-end** — a divorce decree (LP-63) → an obligation finding (LP-66) → a
  suggested need (LP-67), traceable back to the finding.
- **Tenant scoping** — suggestions for a loan file derive only from that file's own
  findings (another file's findings don't appear).
- **No regression** — LP-66 findings + the extractors unchanged.
- **Suite** — `uv run pytest` green (**972 passed**, +9 over LP-66); `ruff` +
  `ruff format` clean; `mypy` clean (137 source files).

## Notes

The mapping is deterministic and refine-able as the needs work (LP-68/69) and
Priya's input land. Suggestions are produced on-demand from live findings — no
persistence, no staleness.

## What's Next

serialization, ingesting LP-67's suggestions as one input source.
