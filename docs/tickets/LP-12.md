# LP-12 — Lender Model

- **Ticket:** LP-12 — Lender model (per-company slug, overlay/program JSON fields)
- **Epic:** Epic 2 — Database & Models (fourth ticket)
- **Status:** Completed
- **Date:** 2026-06-10

## Summary

Created the `Lender` model — the institutions loan files are submitted to (UWM,
Sun-West). A lender **belongs to a company**, making this the first application
of the LP-11 multi-tenancy pattern to a new table. Introduced **per-company
unique slugs** (composite uniqueness — the multi-tenant default, contrasting
with User's globally-unique email), forward-looking JSON fields
(`lender_overlays`, `supported_programs`), contact fields for direct underwriter
communication, and the shared `LoanProgram` enum that loan files will reuse in
LP-13.

Implemented in two reviewed stages: (1) `Lender` + `LoanProgram` + migration,
(2) tests + docs.

`LoanProgram` (`CONVENTIONAL`, `FHA`) is defined in `app/models/lender.py`,

relationship (no destructive cascade).

### Migration

`..._create_lenders.py` — `lenders` table; FK `fk_lenders_company_id_companies`
(`ondelete=RESTRICT`); composite unique `uq_lenders_company_id_slug`; index
`ix_lenders_company_id`; `pk_lenders`. Verified `upgrade head` → `downgrade -1`
→ `upgrade head` clean; table inspected in psql.

## Patterns Applied

- **Multi-tenancy via `company_id`** — lender belongs to a company; queries
  scoped with `scope_to_company()` (LP-11). The tenant-isolation test confirms
  one company never sees another's lenders.
- **Per-company unique slug** — composite unique on `(company_id, slug)`, named
  explicitly because the `uq` naming-convention template only uses the first
  column. Two companies can each have slug `"uwm"`; one company cannot have it
  twice (ADR-045).
- **Forward-looking JSON fields** — `lender_overlays` (`{}` now, structured in
  Phase 3) and `supported_programs` (JSON list of program values). Creating the
  columns now avoids a later migration (ADR-046).

## Decisions Made

- **ADR-045** — Per-company unique slugs (composite uniqueness).
- **ADR-046** — Lender overlays and supported programs as JSON.
- **ADR-047** — `LoanProgram` enum (Conventional, FHA) shared across models.

## Assumptions

- **UWM and Sun-West are the intended initial lenders**, seeded later in LP-48 —
  this ticket uses only test data, no real seed.
- The **overlay structure is a Phase 3** decision; `lender_overlays` stays an
  empty `{}` with no schema enforcement for now.
- `supported_programs` values are validated against `LoanProgram` in application
  code, not by the database (it is a JSON list, ADR-046).
- V1 program scope is Conventional + FHA; Jumbo/VA/USDA are deferred to V2.

## Verification Performed

- `uv run pytest` → **36 passed** (28 prior + 8 new).
- **Per-company uniqueness** (`test_slug_unique_per_company`): slug `"uwm"` in
  company A succeeds, the same slug in company B succeeds, a second `"uwm"` in
  company A raises `IntegrityError` (contained in a `begin_nested()` SAVEPOINT so
  it doesn't poison the rolled-back test transaction).
- **Tenant isolation** (`test_scope_to_company_isolates_lenders`): a query scoped
  to company A returns only A's lenders; every returned row's `company_id`
  matches; B is isolated.
- **JSON**: `supported_programs` round-trips `[CONVENTIONAL, FHA]` → stored as
  `["conventional","fha"]`; `lender_overlays` defaults to `{}` and stores
  structured JSON; both defaults verified.
- **Soft delete / relationship**: `only_active` excludes a soft-deleted lender;
  `Company.lenders` loads via `selectinload`.
- `uv run ruff check`, `ruff format --check`, `uv run mypy app/` — all pass.

## Notes

- Optional contact columns use explicit `String(length) + nullable=True` rather
  than `Mapped[MediumStr | None]` — composing `Optional` with LP-10's `Annotated`
  mapped-column types is fragile in SQLAlchemy 2.x. Lengths still follow the
  conventions (email 256, phone 64, URL/notes 1024).
- `supported_programs` is stored as a JSON list, **not** an enum-typed column, so
  there is no DB CHECK on its contents (ADR-046) — distinct from `User.role`,
  which is a real enum column with `ck_users_userrole` (ADR-037).
- Contact fields target **direct underwriter communication** (per discovery),
  not an account-executive intermediary.
