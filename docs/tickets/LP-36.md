# LP-36 — Document Upload Endpoint

- **Ticket:** LP-36 — Document upload endpoint (multipart, validated, storage-backed) + list/get/download/delete
- **Epic:** Epic 5 — Document Upload & Processing
- **Status:** Completed
- **Date:** 2026-06-11

## Summary

Built the endpoints where documents **enter** the system: a multipart upload (one or many
files) that validates each file, stores the bytes via the LP-35 storage layer, and records
`PENDING` `Document` rows — plus list, get-one (+ current extraction), an auth'd byte
download, and soft-delete. This is the first real consumer of the storage abstraction. The
frontend upload UI is LP-43; classification/extraction and the async pipeline are LP-38/39/42.

## What Was Built

- **Schemas** (`app/schemas/document.py`) — `DocumentResponse` (metadata, **no
  `storage_path`**), `ExtractionPublic`, and `DocumentDetailResponse` (= response +
  `current_extraction | None`).
- **Validation** (`app/services/documents.py`) — `MAX_FILE_SIZE_BYTES` (50 MB),
  `ALLOWED_CONTENT_TYPES`, magic-byte table, `DocumentValidationError` (carries the HTTP
  status), and `validate_upload(content, declared_content_type) -> str`.
- **Service functions** — `create_document` (PENDING), `list_documents`,
  `get_document_for_company` (the flat-route gate), `soft_delete_document`,
  `get_current_extraction`.
- **Endpoints** (`app/api/documents.py`) — a **nested** router (`upload`, `list`) and a
  **flat** router (`retrieve`, `download`, `delete`), both wired into `main.py` under
  `/api/v1`.

## URL Shape & Scoping

| Method | Path | Scoping |
| --- | --- | --- |
| POST | `/loan-files/{file_identifier}/documents` | nested — `ScopedLoanFile` file gate |
| GET | `/loan-files/{file_identifier}/documents` | nested — `ScopedLoanFile` file gate |
| GET | `/documents/{document_id}` | flat — `get_document_for_company` |
| GET | `/documents/{document_id}/download` | flat — `get_document_for_company` |
| DELETE | `/documents/{document_id}` | flat — `get_document_for_company` |

- **Nested** (upload/list): the parent file is resolved with the caller's company **first**
  (LP-29 gate) → `404` if not theirs.
- **Flat** (get/download/delete): a document has **no `company_id`**, so
  `get_document_for_company` **joins `Document → LoanFile`** and filters on the file's
  company (`only_active` on both). It returns `None` — turned into `404` — for a missing,
  soft-deleted, or **out-of-company** document. A flat route **never** loads a document by
  id alone. `company_id` always comes from `current_user`.

## Validation (size / type / magic bytes)

- **Size** — 50 MB cap; the upload is read in 1 MB chunks and aborted the moment it exceeds
  the cap, so an oversized file is never fully buffered → **`413`**.
- **Type** — `application/pdf` / `image/jpeg` / `image/png` only, by **content-type
  allowlist AND magic-byte signature**; the detected type must equal the declared one, so
  a `.txt` (or a PNG mislabelled `application/pdf`) is rejected → **`415`**.
- **Batch policy** — all files are validated **before any are stored**; one invalid file
  rejects the whole request and persists nothing (all-or-nothing).
- Pairs with the LP-35 extension sanitization (defense in depth).

## Storage Integration (LP-35)

Each valid file gets a server-generated `document_id` (the same UUID used for the storage
path and the `Document` row). `get_storage_backend().save(company_id, file_id, document_id,
filename, content)` returns the tenant-prefixed `storage_path`, persisted on the row. The
`storage_path` is **internal** — never in a response; bytes are returned **only** via the
auth'd `/download` route (`Response` with the stored `mime_type` and a sanitized
`Content-Disposition: attachment`). `get_url` is `None` for local.

## Decisions Made

- **ADR-114** — URL shape (nested upload/list, flat get/download/delete) + flat-route
  document→file→company scoping.
- **ADR-115** — upload validation (50 MB, PDF/JPEG/PNG via content-type + magic bytes);
  bytes served only via the auth'd download.
- **ADR-116** — soft-delete preserves stored bytes; documents start `PENDING`.

> **ADR numbering:** LP-35 used 112–113; the next free numbers were **114–116**.

## Assumptions

- **Frontend upload UI** is LP-43; **classification/extraction** are LP-38/39; the **async
  pipeline** (which advances `PENDING`) is LP-42; **PDF text extraction** is LP-40;
  **document type override** is LP-44.
- **No virus scanning / deep content validation** in V1 (basic size/type checks here;
  scanning is later hardening).
- `DOCUMENT_UPLOADED` already existed on `ActivityType` (added with LP-20) — one activity
  per upload request, with a `document_count` detail.
- A whole-request rejection on any invalid file (simplest correct V1) rather than
  per-file partial success.

## Verification Performed

`uv run pytest` → **354 passed** (29 new); ruff check + format, `mypy app/` (strict), and
pre-commit all clean.

- **Upload** — a valid PDF (and PNG/JPEG) → `201`, status `pending`, correct metadata,
  bytes verified via `storage.read`, **`storage_path` absent** from the response; **multiple
  files** in one request all created.
- **Validation** — a disallowed type (`text/plain`) → `415`; **content-type spoofing**
  (PNG bytes declared `application/pdf`) → `415`; **oversize** (lowered cap) → `413`; one
  invalid file in a batch → `415` and **nothing persisted**.
- **List / get / download / delete** — list returns the file's active docs; get returns the
  detail with `current_extraction: null`; download returns the **exact bytes** with the
  right content-type and `attachment` disposition; delete → `204`, subsequent get `404`,
  and the **stored bytes are preserved**.
- **CRITICAL cross-tenant isolation** — a Company A user **cannot** get/download/delete a
  Company B document by id (`404` each, and A's document stays intact), nor upload to/list a
  Company B file (`404`); list never leaks another company's documents.
- **Auth** — unauthenticated upload/list → `401`.

## Notes

- The download `Content-Disposition` is built defensively: an ASCII-sanitized `filename=`
  (quotes/control chars stripped) plus an RFC 5987 `filename*=UTF-8''…` percent-encoded
  name, so a user-controlled filename can't break out of the header.
- Tests point the storage backend at a `tmp_path` root (via a `settings` monkeypatch +
  `get_storage_backend.cache_clear()`), so they never touch the real `./storage` dir.
- Switched the over-limit status to the non-deprecated `HTTP_413_CONTENT_TOO_LARGE`.
