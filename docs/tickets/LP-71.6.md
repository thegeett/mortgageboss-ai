# LP-71.6 — Add universal needs to the deterministic floor (borrower ID, per-borrower)

- **Ticket:** LP-71.6 — Universal floor needs (borrower ID)
- **Epic:** Phase 2 — Document Handling → the needs list
- **Status:** Completed
- **Date:** 2026-06-24

## Summary

A real MISMO import produced a needs list with **no borrower identification** (driver's
license / government ID) — a near-universal requirement on every loan file (lenders
verify identity per Patriot Act / KYC). This adds **universal needs** to the
deterministic floor, starting with a borrower **Government ID**, seeded **per
borrower**. See **ADR-182**.

## What Was Built

`seed_floor_needs` (the deterministic floor) now seeds, before its conditional rules,
a clearly separated **universal-needs** section:

- A **Government ID** (`needs_type=drivers_license`, category `BORROWER_INFO`,
  `origin=FLOOR`, `disposition=CONFIRMED`) for **each borrower** on the file.
- Two extensible lists drive it: `_PER_BORROWER_UNIVERSAL` (one per borrower — the ID)
  and `_PER_FILE_UNIVERSAL` (one per file — empty for now). Adding another always-required
  need is a one-line append.
- A new `_active_borrowers` query (ordered by `borrower_position`) reads the borrowers
  safely in the async import path (not via the lazy `loan_file.borrowers`).

## Why the Floor, Not the AI

Needs come from two sources: the deterministic **floor** (always-true) and LP-69's **AI
reasoning** (situation-specific). An ID is **universal** — required on every file
regardless of the borrower's situation — so it is the *opposite* of distinctive. The AI
surfaces what's *special* about a file (self-employment → tax returns; a gift → a gift
letter), so it **under-proposes** an obvious always-true need like an ID. The right home
for always-true needs is the floor: deterministic, and it fires immediately on import.
**The floor being "thin" must not mean missing its universal baseline.**

## Per-Borrower

Each borrower needs their own ID, so the rule iterates the file's borrowers: a
single-borrower file → 1 ID need; a 2-borrower (co-borrower) file → 2 distinct ID needs,
each identifying its borrower (title `"Government ID — {name}"` + a `borrower_id` link;
a name-less borrower falls back to `"Borrower {position}"`).

## Extensible + Refine-With-Priya

The universal-needs section is separated and commented so Priya's future additions drop
in cleanly. The ID is the first/clearest universal need and is safe to add now; the full
list (e.g. a credit authorization, certain initial disclosures) **refines with Priya**.

## Deterministic

The ID fires from the floor on import — immediately, independent of the AI/worker — like
the other floor rules. It reads the borrowers (visible post-flush, per LP-71.5). No
duplicate IDs: the floor's existing "already seeded" guard covers the universal needs too.

## Decisions Made

ADR-182 (the floor covers universal needs; universal → floor, situation-specific → AI;
per-borrower; extensible/refine-with-Priya).

## Assumptions

- The full universal-needs list refines with Priya; the ID is clearly universal and safe
  to add now.
- LP-63's `drivers_license` extractor handles the uploaded ID (this ticket is about the
  **need**, not the extraction).
- Manually-created files get template needs via the LP-30 setup path; the MISMO floor
  (and thus the universal ID rule) applies to imports.

## Verification Performed

- Single-borrower → 1 ID need from the floor (+ the existing conditional needs).
- Multi-borrower → an ID need **per borrower** (2 → 2 distinct, each identifying its
  borrower, each `borrower_id`-linked).
- Deterministic — the ID is present from seeding alone, no AI/worker.
- No duplicates on re-seed (idempotent).
- The import proving test (LP-71.5) updated: single-borrower import now yields the 4
  conditional needs + 1 ID = 5.
- No regression: the conditional floor rules, LP-69's reasoning, the needs engine, and the
  dashboard are unchanged. Backend `1017 passed`, mypy + ruff clean.
