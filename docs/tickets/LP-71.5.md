# LP-71.5 — Fix the floor's flush-timing bug + surface AI-needs pending/failure

- **Ticket:** LP-71.5 — Floor flush-timing fix + AI-needs visibility (a fix ticket)
- **Epic:** Phase 2 — Document Handling → the needs list
- **Status:** Completed
- **Date:** 2026-06-24

## Summary

A read-only diagnostic found a real MISMO import (employment income + self-employment
+ a gift + assets, Conventional Purchase) producing a needs list with **only
"Purchase agreement"** instead of the expected rich list. Two independent defects;
this ticket fixes both **code** defects. (A third issue — no Celery worker running —
is operational, out of scope.) See **ADR-180**.

## The two defects (diagnostic evidence)

**Defect 1 — the floor's stated-data rules were dead-on-arrival on import.** The
session runs `autoflush=False` (`app/core/database.py:32`). In
`create_loan_file_from_mismo` the `StatedIncomeItem` / `StatedAsset` rows were
`db.add`-ed but **not flushed** before `seed_floor_needs`. The floor's
`_has_stated_employment_income` / `_has_stated_assets` run SELECTs that, with
autoflush off, **couldn't see the pending rows** → the employment (→ pay stubs +
W-2) and asset (→ bank statements) rules returned False. Only the purchase rule fired
(it reads `loan_file.loan_purpose` in memory). Proof: imported files had the
income/assets committed, yet only `purchase_agreement` seeded; the DB had **zero**
`ai_reasoning` needs.

**Defect 2 — the import silently "promised" AI needs.** LP-69's reasoning runs as an
async Celery task; with no worker (or on an AI failure that `propose_needs` swallows
to `[]`), the floor-only list was presented with **no signal** that more needs were
pending or that reasoning failed — a safety gap (a processor may not chase documents
they need).

## The fixes

**Fix 1 (the bug) — flush before the floor seeds.** `seed_floor_needs` now
`await db.flush()`es **first** (inside the function, so every caller is protected),
making a caller's just-added stated rows visible to its SELECTs regardless of the
session's autoflush setting. The floor's **rules are unchanged** — they were correct;
they just couldn't see the data. The deterministic floor now fires the
employment/asset needs on import **independent of the AI/worker**.

**Fix 2 (visibility, minimal) — `ai_needs_status` on the loan file.** A nullable
column (`pending` / `completed` / `failed`; NULL = not triggered):
- The MISMO import sets **PENDING** (reasoning enqueued).
- The task entrypoint (`apply_ai_needs_for_file_id`) flips it to **COMPLETED** on a
  successful run.
- The swallowed `AIClientError` in `propose_needs` records **FAILED** (no longer
  silent).
- The needs dashboard surfaces it: "AI is still reviewing — more needs may appear"
  (pending) / "AI review didn't finish — this list may be incomplete" (failed). A
  settled/complete list shows no note.

It is **informational, never blocking** — the import and the floor succeed
regardless.

## Out of scope (operational)

The Celery worker not running is fixed by starting it
(`docker compose --profile worker up -d worker`), not by code. This ticket ensures
(a) the floor works without the worker, and (b) the worker's absence/failure is
visible, not silent.

## Verification Performed

- **Fix 1 proving test** (`tests/mismo/test_import_floor_and_ai_status.py`): a MISMO
  import with employment income + assets produces `{pay_stub, w2, bank_statement,
  purchase_agreement}` (not just `purchase_agreement`); the purchase rule still
  fires; no duplicate needs.
- **Fix 2 tests:** the import marks `PENDING`; a successful run marks `COMPLETED`; a
  simulated `AIClientError` marks `FAILED` (not silent) and the import/floor are
  unaffected (`tests/services/test_needs_ai.py`). Frontend: the pending/failed notes
  render; a completed status shows none (`needs-dashboard.test.tsx`).
- Backend `1010 passed`, mypy strict clean, ruff clean, migration round-trips.
  Frontend `132 passed`, tsc + biome clean, build passes. No regression on
  LP-68/69/70, the MISMO import, or the extractors.
